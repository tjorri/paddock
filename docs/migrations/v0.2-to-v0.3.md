# Migrating from v0.2 to v0.3

v0.3 is a breaking release. The credential wiring moves from per-template
Secret references to a broker-issued model. There is no migration tool;
the rewrites below are mechanical.

See [spec 0002](../specs/0002-broker-proxy-v0.3.md) for the design and
[ADR-0014](../adr/0014-capability-model-and-admission.md) for the
capability model.

## What's breaking

1. `ClusterHarnessTemplate.spec.credentials` / `HarnessTemplate.spec.credentials`
   — **removed**. Replaced by `spec.requires`.
2. `HarnessRun.spec.extraEnv` entries that reference a `secretKeyRef`
   — **rejected at admission**. Credentials must flow through the
   broker.
3. Running a template that declares `spec.requires` in a namespace with
   no backing `BrokerPolicy` — **rejected at admission**. Operator
   consent is now explicit.

v0.2 templates that declared no `credentials` continue to work unchanged.

## Step-by-step

### 1. Install v0.3

```
helm upgrade --install paddock ./charts/paddock \
  -n paddock-system --create-namespace \
  --set image.tag=v0.3.0
```

The chart bumps CRDs for `BrokerPolicy`, `AuditEvent`, and the new
`requires` field on templates. Existing v0.2 objects are preserved:
the template `credentials` field simply disappears from the schema —
Kubernetes drops the now-unknown field on the next apply.

### 2. Rewrite templates

**Before (v0.2):**
```yaml
apiVersion: paddock.dev/v1alpha1
kind: ClusterHarnessTemplate
metadata: {name: claude-code}
spec:
  image: ghcr.io/tjorri/paddock-claude-code:v0.3.0
  command: [/usr/local/bin/paddock-claude-code]
  credentials:
    - name: anthropic-api-key
      envKey: ANTHROPIC_API_KEY
      secretRef: {name: anthropic-api, key: key}
```

**After (v0.3):**
```yaml
apiVersion: paddock.dev/v1alpha1
kind: ClusterHarnessTemplate
metadata: {name: claude-code}
spec:
  image: ghcr.io/tjorri/paddock-claude-code:v0.3.0
  command: [/usr/local/bin/paddock-claude-code]
  requires:
    credentials:
      - name: ANTHROPIC_API_KEY     # the env-var the agent reads
        purpose: llm                # llm | gitforge | generic
    egress:
      - host: api.anthropic.com
        ports: [443]
```

Field-by-field:

- `credentials[*].envKey` → `requires.credentials[*].name`. The agent
  env-var name is now the logical name; there is no separate
  `envKey`.
- `credentials[*].secretRef` → **move into the namespace's `BrokerPolicy`**
  (next step).
- `credentials[*].name` → remove (or repurpose as the env-var name).

`requires.egress[]` is new — declare every upstream host the agent will
connect to. The proxy blocks anything not declared here once M5
(transparent interception) lands.

### 3. Create a `BrokerPolicy` per namespace

Namespaces with zero `BrokerPolicy` objects reject every run. To un-break
an existing namespace:

```yaml
apiVersion: paddock.dev/v1alpha1
kind: BrokerPolicy
metadata: {name: allow-claude-code, namespace: my-team}
spec:
  appliesToTemplates: ['claude-code']
  grants:
    credentials:
      - name: ANTHROPIC_API_KEY
        provider:
          kind: AnthropicAPI
          secretRef: {name: anthropic-api, key: key}
    egress:
      - host: api.anthropic.com
        ports: [443]
        substituteAuth: true         # required for AnthropicAPI
```

`kubectl paddock policy scaffold <template> -n <ns>` generates this
skeleton interactively (M10). Until it lands, copy the example in
`config/samples/paddock_v1alpha1_brokerpolicy.yaml`.

**One-liner for the v0.2 "just read this Secret" shape:**

```yaml
apiVersion: paddock.dev/v1alpha1
kind: BrokerPolicy
metadata: {name: legacy-static, namespace: my-team}
spec:
  appliesToTemplates: ['*']
  grants:
    credentials:
      - name: MY_TOKEN
        provider:
          kind: Static
          secretRef: {name: my-secret, key: token}
```

`StaticProvider` preserves v0.2 ergonomics while routing the read
through the broker's audit trail (ADR-0015).

### 4. Purge Secret refs from `HarnessRun.spec.extraEnv`

If you had:

```yaml
extraEnv:
  - name: GITHUB_TOKEN
    valueFrom:
      secretKeyRef: {name: gh-token, key: token}
```

replace with a `requires.credentials` entry on the template and a
`BrokerPolicy` grant. Literal `extraEnv` values (`name:`/`value:` form)
still work.

### 5. Workspace seed-job credentials

`Workspace.spec.seed.repos[*].credentialsSecretRef` continues to work
in v0.3 (ADR-0006). When a `BrokerPolicy` in the namespace grants
`gitforge`, the reconciler prefers broker-issued tokens; the v0.2 path
is kept as a fallback for backward compatibility with docs that aren't
yet updated. v0.4 will remove the v0.2 path.

## Timeline

- **v0.3.0-alpha.*** through **M4** — broker skeleton + proxy in
  cooperative mode are available; transparent interception lands in
  M5, auth substitution in M7.
- Until M3 ships, any template declaring `requires` will be rejected
  at admission with a clear diagnostic. Keep v0.2 templates (no
  `credentials`, no `requires`) running in the meantime, or pin to
  v0.2.x.
- **v0.4** removes the v0.2 `credentialsSecretRef` path on workspaces.

## Troubleshooting

- **"template X declares spec.requires but no BrokerPolicy in namespace Y
  backs it"** — you're on M2 pre-M3. Either remove `requires` from the
  template (use v0.2-style standalone templates) or wait for M3.
- **"secret-valued env vars must flow through the broker"** — move the
  Secret reference to a `BrokerPolicy`-backed credential (step 3).
- **Kubernetes drops `credentials` on apply without an error** — that
  field is removed from the schema; the apiserver ignores it. Move the
  config to `requires` + `BrokerPolicy`.
