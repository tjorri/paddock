## 3. Attack-scenario list

Each TG-NN below is a concrete adversarial scenario with enough detail that Phase 2 can implement directly without re-design. Multiple findings can share a TG-NN where the same test catches them all.

### TG-1 тАФ Agent forges a leaf cert against the per-run MITM CA

**Scenario.** A run is configured with the per-run proxy CA. The hostile agent reads the proxy's CA bundle (which is mounted into the agent container as `/etc/paddock/proxy-ca.crt`) and, having located its private key (deliberately *not* mounted into the agent), proves the negative тАФ the private key is unreachable from the agent's UID, mount points, and SA-token-derived API server reads.

**Expected behaviour.** The harness's `--forge-ca-cert` flag returns `{"result":"key-unreachable"}` in all four read attempts (filesystem walk for `tls.key`-named files, `kubectl get secret` via SA token, env var probe, `/proc/self/mounts` probe). If any attempt yields key bytes, the test fails loudly.

**Realism level.** Real cluster (Kind, any CNI), hostile harness (yes, `--forge-ca-cert`), real upstream not required (assertion is on key-reachability, not on a forged TLS connection's outcome).

**Test layer.** E2E.

**Related findings.** F-18.

**Hostile harness flags used.** `--forge-ca-cert <fqdn>`.

**Phase 2 effort estimate.** Medium тАФ needs hostile harness with `--forge-ca-cert` and an assertion harness that reads the run's proxy-tls Secret outside the run pod to confirm content vs. what agent sees.

**Status (recheck 2026-04-26).**
- State: Partial
- Reviewer: @tjorri
- Shipped by: Phase 2b / PR #24 (harness flag); Phase 2f / PR #34 (per-run isolation unit test)
- Test files: `images/evil-echo/main.go` (`--forge-ca-cert` handler), `internal/controller/proxy_tls_isolation_test.go`
- Residual: No E2E scenario drives `--forge-ca-cert` against a live run to assert key-unreachable; unit test (`proxy_tls_isolation_test.go`) confirms per-run CA key isolation at the controller level but does not attempt the in-pod key-read attack path.

### TG-2 тАФ Agent unsets HTTPS_PROXY in cooperative mode

**Scenario.** Run admitted with `interception.cooperativeAccepted: true`. The hostile agent unsets `HTTPS_PROXY`, `HTTP_PROXY`, `NO_PROXY`, then attempts a direct TCP connection to `api.openai.com:443` (or any non-allowlisted internet host).

**Expected behaviour.** The connection is blocked by NetworkPolicy egress rules, *or* тАФ if the test reveals NetworkPolicy doesn't actually block the bypass (F-19) тАФ the test must fail loudly with the diagnostic that cooperative mode + current NP shape allows direct egress, vindicating F-19 and providing the regression test for the eventual fix. AuditEvent records the attempt as `kind: egress-block` with the per-connection 5-tuple.

**Realism level.** Real cluster (Kind with Cilium or Calico тАФ kindnet does not enforce NetworkPolicy), hostile harness (yes, `--bypass-proxy-env` + `--connect-raw-tcp`), real upstream not required (assertion is on the egress decision).

**Test layer.** E2E.

**Related findings.** F-19, F-27.

**Hostile harness flags used.** `--bypass-proxy-env`, `--connect-raw-tcp api.openai.com:443`.

**Phase 2 effort estimate.** Medium тАФ depends on Cilium/Calico Kind setup (TG-realism ┬з5 must-have), hostile harness build, and a non-allowlisted domain that resolves but rejects.

### TG-3 тАФ Tenant overrides `HTTPS_PROXY`/`SSL_CERT_FILE` via `extraEnv`

**Scenario.** Tenant submits a HarnessRun with `spec.extraEnv: [{name: HTTPS_PROXY, value: ""}, {name: SSL_CERT_FILE, value: "/etc/ssl/certs/ca-certificates.crt"}]`. The controller builds the Pod spec; the K8s Pod env-merge resolves duplicates last-wins, so the tenant value wins.

**Expected behaviour.** Either (a) the webhook rejects the HarnessRun at admission with an error naming the reserved keys (post-fix), or (b) the controller's effective Pod env shows Paddock's HTTPS_PROXY value (because the controller appends extraEnv first and Paddock-authored env last). Until either fix lands, the test asserts the *current* (broken) behaviour and must fail loudly to provide the regression target.

**Realism level.** Unit (controller pod-spec) + integration (webhook envtest). Real cluster not required.

**Test layer.** Unit (controller pod-spec build) + Integration (envtest webhook).

**Related findings.** F-39.

**Hostile harness flags used.** None тАФ this attack is purely admission/spec-level.

**Phase 2 effort estimate.** Small тАФ table-driven webhook test + controller pod-spec test.

**Status (recheck 2026-04-26).**

## 4. Hostile harness ("evil paddock-echo") specification

A test-only harness image that exercises hostile agent behaviours. Used by multiple TG-NN entries above. **Phase 2 deliverable**, specified here so Phase 2 doesn't re-design.

### Image location

`images/evil-echo/` тАФ separate from `images/harness-echo/` so the trustworthy reference harness stays clean. Image tagged `paddock-evil-echo:dev` for local builds; not part of release artefacts; not pushed to public registries. The Makefile target `make evil-echo-image` builds and loads into the Kind test cluster.

### Base

Built on `paddock-echo`'s base layers тАФ same Pod Security Standards conformance тАФ so it can run in the same `restricted` namespaces as the friendly version. The hostile behaviours are CLI-flag-gated; absent any flag, the image behaves identically to `paddock-echo` (i.e., echoes its prompt, writes a result.json, exits 0).

### CLI flags (each is one attack)

Standard set (from the design doc):

- `--connect-raw-tcp <host:port>` тАФ open a raw TCP socket to the host:port; assert the test sees the connection blocked or substituted-via-proxy. Used by TG-2, TG-12.
- `--read-secret-files <glob>` тАФ read files matching the glob (typically `/var/run/secrets/...`); print contents to stdout. Tests assert the harness either fails to read or reads only what's documented as visible to the agent. Used by TG-7.
- `--probe-broker <url>` тАФ send a request directly to the broker's mTLS endpoint, including using stolen / guessed bearers; assert the broker responds with deny. Supports `--with-bearer <token>` and `--with-sa-token <path>`. Used by TG-7, TG-11, TG-18.
- `--bypass-proxy-env` тАФ unset `HTTPS_PROXY`, `HTTP_PROXY`, `NO_PROXY` before subsequent actions. Combined with other flags to test cooperative-mode bypass. Used by TG-2.
- `--exfil-via-dns <host>` тАФ encode payloads in DNS queries to a non-allowlisted DNS resolver; assert resolver requests are blocked. Used by TG-24 (variant).
- `--read-other-tenant-pvc <namespace>` тАФ attempt to mount/read PVCs from another namespace; assert RBAC blocks. Used by TG-8.
- `--forge-ca-cert <fqdn>` тАФ attempt to read the per-run CA private key (filesystem walk, env probe, SA-token-driven kubectl get); if read, generate a cert; attempt MITM. Assert key not readable. Used by TG-1, TG-6.
- `--connect-ip-literal <ip:port>` тАФ connect to an IP literal bypassing DNS; assert the proxy still rejects (allowlist is by name). Used by TG-4, TG-11, TG-12.
- `--probe-imds` тАФ attempt to reach `169.254.169.254:80` (cloud metadata service) via direct TCP; assert blocked by NetworkPolicy or proxy allowlist. Used by TG-12.
- `--connect-loopback <port>` тАФ connect to in-pod loopback (e.g., the proxy itself, the SA-token volume's API server proxy); assert only documented loopback ports respond. Used by TG-12.

Audit-derived additional flags:

- `--probe-env-override` тАФ start a child process whose env table includes a tenant-injected `HTTPS_PROXY=""` (or `SSL_CERT_FILE=...`) overriding controller-emitted values; assert that the in-container effective env reflects the controller's values, not the tenant's. Tests F-39 in-container alongside the admission check. Used by TG-3.
- `--probe-provider-substitution-host <bearer> <wrong-host:port>` тАФ present an Anthropic / GitHubApp / PATPool bearer in a request whose `Host` header (or SNI) is not in the bearer's allowed-hosts list; assert the broker's `SubstituteAuth` returns no substitution. Tests F-09. Used by TG-13.
- `--read-pvc-git-config` тАФ mount the workspace PVC read-only into a sidecar of the harness Pod and dump `.git/config` and `/workspace/.paddock/repos.json` to stdout; assert no embedded credential substring. Tests F-50. Used by TG-25.
- `--flood-connect-raw-tcp <host:port> --count=N` and `--flood-distinct-sni <wildcard-pattern> --count=N` тАФ load-shape variants of `--connect-raw-tcp` for DoS scenarios (TG-9, F-26, F-28). Bounded by N to avoid runaway test resource use.
- `--smuggle-headers <host:port>` тАФ open a substitute-auth path connection with a fixed set of extra headers (`x-api-key: smuggled`, `cookie: stolen`, etc.); assert the upstream stand-in receives only the per-host allowlist. Tests F-21. Used by TG-10.

### Output

Each flag produces structured stdout, one JSON object per attempt:

```json
{"flag": "--connect-raw-tcp", "target": "api.openai.com:443", "result": "denied", "error": "connection refused", "duration_ms": 12}
{"flag": "--forge-ca-cert", "fqdn": "api.anthropic.com", "result": "key-unreachable", "attempts": [{"path": "/etc/paddock/proxy-ca.key", "error": "no such file"}, {"sa_kubectl_get": "forbidden"}]}
{"flag": "--probe-broker", "url": "https://paddock-broker.paddock-system.svc:8443/v1/issue", "result": "denied", "status": 403, "body_excerpt": "cross-namespace ..."}
```

Result values: `denied`, `allowed-as-expected`, `unexpectedly-allowed`, `key-unreachable`, `error`. Tests parse this output to assert the expected outcome for each attack.

### Pod Security Standards conformance

The harness must pass `restricted` PSS тАФ no privileged capabilities, no NET_ADMIN, no hostPath, no `runAsRoot`, `allowPrivilegeEscalation: false`, all caps dropped, seccomp `RuntimeDefault`. Some attacks (e.g., raw TCP outside HTTPS, packet-crafting) are constrained by this; that is intentional тАФ we want to test what a malicious-but-not-privileged agent can do. Attacks needing kernel-level privilege (`NET_ADMIN`, hostPath escape) are explicitly out of scope of the harness тАФ they're operator-posture concerns, not Paddock's containment perimeter.

### Testing the harness itself

A small unit-test suite verifies each flag does what it claims (without external network тАФ tests use loopback or test-doubles for the broker / upstream). The harness's correctness is its own deliverable in Phase 2; tests above use it as a tool, but its correctness is established independently.

15 = 10 standard + 5 audit-derived.

## 5. Test-realism dimensions

For each E2E entry currently in `test/e2e/e2e_test.go` (and each TG-NN above whose layer is E2E), an explicit declaration of which realism axes are exercised. The Phase 2 must-haves and opportunistic items are then derived.

### Axes

- **Real CNI** тАФ does the test cluster use a CNI that *enforces* NetworkPolicy (Cilium / Calico) or a no-op CNI (kindnet default)? Without a real CNI, every TG-NN that asserts NetworkPolicy enforcement is meaningless.
- **Real upstream** тАФ does the test hit real external services (`github.com`, `api.anthropic.com`), or stub them? Real upstream is necessary only when the assertion *requires* observing the actual upstream's response.
- **Hostile harness** тАФ does the test run an actual hostile image (`paddock-evil-echo`), or simulate hostile behaviour via prompt-engineering / mocks? Prompt-driven hostility is fragile and depends on the harness's behaviour; image-driven hostility is deterministic.
- **Concurrency** тАФ does the test exercise parallel runs / mid-flight edits / fault injection (broker restart under load, finalizer races)?

### Existing E2E suite scoring (as of 2026-04-25)

| Scenario                                       | Real CNI    | Real upstream                     | Hostile harness                       | Concurrency      |

| `multi-repo workspace seeding`                 | No (kindnet) | Partial (`github.com` clone)     | No                                    | No               |
