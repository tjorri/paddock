# v0.4 Security Review — Phase 2c (Theme A: audit completeness) Design

- Owner: Tuomo
- Date: 2026-04-25
- Status: Draft
- Predecessor: Phase 2b (test infrastructure + adversarial coverage, merged in #24)
- Successor (planned): one of Theme B (per-run isolation), Theme C (runtime egress), or the kube-apiserver NetworkPolicy carry-over from Phase 2b — chosen via a separate brainstorm after Phase 2c ships

## 1. Summary

Phase 2c closes the audit-completeness findings clustered as Theme A in the v0.4 security audit:

- **F-12** — Broker audit writes are logged-and-swallowed; credential is still returned on audit failure.
- **F-13** — Three infrastructure-error paths in `Server.issue` skip AuditEvent emission entirely.
- **F-24** — Proxy audit writes are best-effort with errors silently dropped on both deny and allow paths.
- **F-32** — Admission webhooks emit no AuditEvent on any decision; ADR-0014's `kind: policy-applied` trail is unimplemented.
- **F-40** — Controller emits no AuditEvent for any decision; threat-model B-2 Repudiation row is unbacked.

Plus a directly-entangled webhook fix:

- **F-33** — `ValidatingWebhookConfiguration` for AuditEvent uses `failurePolicy: Fail` cluster-wide, which combined with fail-closed audit semantics would create a self-DoS during controller-pod outages.

The cross-cutting fix is to extract a shared `internal/auditing` package with a `Sink` interface, default `KubeSink` implementation, and per-kind builder functions, then route every audit emission across broker, proxy, webhook, and controller through it. Fail-closed semantics are applied to the broker and proxy deny paths (per audit-doc recommendation); admission and controller emit best-effort with loud logging and Prometheus counters.

The test layer adds two new adversarial e2e scenarios on top of the Phase 2b hostile-suite infrastructure: TG-19 (broker fail-closed when audit unavailable) and an admission-rejected variant that exercises F-32's `policy-rejected` AuditEvent.

## 2. Goals and non-goals

**Goals:**

1. Extract `internal/auditing` package as the single source of truth for AuditEvent emission across all components.
2. Make broker and proxy audit emission fail-closed on the deny path; broker issuance writes audit *before* releasing the credential.
3. Add three controller-side and one admission-side AuditKind constants and emit them at the load-bearing decision points.
4. Set the AuditEvent validating webhook to `failurePolicy: Ignore` so audit-trail availability isn't gated on the controller pod's webhook server.
5. Validate end-to-end with two new adversarial e2e scenarios using the Phase 2b hostile-suite infrastructure.
6. Update the audit findings doc, test-gaps doc, and ADRs 0014 / 0016 to reflect the resolved findings.

**Non-goals (deferred):**

- F-25 (per-request validator re-check on substitute path) — Theme C runtime-egress.
- F-30 (`appliesToTemplates` glob support) — Theme B per-run isolation.
- F-41 (`Owns()` declarations for Secret / NetworkPolicy / SA / Role / RoleBinding) — deferred follow-up; not a Theme A finding.
- F-43 (audit-correlation between NP enforcement flips and run state) — needs both audit emission and NP-flip observability; defer to whichever phase tackles F-43 directly.
- Controller emit sites 4 (NP flip) and 6 (finalizers) — Question 4 explicitly excluded.
- The bounded retry buffer from F-24's recommendation — Question 2 explicitly rejected in favour of fail-closed semantics.
- The kube-apiserver NetworkPolicy egress carry-over from Phase 2b — Question 1 explicitly excluded.

## 3. Architecture

A new `internal/auditing` package becomes the single source of truth for AuditEvent emission. It exports:

```
internal/auditing/
  sink.go        # Sink interface + KubeSink + NoopSink
  builders.go    # NewCredentialIssued, NewEgressBlock, NewPolicyApplied, ...
  metrics.go     # paddock_audit_write_failures_total{component, decision, kind}
  errors.go      # ErrAuditWrite (sentinel for fail-closed callers)
  sink_test.go
  builders_test.go
```

All four components (broker, proxy, webhook, controller) construct an `auditing.Sink` at startup and route every audit decision through it.

### 3.1 Failure semantics

`Sink.Write` returns an error. Callers decide policy:

| Component | Path | On `Sink.Write` error |
|-----------|------|-----------------------|
| Broker | issuance | Audit written *before* response; on error → `503 paddock-broker: audit unavailable, please retry` |
| Broker | deny | Audit written *before* response; on error → `503 paddock-broker: audit unavailable, please retry` |
| Broker | infrastructure-error (F-13 paths) | Construct synthetic `credential-denied` audit; on error → `503` |
| Proxy | deny (CONNECT) | On error → `502 paddock-proxy: audit unavailable` (overrides original 403) |
| Proxy | deny (transparent) | On error → close connection with TCP RST |
| Proxy | allow | On error → log + counter, MITM proceeds |
| Admission | admit / reject | On error → log at `Error()`, validator's decision stands |
| Controller | all sites | On error → log at `Error()`, status condition is canonical |

Every failure increments `paddock_audit_write_failures_total{component, decision, kind}` regardless of fail-closed vs fail-open.

**No retry buffer.** F-24's recommendation included a 1024-event in-memory ring buffer for retry. We rejected it in favour of strict fail-closed: the retryable surface is the caller's natural retry (agents retry 503; the broker's next request's audit write is the recovery path). Simpler state, no broker-restart loss.

### 3.2 F-33 — AuditEvent webhook fail-policy

Setting fail-closed on the broker/proxy deny paths makes the AuditEvent webhook's availability load-bearing. If the webhook (which runs in the controller pod in `paddock-system`) is down, every audit write fails, and every deny is converted to 503/502 — a self-DoS.

Mitigation: the `ValidatingWebhookConfiguration` for path `/validate-paddock-dev-v1alpha1-auditevent` gets `failurePolicy: Ignore`. All other webhooks keep `failurePolicy: Fail`. The trade-off is documented in ADR-0016: write-once enforcement is bypassed during webhook outages, but audit-trail availability isn't gated on the controller pod's health.

## 4. The `auditing` package

### 4.1 Sink interface

```go
package auditing

type Sink interface {
    Write(ctx context.Context, ae *paddockv1alpha1.AuditEvent) error
}

type KubeSink struct {
    Client    client.Client
    Component string  // "broker" | "proxy" | "webhook" | "controller"
}

func (s *KubeSink) Write(ctx context.Context, ae *paddockv1alpha1.AuditEvent) error {
    // Stamp the component label so consumers (and our own counters) can
    // distinguish e.g. broker-emitted from controller-emitted events of
    // the same kind.
    if ae.Labels == nil {
        ae.Labels = map[string]string{}
    }
    ae.Labels["paddock.dev/component"] = s.Component
    if err := s.Client.Create(ctx, ae); err != nil {
        auditWriteFailures.WithLabelValues(s.Component, string(ae.Spec.Decision), string(ae.Spec.Kind)).Inc()
        return fmt.Errorf("%w: %v", ErrAuditWrite, err)
    }
    return nil
}

type NoopSink struct{}
func (NoopSink) Write(_ context.Context, _ *paddockv1alpha1.AuditEvent) error { return nil }
```

### 4.2 Builders

Builders are pure constructors — no I/O, no clients. Each takes a flat parameter struct; output is a fully-populated `*AuditEvent` with standard labels (`paddock.dev/run`, `paddock.dev/decision`, `paddock.dev/kind`), `GenerateName` prefix per kind family, and `Spec` populated.

```go
type CredentialIssuedInput struct {
    RunName, Namespace, CredentialName, Provider, MatchedPolicy, Reason string
    When time.Time
}
func NewCredentialIssued(in CredentialIssuedInput) *paddockv1alpha1.AuditEvent
func NewCredentialDenied(in CredentialDeniedInput) *paddockv1alpha1.AuditEvent

type EgressInput struct {
    RunName, Namespace, Host string
    Port int
    Decision paddockv1alpha1.AuditDecision
    MatchedPolicy, Reason string
    When time.Time
    Kind paddockv1alpha1.AuditKind  // optional override
}
func NewEgressAllow(in EgressInput) *paddockv1alpha1.AuditEvent
func NewEgressBlock(in EgressInput) *paddockv1alpha1.AuditEvent
func NewEgressDiscoveryAllow(in EgressInput) *paddockv1alpha1.AuditEvent

type AdmissionInput struct {
    RunName, Namespace, TemplateRef, Reason string
    When time.Time
    OwnerRef *metav1.OwnerReference  // set on admit, nil on reject
}
func NewPolicyApplied(in AdmissionInput) *paddockv1alpha1.AuditEvent
func NewPolicyRejected(in AdmissionInput) *paddockv1alpha1.AuditEvent

type RunDecisionInput struct {
    RunName, Namespace, Reason string
    Decision paddockv1alpha1.AuditDecision
    When time.Time
}
func NewRunFailed(in RunDecisionInput) *paddockv1alpha1.AuditEvent
func NewRunCompleted(in RunDecisionInput) *paddockv1alpha1.AuditEvent

type CAProjectionInput struct {
    RunName, Namespace, SecretName, Reason string
    When time.Time
}
func NewCAProjected(in CAProjectionInput) *paddockv1alpha1.AuditEvent
```

GenerateName prefixes by kind family:
- `ae-cred-` for credential-* kinds
- `ae-egress-` for egress-* kinds
- `ae-policy-` for policy-applied / policy-rejected
- `ae-run-` for run-failed / run-completed
- `ae-ca-` for ca-projected

### 4.3 New AuditKind and label constants

Added to `api/v1alpha1/auditevent_types.go`:

- `AuditEventLabelComponent = "paddock.dev/component"` — populated by `KubeSink.Write` from its `Component` field; values are `broker` | `proxy` | `webhook` | `controller`.

- `AuditKindPolicyRejected = "policy-rejected"` — admission deny.
- `AuditKindRunFailed = "run-failed"` — controller terminal-failure (BrokerDenied, WorkspaceRequired, etc.).
- `AuditKindRunCompleted = "run-completed"` — controller terminal-phase commit (Succeeded / Failed / Cancelled).
- `AuditKindCAProjected = "ca-projected"` — controller CA Secret create (`<run>-broker-ca`, `<run>-proxy-tls`).

The `+kubebuilder:validation:Enum=...` annotation on `AuditKind` extends to include these. CRD regeneration via `make manifests`. The AuditEvent validating webhook is permissive on `kind` (only enforces write-once on `spec`), so no webhook-code change is needed for the new kinds.

### 4.4 Migration

Step 1 of the implementation plan migrates existing emitters to delegate to the new package as a no-op refactor:

- `internal/broker/audit.go` — `AuditWriter.CredentialIssued`/`CredentialDenied` become thin wrappers around `Sink.Write(ctx, auditing.NewCredentialIssued(...))`.
- `internal/proxy/audit.go` — `ClientAuditSink.RecordEgress` becomes `s.Sink.Write(ctx, auditing.NewEgressBlock(...))`. Signature changes: it now returns an error (today returns nothing).

Existing callers of those wrappers don't change in step 1. Behavior changes (fail-closed semantics) ship in steps 2 and 3.

## 5. Per-component changes

### 5.1 Broker (F-12, F-13)

**Files:** `internal/broker/server.go`, `internal/broker/audit.go`.

**`handleIssue` issuance path (server.go ~128–151):**
- Construct `CredentialAudit`, write audit *before* writing the response. Specifically: after the provider has minted the credential and recorded the lease in memory, but before writing the JSON response.
- On `Sink.Write` error → return `503 paddock-broker: audit unavailable, please retry`. The credential has been minted at this point but not yet returned; the caller sees a 503 and retries.

**`handleIssue` deny path (server.go ~132–137):**
- Same shape — audit-write must succeed before the deny is returned. 503 on audit failure.

**`Server.issue` infrastructure-error paths (server.go 222, 227, 246):**
- Each constructs a `CredentialAudit` with `Reason: "broker infrastructure error: <op>"` and routes through the sink as a denial.
- If the audit write also fails, return 503 (the infra was already broken; the audit gap matters more than a 500-vs-503 distinction).
- The three paths today return `(IssueResult{}, nil, nil, err)` with the audit pointer set to nil; the change replaces the nil with a populated `CredentialAudit` and lets `handleIssue` emit it consistently.

**`handleSubstituteAuth` (server.go 412–482):**
- Currently emits no AuditEvent. F-12's "consider also" note flagged this. We add audit emission for both successful substitution (`credential-issued` with `decision: granted`) and rejection (`credential-denied` with `decision: denied`). Same fail-closed shape on the deny path. Including this now since we're rebuilding the audit ordering anyway; it's trivially droppable in review if scope creep is a concern.

### 5.2 Proxy (F-24)

**Files:** `internal/proxy/audit.go`, `internal/proxy/server.go`, `internal/proxy/mode.go`, `cmd/proxy/main.go`.

**`ClientAuditSink.RecordEgress` (audit.go ~75–119):**
- Returns an `error` (signature change — currently returns nothing).
- Replace `_ = s.Client.Create(...)` with explicit `if err := s.Sink.Write(ctx, ae); err != nil { logger.Error(err, ...); return err }`.
- The sink is the new `auditing.Sink`; `ClientAuditSink` becomes a thin per-run/namespace adapter that constructs the `EgressEvent`-shaped builder input and calls through.

**`Server.handleConnect` deny paths (server.go 128–133, 138–143):**
- On audit-write failure, return `502 paddock-proxy: audit unavailable` instead of the original 403. The agent learns the action is in an indeterminate state and can retry.

**`mode.go` transparent-mode deny paths (~78–96):**
- On audit-write failure, close the connection with a TCP RST. No HTTP response surface in transparent mode; RST is the only signal. Implement via `tcpConn.SetLinger(0); tcpConn.Close()`.

**Allow path (server.go 215–221, mode.go 144–150):**
- On audit-write failure, log at `Error()` level and increment counter. MITM proceeds. Allowing a connection without an audit record is a known soft loss; failing it would block legitimate traffic on transient etcd hiccup.

**`buildAuditSink` (cmd/proxy/main.go 263–278):**
- Today returns `NoopAuditSink` silently when the cluster client cannot be built or run-name/namespace are empty. Add a warning log so the fall-back is visible in proxy startup logs.

### 5.3 Webhook (F-32)

**Files:** `internal/webhook/v1alpha1/harnessrun_webhook.go`, `internal/webhook/v1alpha1/brokerpolicy_webhook.go`, `cmd/main.go` (controller-manager bootstrap).

**`HarnessRunCustomValidator.ValidateCreate` and `ValidateUpdate`:**
- After the validator decides admit/reject, emit `policy-applied` (admit) or `policy-rejected` (reject) before returning.
- Construction uses `auditing.NewPolicyApplied`/`NewPolicyRejected`.
- `OwnerReference` rules:
  - `ValidateCreate` admit → owner-ref the (about-to-be-created) HarnessRun via the post-admission UID once the apiserver assigns it. Since the validator runs before the object is persisted, owner-ref is left unset on the AuditEvent and the 30-day retention reaper governs lifetime.
  - `ValidateCreate` reject → owner-ref unset; the run never existed.
  - `ValidateUpdate` admit/reject → owner-ref the existing HarnessRun (the object exists in etcd; only the proposed mutation is rejected).

**`BrokerPolicyCustomValidator.ValidateUpdate`:**
- For policy-edit decisions, emit `policy-applied` (admit) or `policy-rejected` (reject) with the BrokerPolicy as `OwnerReference`.

**Fail-open semantics:**
- On `Sink.Write` error, log at `Error()` level and let the validator's decision stand. Admission must not block on audit availability per F-32's own recommendation.

**Wiring:**
- Add a `Sink auditing.Sink` field to each validator.
- `cmd/main.go` constructs an `auditing.KubeSink{Client: mgr.GetClient(), Component: "webhook"}` and passes it into the validators during webhook setup.

### 5.4 Controller (F-40, sites 1–3 + 5)

**Files:** new `internal/controller/audit.go`, modifications to `harnessrun_controller.go`, `proxy_tls.go`, `broker_ca.go`, `broker_credentials.go`, `cmd/main.go`.

**`internal/controller/audit.go`:**
- Thin per-controller helper: holds an `auditing.Sink` reference, exposes `emitRunFailed`, `emitRunCompleted`, `emitCAProjected`, `emitCredentialIssuedSummary` methods that wrap the package builders + sink call + error logging + counter increment.

**Site 1 (credential-issuance loop completion in `ensureBrokerCredentials`):**
- After the loop in `internal/controller/broker_credentials.go` writes the `<run>-broker-creds` Secret, emit one `credential-issued` AuditEvent summarising the credentials projected.
- Disambiguation from the broker's per-credential `credential-issued` events: the controller's summary sets `Spec.Count = N` (where N is the number of credentials projected) and is labelled `paddock.dev/component=controller`; the broker's per-credential events leave `Count` unset (implicit 1) and are labelled `paddock.dev/component=broker`. The `paddock.dev/component` label is set automatically by `KubeSink.Write` from its `Component` field on every emission across all components, so callers don't have to remember.

**Site 2 (`fail()`):**
- Every terminal-failure path in `harnessrun_controller.go:1175–1195` and elsewhere — `BrokerDenied`, `BrokerUnavailable`, `PromptSourceNotFound`, `PromptKeyMissing`, `WorkspaceRequired`, `InterceptionUnavailable`, `BrokerNotConfigured`, `BrokerCAPending`, `ProxyCAPending`, `BrokerProxyNotConfigured` — emits `run-failed` with `decision: denied` and `reason: <FailureReason>`.

**Site 3 (terminal-phase commit):**
- In the main reconcile loop's terminal path and `reconcileDelete` (`harnessrun_controller.go:1098–1116`), emit `run-completed` once per run-lifetime:
  - `decision: granted` on Succeeded.
  - `decision: denied` on Failed (so Site 2's `run-failed` is followed by a `run-completed{denied}` for the lifecycle event; the kinds are semantically distinct — terminal cause vs. lifecycle close).
  - `decision: warned` on Cancelled.
- Idempotency: emit only on the reconcile that actually flips status.phase to terminal; skip on subsequent reconciles of an already-terminal run. Implemented by emitting from the same code path that sets `status.phase` to terminal, before the status update.

**Site 5 (CA projection):**
- `proxy_tls.go:113–128` (`<run>-proxy-tls` Secret create) and `broker_ca.go:87` (`<run>-broker-ca` Secret create) emit `ca-projected` whenever the per-run CA Secret is created.
- Idempotency: emit on Create, not on no-op reconciles where the Secret already exists with the right bytes. Detect via the `IsAlreadyExists`-vs-`nil` return from the Create call.

**Fail-open semantics:**
- All controller emits log at `Error()` + counter on `Sink.Write` error, don't block reconcile. Status conditions remain canonical for run state.

**Wiring:**
- Add a `Sink auditing.Sink` field to `HarnessRunReconciler`.
- `cmd/main.go` constructs `auditing.KubeSink{Client: mgr.GetClient(), Component: "controller"}` and passes it in.

### 5.5 F-33 — AuditEvent webhook fail-policy

**Files:** `config/webhook/manifests.yaml`, `charts/paddock/templates/paddock.yaml`, `docs/observability/` (new note).

- The `ValidatingWebhookConfiguration` block whose `path: /validate-paddock-dev-v1alpha1-auditevent` gets `failurePolicy: Ignore`. All other webhook blocks keep `failurePolicy: Fail`.
- Add an inline comment in the manifest: "Ignore on the AuditEvent webhook only — write-once enforcement bypassed during webhook outages, but audit-trail availability is the higher priority. See ADR-0016 §F-33."
- Add a Prometheus alert example to `docs/observability/audit-monitoring.md` (new file or extension): operators detect AuditEvent writes that bypass validation by alerting on (a) the controller pod's webhook handler being down for >5min via `paddock_webhook_up == 0`, or (b) `paddock_audit_write_failures_total` rate exceeding a threshold (e.g., 5% over 5 min).

## 9. Risks and mitigations

1. **Issuance retry-storm under prolonged AuditEvent outage.** If etcd is down or the AuditEvent CRD is mis-installed for a long period, every issue request returns 503 and agents retry. The broker's PATPool may briefly over-lease as each retry mints fresh credentials before failing. Mitigation: F-33's `failurePolicy: Ignore` removes one failure mode; the remaining failure modes (etcd partition, RBAC drift) are observable via `paddock_audit_write_failures_total`. Operators set an alert threshold (e.g., 5% failure rate over 5 min) and pause issuance via cluster-level circuit-breaker if needed. Documented in chart values comment.
2. **Webhook self-DoS during AuditEvent webhook chart install.** The new fail-policy means the AuditEvent webhook can be missing during initial install without blocking AuditEvent writes from broker/proxy/controller. The trade-off: write-once enforcement is bypassed during webhook outages. Documented in ADR-0016.
3. **Interface churn during foundation refactor.** Step 1 commits the package interface; if step 4 (admission emission) reveals a missing input field, we extend a builder and risk re-touching step-1's broker migration. Mitigation: builders use parameter structs, not positional args — adding a field is additive.
4. **Broker `handleSubstituteAuth` audit emission as in-scope creep.** F-12's recommendation says "consider also emitting" — we're doing it. If review prefers to defer, it's a separate commit and trivially droppable.
5. **TG-19 adversarial test relies on RBAC patching mid-test.** The test patches the broker's RoleBinding to revoke `auditevents create`, then restores it. If the test panics between patch and restore, subsequent runs may inherit a broken broker. Mitigation: use Ginkgo's `DeferCleanup` for the restore, plus assert the restore succeeded in `AfterAll`. The `make test-e2e` cluster is teardown-on-success anyway.
