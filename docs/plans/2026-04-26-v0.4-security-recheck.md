### Status block template (append after existing prose for each finding)

```
**Status (recheck 2026-04-26).**
- State: <Resolved | Partially resolved | Deferred-by-design | Won't-fix | Open>
- Verification: <Light | Medium | Heavy | N/A>
- Reviewer: @tjorri
- [Residual: <pointer to where residual is tracked>]            ← only on Partially resolved
- [Rationale: <why deferred-by-design or won't-fix>]            ← only on Deferred-by-design or Won't-fix
- [Revisit: <trigger condition, e.g., "at v1.0 GA">]            ← only on Won't-fix
- [Regression: <what broke + when>]                             ← only when rigor check finds Resolution claim no longer holds
```

### Verification rigor recipes

**Light** (~2 min/finding) — for P2 and P3:
1. Read the finding's `**Resolution.**` line (if present) or `**Recommendation.**` line.
2. `grep` or `Read` the cited file(s) to confirm the named symbol/path exists.
3. If cited code matches the prose, write `Verification: Light` and the appropriate State.

**Medium** (~10 min/finding) — for P1 not already Heavy:
1. Light steps 1–2.
2. Read the cited unit test(s) — confirm they actually exercise the claimed behaviour, not just compile.
3. Run the cited unit test(s) locally: `go test -tags=e2e ./internal/<component>/... -run <TestName> -v -count=1`.
4. Write `Verification: Medium` and the appropriate State.

**Heavy** (~30+ min/finding plus one shared e2e run) — for original P0s and Risk-High findings:
1. Medium steps 1–3.
2. Run the relevant e2e spec(s) locally per CLAUDE.md guidance: `make test-e2e 2>&1 | tee /tmp/e2e-recheck-F-NN.log` (use `FAIL_FAST=1` for iteration if needed). For multiple Heavy findings in one component, batch into one e2e run when specs overlap.
3. Confirm e2e exit code 0 and no skipped specs in the relevant area.
4. Write `Verification: Heavy` and the appropriate State.

For unresolved findings the default State is **Open**. A finding is only **Deferred-by-design** or **Won't-fix** if there's an explicit, documentable rationale (typically from a Phase 2X design doc or a code comment); otherwise it's Open.

### Targeted re-scan recipe (Step 4 of every component sweep)

1. Run `gosec ./internal/<component>/...` (capture output to `/tmp/gosec-<component>.txt`). Compare against the Phase 1 baseline noted in the audit doc; any new hits become candidates for new F-NN findings.
2. Walk Phase 2 commits that touched this component:
   ```
   git log dd1e028..HEAD --oneline -- internal/<component>/
   ```
   For each commit, read the diff (`git show <sha> -- internal/<component>/`) with security eyes — look for new code paths, new error returns, new `Unmarshal` / `os.ReadFile` / env-var reads, new HTTP handlers, new RBAC verbs.
3. Cross-reference the threat-model trust boundary the component owns. Open `docs/security/threat-model.md`; find the B-N section for this component; ask: does Phase 2's code change the STRIDE picture for this boundary?

Anything new gets a fresh F-NN entry slotted into §2's findings table and a full finding entry written matching the existing entry shape (Component / Threats / Severity / Priority / Test coverage / Description / Evidence / Recommendation / Status block from creation).

### Commit message format

```
chore(security): recheck — <component or pass name>

<one-paragraph body listing what changed:>
- Verified N findings (R Resolved, P Partially, D Deferred-by-design, W Won't-fix, O Open).
- M state changes from prior baseline (e.g., Resolved → Open for F-XX due to <reason>).
- K new findings opened (F-NN..F-NM).
- L static-tool hits triaged (J reconciled, K opened as new findings).
```

### Mid-flight checkpoint protocol

After Task 6 (CLI sweep, last component sweep) lands and **before** Task 7 (Pass A), surface a single brief status message to the user containing:

- Headline counts so far (R / P / D / W / O across all six components).
- Any borderline Deferred-by-design calls — list with one-line rationale per finding so the user can correct before they lock in.
- Any new findings opened, grouped by component.
- Any Resolution-claim regressions found (Resolved finding downgraded to Open).

Wait for user response before starting Task 7. If the user wants changes, fix in-place via additional commits before continuing.
