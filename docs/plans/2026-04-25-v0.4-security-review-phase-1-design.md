# 2026-04-25 — v0.4 holistic security review (Phase 1: audit) — design

- Status: Draft
- Phase: 1 of 2 (audit-first; remediation is Phase 2, separate spec)
- Target output: three durable artefacts under `docs/security/` plus a CI tooling workflow

## 1. Summary

Paddock has shipped a substantial security architecture across v0.1–v0.4 — capability-scoped admission, broker-mediated credential issuance, per-run egress proxy with TLS MITM, transparent and cooperative interception, AuditEvent CRD, default-deny egress with bounded discovery windows, in-container credential delivery as opt-in-with-reason. None of this has had a holistic security review.

This design covers Phase 1 of that review: an audit-first phase that produces a refreshed threat model, a security audit findings document, and a test-gap inventory — without making any code changes. Phase 2 (remediation of findings, new tests, hostile harness, additional E2E scenarios) is a separate spec, brainstormed once Phase 1's findings are in hand.

## 2. Goals and non-goals

**Goals.**

1. Produce a durable threat model for Paddock that future audits can cite without rewriting.
2. Identify security findings in the v0.4 architecture, design, and implementation, prioritised so Phase 2 has a clear remediation backlog.
3. Inventory the security-relevant gaps in the test suite (unit, integration, E2E) using three complementary views — threat-coverage matrix, per-component coverage, attack-scenario list.
4. Wire security-focused static analysis tooling into CI as a baseline-and-gate, so regressions are caught.
5. Specify (but not implement) a hostile harness ("evil paddock-echo") that Phase 2 will use to drive realistic E2E adversarial tests.

**Non-goals (deferred to Phase 2 or later):**

- Code changes to fix findings.
- New tests, the hostile harness image, or expanded E2E coverage.
- Bridges component (`paddock-bridges`) design or audit — not yet built.
- Performance/DoS modelling under load.
- Concurrency / lifecycle E2E scenarios beyond what's already in the suite.
- Full SBOM review, SLSA provenance, signed releases (beyond what `govulncheck` covers).

## 3. Threat-model approach

The audit anchors on a refreshed and formalised threat model produced as the first deliverable of Phase 1. The existing threat model in spec 0002 §2 (untrusted agent, untrusted prompt/workspace, operator misconfiguration) is preserved and expanded to include:

- **Co-tenant attacker** — another tenant's namespace in the same cluster.
- **Compromised paddock-system component** — CVE in controller, broker, or proxy.
- **Supply-chain attacker** — harness image, Go module, MCP server.
- **Cluster operator** — assumed trusted, documented as an explicit trust assumption.
- **Lifecycle / teardown attacker** — finalizer races, leftover state after run completion.

The threat model document is *defence-aware but not defence-evaluating* — it states what the defences are; the audit is what evaluates whether they actually hold. This split keeps the threat model durable across releases (defences change; threats don't).

## 4. Audit methodology

Four parallel work streams, converging into the findings document.

### 4.1 Manual code review (Stream 1)

Per-component review pass against threat model + ADRs + spec invariants. Order:

1. `internal/broker` — trust root.
2. `internal/proxy` — runtime defence.
3. `internal/webhook` + `internal/policy` — admission.
4. `internal/controller` — biggest component; benefits from others reviewed first.
5. Workspace seed paths in controller — narrow surface.
6. CLI credential paths in `internal/cli` — last; mostly UX, but the v0.1.1 shell-injection fix shows the surface is real.

For each component, the reviewer walks every STRIDE cell in the threat model that touches the component, every applicable ADR, and every spec-level invariant (e.g., spec 0002 §4 "Key invariants", spec 0003 §3.4 admission rules). Findings are written into the audit doc as identified.

### 4.2 Static analysis tooling (Stream 2)

Tools, with their purpose and gating posture:

| Tool             | Purpose                                | CI gate                          |
|------------------|----------------------------------------|----------------------------------|
| `gosec`          | Security-focused Go linter             | Fail on new findings; baseline   |
| `govulncheck`    | Known CVEs in module graph             | Fail on any finding              |
| `staticcheck`    | Logic / correctness                    | Fail on new findings; baseline   |
| `golangci-lint`  | Bundle (errcheck, bodyclose, noctx, …) | Fail on new findings; baseline   |
| `trivy`          | Container image vulns                  | Fail on HIGH/CRITICAL            |
| `kube-linter`    | Helm chart + manifests                 | Fail on new findings; baseline   |

Each tool's configuration committed to the repo; baselines accepted on first merge; the audit doc records what was triaged-and-suppressed with rationale.

### 4.3 STRIDE walkthroughs per trust boundary (Stream 3)

For each of the seven trust boundaries (see threat model §4), a structured walkthrough produces an appendix in the findings doc. Each walkthrough covers what crosses the boundary, what enforces it, and STRIDE-per-boundary (Spoofing, Tampering, Repudiation, Information disclosure, DoS, Elevation of privilege) — what's the threat, what's the defence, is it sufficient, evidence.

### 4.4 Attack-tree appendices (Stream 4)

Three high-value targets get attack-tree treatment:

- **Goal: agent exfiltrates a real upstream credential.**
- **Goal: agent reaches a non-allowlisted host.**
- **Goal: tenant A reads tenant B's workspace or broker bearer.**

Each tree's leaves either point to a defence (good) or become a finding (gap).

## 5. Deliverables

Phase 1 produces three documents and one CI workflow.

### 5.1 `docs/security/threat-model.md`

Durable artefact. Structure:

1. **Scope and assumptions** — explicitly in/out of scope; trust assumptions.
2. **Assets** — long-lived upstream secrets, broker-issued bearers, workspace contents, AuditEvent stream, prompts, MITM CA private key.
3. **Actors** — the eight categories from §3 above; per-actor capabilities, motivation, expected attacks, current defences.
4. **Trust boundaries** — diagram + table covering: cluster operator ↔ paddock-system; paddock-system ↔ run namespace; run pod ↔ broker; run pod ↔ proxy sidecar; run pod ↔ external internet; broker ↔ upstream Secrets; workspace seed Job ↔ git host.
5. **STRIDE-per-boundary table** — the spine of the audit walkthroughs.

### 5.2 `docs/security/2026-04-25-v0.4-audit-findings.md`

Date-stamped + version-stamped. Future audits get their own dated doc; this one is not overwritten. Structure:

1. **Executive summary** — half a page: scope, period, reviewer, method, headline counts, top three to fix.
2. **Findings list** — each finding numbered `F-NN`, with the per-finding structure below.
3. **Severity scheme reference** — canonical for this audit and successors.
4. **Static-analysis tooling summary** — one section per tool: baseline, top-N, false-positives suppressed, findings extracted.
5. **Appendices** — STRIDE walkthroughs (one sub-section per trust boundary), attack trees (one sub-section per high-value target from §4.4), references (links to threat model, ADRs cited, specs cited).

**Per-finding structure:**

```markdown
### F-NN — Title

| Field          | Value                                                    |
|----------------|----------------------------------------------------------|
| Component      | broker / proxy / controller / webhook / policy / workspace / cli |
| Threats        | T-N references                                           |
| Severity       | Likelihood: L/M/H · Impact: L/M/H · Risk: L/M/H          |
| Priority       | P0/P1/P2/P3                                              |
| Test coverage  | covered / partial / gap → TG-NN                          |

**Description.** What the issue is.

**Evidence.**
- `path/to/file.go:LL–LL`
- ADR-NNNN §X
- E2E scenario name

**Recommendation.** What Phase 2 should do.

**Test gap correlation.** Link to test-gap inventory rows.
```

**Severity scheme** — OWASP-style risk = Likelihood × Impact, with explicit L/M/H criteria. The criteria below are the *seed content* for the findings doc's `§3 Severity scheme reference`; they're written here so the design is self-contained and refined in place during the audit when concrete examples force the wording. Priority (P0/P1/P2/P3) is *separate* from severity: severity = "how bad if exploited"; priority = "how soon we should fix". The split keeps Phase 2 planning honest.

Likelihood criteria:
- High = an attacker who already has tenant-namespace access can trigger this with default Paddock config.
- Medium = requires a misconfiguration, a co-tenant, or non-default settings.
- Low = requires a specific chain of conditions or actor capabilities (e.g., compromised paddock-system).

Impact criteria:
- High = credential disclosure, cross-tenant data leak, audit suppression, or root-equivalent escalation.
- Medium = single-tenant data leak, denial of service against one run, audit completeness gap.
- Low = informational, hardening, defence-in-depth.

Priority criteria:
- P0 = drop everything; ship a fix or workaround for the next release.
- P1 = must fix before v1.0.
- P2 = should fix; schedule into a milestone.
- P3 = nice-to-have / hardening / track-but-don't-block.

### 5.3 `docs/security/2026-04-25-v0.4-test-gaps.md`

Three views over the same gap set, plus the hostile harness specification.

**§1 Threat-coverage matrix.** Rows = threats from the threat model; columns = test layers (unit, integration with envtest, E2E); cells = test names + coverage status. Each gap cell links to a `TG-NN` row in §3.

**§2 Per-component coverage view.** For each component, list of security-critical paths with coverage assessment. Developer-facing — used when working on a component.

**§3 Attack-scenario list.** The richest section. Each entry is a concrete adversarial scenario as a test specification with: scenario, expected behaviour, realism level (real cluster yes/no, real CNI yes/no, real upstream yes/no/stub, hostile harness yes/no), test layer, related findings, Phase 2 effort estimate. Detailed enough that Phase 2 can implement directly without re-design.

**§4 Hostile harness ("evil paddock-echo") specification.** A test-only image based on `paddock-echo`, with CLI flags for specific attacks (`--connect-raw-tcp`, `--read-secret-files`, `--probe-broker`, `--bypass-proxy-env`, `--exfil-via-dns`, `--read-other-tenant-pvc`, …). Specified here once because it's used by multiple `TG-NN` entries; ships in `images/evil-echo/` in Phase 2.

**§5 Test-realism dimensions.** For E2E entries, explicit declaration of which axes are exercised: real CNI, real upstream, hostile harness, concurrency. The audit notes where the current E2E suite scores low and recommends:

- **Real CNI everywhere (must-have).** Today's Kind suite runs whatever CNI ships by default; NetworkPolicy enforcement may be a no-op in test. Phase 2 swaps in Cilium or Calico in Kind.
- **Hostile harness for runtime-isolation tests (must-have).** Drives the realistic adversarial scenarios.
- **Real upstream opportunistically.** Where the assertion depends on it (substitution correctness); stub elsewhere.
- **Concurrency / lifecycle (flagged for Phase 2).** Not a Phase 1 deliverable.

**Cross-references.** Findings cite `TG-NN`; test-gap rows cite `F-NN`. Bidirectional links keep the two artefacts coherent.

### 5.4 `.github/workflows/security.yml`

CI workflow running the static-analysis tooling on every PR. Configurations committed (`.gosec.yaml`, `.golangci.yml`, `trivy.yaml`, `kube-linter.yaml`). Baselines accepted on first merge; new findings gate. Phase 1 *configures and gates*; remediating the noise is part of the audit; persistent items become findings.

## 6. Sequencing

Five rough stages with overlap. Total estimate ~3 weeks of focused work; no hard date.

1. **Threat model first (1–2 days).** End-to-end. Stable before moving on.
2. **Static analysis tooling baseline (1 day).** Install, configure, capture baseline. Runs early because (a) noise needs triage time, (b) output informs which manual-review areas to prioritise, (c) the CI workflow takes iterations to settle.
3. **Manual review per component (1–2 weeks).** Order: broker → proxy → webhook+policy → controller → workspace seed → CLI.
4. **STRIDE walkthroughs and attack trees (3–4 days).** After the manual pass, so walkthroughs aren't blind. May surface findings the linear pass missed.
5. **Test gap inventory (2–3 days).** Last; depends on everything else.

The findings doc is "live" during the audit — appended to as findings are identified, not produced at the end.

## 7. Exit criteria

Phase 1 is done when all of:

1. `threat-model.md` is committed and self-consistent (§1–§5 populated, no TBDs).
2. `2026-04-25-v0.4-audit-findings.md` is committed with: executive summary, ≥0 findings (could legitimately be empty if the project is in great shape — that becomes a finding-of-its-own with rationale), severity scheme reference, tooling summary, all seven STRIDE walkthroughs as appendices, all three attack-tree appendices.
3. `2026-04-25-v0.4-test-gaps.md` is committed with all three views populated and the hostile-harness spec in §4.
4. `.github/workflows/security.yml` is merged with all six tools running on every PR; baselines accepted; new findings gate.
5. Cross-references between the findings doc and test-gap doc are consistent: every `F-NN` with `test-coverage: gap` cites a `TG-NN` that exists; every `TG-NN` claiming to relate to a finding matches a real `F-NN`. A single grep'able consistency check at the end.
6. The user has reviewed all three docs and signed off (separate review gate at Phase 1 *end*, not at design approval time).

## 8. What Phase 1 explicitly does NOT produce

- Code changes to fix any finding (Phase 2).
- New tests, hostile harness image, or expanded E2E coverage (Phase 2).
- Bridges component design or audit (out of scope per §2).
- Performance/DoS modelling (out of scope per §2).

## 9. Out-of-scope items (audit boundaries)

- The Kubernetes substrate itself (cluster, kubelet, etcd, API server, CNI). We audit how Paddock uses them, not whether they're hardened.
- Harness images themselves. Boundary-only audit (what the harness can do/reach), not whether `@anthropic-ai/claude-code` has CVEs.
- `paddock-bridges` — not yet built.
- Cryptography primitives. We audit how we use them, not the algorithms.
- Supply chain on Go modules beyond `govulncheck`. SBOM, SLSA, signed releases — flag as findings if relevant; don't fix in this scope.
- Performance / DoS resilience under load. DoS *vectors* noted; load-testing is Phase 2 or later.
- Documentation, cookbooks, README. Treated as specifications of intent; flag where docs and code diverge but don't restructure docs.
- The CLI plugin (`kubectl-paddock`). Audit only the parts that touch credentials, secrets, or shell-out to user input.

## 10. Risks and how they're handled

- **Audit reveals fundamental design flaw.** Documented as a finding with `priority: P0` and a recommendation that may be "redesign X before v1.0". The redesign brainstorm is its own work item, not part of Phase 2 test-writing.
- **Findings count explodes (>50).** Triage by priority; Phase 2 takes P0+P1; P2+P3 deferred with their own design.
- **Static-analysis tooling produces unmanageable noise.** Configurable; suppress noisy categories with rationale recorded in the tool config files and called out in §5.4 of the findings doc.
- **Threat model needs revision mid-audit.** Edits land directly in `threat-model.md` with a one-line note in the audit doc explaining the change. The threat model is allowed to evolve.

## 11. Open questions

None blocking. The L/M/H criteria seeded in §5.2 are refined in place in the findings doc's `§3 Severity scheme reference` once the audit produces concrete examples that force precise wording.
