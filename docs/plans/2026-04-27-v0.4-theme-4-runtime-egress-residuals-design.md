## Context

Theme 4 of the v0.4 security follow-up roadmap closes the runtime-egress half of the post-Phase-2g recheck. All six findings live in the proxy connection lifecycle (`internal/proxy/`) and the iptables-init rule set (`cmd/iptables-init/`), with a thin slice in the controller's pod-spec generator (`internal/controller/pod_spec.go`) for sidecar UID pinning and init-container args. None require changes to the broker hot path, the admission webhooks (Theme 3 already closed those), or the CRD APIs.

The findings interlock: F-20 (broad RFC1918 RETURN in transparent mode) and F-19 residual (public-internet egress in cooperative mode) both name the same attack — agent reaches a destination without the proxy seeing it — from different layers. F-22 (DNS rebinding undetected) is the third leg of the same triad: even when the agent goes through the proxy, the IP it ultimately dials may not be the one the proxy authorized. F-26 (no connection cap, slow-loris) and F-28 (forged-leaf cache unbounded) are per-tenant DoS hardenings that compound the trust the proxy must hold against a hostile agent. F-27 (default flag combinations silently disable audit) is the operator-side footgun.

Pre-1.0, breaking changes are accepted in place rather than flag-aliased (CLAUDE.md). This theme renames `cmd/iptables-init`'s `--proxy-uid` flag to `--bypass-uids`, removes the `--mode` default in `cmd/proxy`, and shifts cooperative mode from "supported equal" to "loud opt-in single-point-of-trust." Operators discover the flag rename and the refuse-to-start gates on first deploy of the new binary.

## Goals

- Close F-19 (residual), F-20, F-22, F-26 (residual), F-27, F-28 in a single PR, commit-decomposed per finding so each `feat(security)!:` Conventional Commit lands its own breaking-change marker for `release-please`.
- Convert cooperative mode from a coequal alternative to a documented loud opt-in: refuse-to-start when chosen by default, WARN when active, AuditEvent at startup, threat-model + ADR rewrite making single-point-of-trust posture explicit.
- Make transparent mode actually bypass-resistant: drop the broad RFC1918 RETURN in favor of UID-based RETURN for sidecars, defend the dial path against DNS rebinding (both the `/etc/hosts`-style agent-side override and the cluster-DNS-as-attacker-vector cases), reject private/cluster-internal IPs at the proxy regardless of how they arrive.
- Bound proxy resource use under a hostile agent: connection cap with accept-then-reject behavior, slow-loris timeouts, bounded LRU on the forged-leaf cache, observability via Prometheus metrics.
- Reuse Phase 2 patterns: idle-timeout's `deadlineExtendingReader` composition (Phase 2g), fail-closed-on-audit-failure (Phase 2c), per-run NetworkPolicy already covers the cluster-CIDR carve-out (Phase 2a), `Owns()` registration (Phase 2d).

## Non-goals

- **Sibling-Pod proxy refactor** (F-19 audit option A — agent and proxy in different Pods sharing nothing). The structural fix for cooperative mode's same-Pod-NetworkPolicy-can't-distinguish problem. Theme-sized in its own right; deferred to v1.0 if the loud-opt-in shape proves insufficient.
- **QUIC / UDP / HTTP/3 interception.** iptables `REDIRECT` only catches TCP today; an agent using HTTP/3 over UDP/443 bypasses everything regardless of cooperative-vs-transparent. Mitigation requires either a UDP-aware proxy (substantial new code) or a NetworkPolicy deny on UDP/443 to force TCP fallback. Tracked as a follow-up finding (new F-NN to be opened post-Theme-4).
- **Generic TCP byte-shuttle for non-TLS protocols** (SSH, postgres, redis). Additive feature; the proxy is HTTPS-only today. Allowlist + raw `io.Copy` is feasible but separate work.
- **Connection rate-limiting per-source IP.** The proxy lives in the Pod netns; the source is always loopback (the agent). Process-wide concurrent-connection cap covers the realistic threat.
- **Operator-tunable LRU cap, timeout values, etc.** Constants in code unless an explicit deployment hits a wall. Adding flags is cheap later.

## Open-question decisions (resolved at design time)

Four open questions were deferred from the roadmap memo plus the live brainstorming session on 2026-04-27. Decisions taken at design review:

1. **F-19 residual shape — sibling-Pod refactor vs document gap vs admission gate.** Keep cooperative mode; treat it as a documented loud opt-in (audit option B + C combined). Sibling-Pod refactor (option A) deferred to v1.0. The BrokerPolicy `spec.interception.transparent` / `cooperativeAccepted` opt-in scaffolding already exists (api/v1alpha1, webhook, controller's `resolveInterceptionMode`); Theme 4 adds the proxy-side loudness (refuse-to-start without explicit `--mode`, WARN log, startup AuditEvent) and the threat-model + ADR rewrite.
2. **F-20 RFC1918 strategy — per-target ALLOW vs UID-based RETURN vs hybrid.** UID-based RETURN for sidecars (audit option B). Pin `runAsUser` on the adapter and collector containers so iptables-init can RETURN sidecar-originated traffic by owner-UID, then drop the broad RFC1918 RETURN entirely (loopback `127/8` stays). Agent traffic gets REDIRECTed for *every* destination on TCP/80 and TCP/443; the proxy enforces by BrokerPolicy. Cleanest defense-in-depth and reuses the existing `--uid-owner=1337` mechanism. Per-target ALLOW (option A) leaves a fragile tie between iptables rules and dynamic IPs (apiserver IPs change at node rotation; broker ClusterIP changes on Service recreate); option B avoids the coupling.
3. **F-22 layer ordering — Layer 1 only / Layer 2 only / both / sequential.** Both layers in this theme. Layer 2 (denied-CIDR check on origIP, IP-literal CONNECT, post-resolution IPs) is the realistic DNS-rebinding defense — catches the cluster-DNS-as-attacker-vector case where the proxy's own resolver returns the rebound address. Layer 1 (proxy re-resolves SNI, compares to agent's SO_ORIGINAL_DST) catches the narrower `/etc/hosts`-style agent-side override. The marginal cost of Layer 1 is small once we're in the dial path for Layer 2; sequential ordering would be busywork.
4. **F-26 connection-cap shape and value.** Process-wide accept-then-reject at 256 concurrent connections. Cooperative path returns `503 Service Unavailable` with audit; transparent path closes with TCP RST (`SetLinger(0)`) with audit. `--max-connections=256` flag. Memory math: ~80 KiB per connection (TLS state + two goroutine stacks); 256 conns ≈ 20 MiB inside the proxy's 128 MiB limit. Cap surfaced as a flag; timeouts (`ReadTimeout=60s`, `WriteTimeout=90s`, `IdleTimeout=60s`, `MaxHeaderBytes=16<<10`) as constants in the proxy package.

## Components

### F-19 residual — cooperative as documented loud opt-in

`cmd/proxy/main.go`:

- `--mode` loses its default; `flag.StringVar(&mode, "mode", "", "...")`. New startup gate refuses to start when empty:
  ```go
  if mode == "" {
      setupLog.Error(nil, "--mode must be set explicitly: 'transparent' or 'cooperative'")
      os.Exit(1)
  }
  ```
- New flag `--interception-acceptance-reason` (string, default `""`). Required when `--mode=cooperative`; refuse-to-start otherwise. Carries the BrokerPolicy `cooperativeAccepted.reason` from the controller for audit-trail purposes.
- New flag `--deny-cidr` (CSV string, default `""`). Required for non-trivial deployment; CIDRs from controller (cluster pod + service + RFC1918 + link-local).
- Cooperative-mode startup logs at WARN with explicit text:
  ```
  WARN  proxy  cooperative interception is single-point-of-trust — F-19 residual:
                a hostile agent that ignores HTTPS_PROXY can dial the public internet
                directly. Transparent mode is the recommended posture; cooperative is
                appropriate only with a trusted agent image.
                accepted-by: <BrokerPolicy reason>
  ```
- Cooperative-mode startup emits one AuditEvent immediately after audit-sink construction, before listener bind:
  - `Kind=interception-mode-cooperative-accepted`
  - `Decision=granted` (the cooperative posture has been admitted by BrokerPolicy)
  - `Reason=<--interception-acceptance-reason value>`
  - `MatchedPolicy=<BrokerPolicy name from the reason metadata>` (best-effort; if reason doesn't carry it, leave empty)
  - Standard run-name / run-namespace labels.
- New `paddockv1alpha1.AuditKindInterceptionModeCooperativeAccepted` enum value (api/v1alpha1) with admission validation in the AuditEvent webhook.

`docs/security/threat-model.md` B-4 (Tampering) and B-5 (Spoofing) rows replace the existing "Cooperative requires opt-in" cell with: "Cooperative is single-point-of-trust — agent must honour HTTPS_PROXY. F-19 residual: NetworkPolicy cannot distinguish agent from proxy traffic in same-Pod cooperative mode; transparent mode is the recommended posture for hostile-agent threat model. Sibling-Pod refactor tracked for v1.0." Add citation to AuditKindInterceptionModeCooperativeAccepted as the audit-trail anchor.

`docs/adr/0013-proxy-interception-modes.md` Phase update appended at the bottom of the file (per CLAUDE.md ADR-flat convention):
> **Phase 2h Theme 4 update (2026-04-27).** Cooperative mode is retained for environmental compatibility (PSS-restricted-enforced namespaces, eBPF-only / non-iptables clusters) but reclassified as single-point-of-trust loud opt-in: refuse-to-start without explicit `--mode`, WARN log on every cooperative startup, AuditEvent emitted with the BrokerPolicy reason. Transparent mode is the recommended posture for hostile-agent threat models. F-19 residual (agent-vs-proxy NetworkPolicy distinction in same-Pod cooperative mode) is structurally documented rather than fixed; the sibling-Pod refactor option is deferred to v1.0.

`docs/specs/0002-broker-proxy-v0.3.md` §4 invariant 1 gains a footnote:
> Footnote: in cooperative mode this invariant is enforced by the agent's honour of `HTTPS_PROXY` and is bypassable by a hostile agent. Transparent mode enforces it via iptables REDIRECT. See ADR-0013 §7.2 Phase 2h Theme 4 update.

NetworkPolicy shape **unchanged** — the Phase 2a CIDR-exclusion stands; the public-internet residual is documented, not closed structurally.

### F-20 — UID-based RETURN for sidecars; drop broad RFC1918

`internal/controller/pod_spec.go`:

- New constants alongside `proxyRunAsUID`:
  ```go
  adapterRunAsUID   = 1338
  collectorRunAsUID = 1339
  ```
  UIDs picked next to `proxyRunAsUID=1337`. No allocation policy beyond "next free integer in the 133x neighborhood"; if a future sidecar needs another UID, append at 1340 etc.
- `buildAdapterContainer` and `buildCollectorContainer` set `SecurityContext.RunAsUser = ptrInt64(adapterRunAsUID)` / `ptrInt64(collectorRunAsUID)`. Image-baked UIDs that conflict will fail at container start; document in CHANGELOG.
- `buildIPTablesInitContainer` passes:
  ```
  --bypass-uids=1337,1338,1339
  ```
  alongside the existing `--proxy-port=15001` and `--ports=80,443` args.

`cmd/iptables-init/main.go`:

- New `--bypass-uids` flag (CSV string, required). `--proxy-uid` removed; pre-1.0, no alias.
- `privateRanges` slice deleted; loopback (`127.0.0.0/8`) RETURN preserved as a single hardcoded entry (the proxy's probe endpoints + any future loopback-only services).
- The owner-RETURN loop replaces the single-UID rule:
  ```go
  for _, uid := range bypassUIDs {
      appendIfMissing(runner, "-t", "nat", "-A", defaultChainName,
          "-m", "owner", "--uid-owner", uid, "-j", "RETURN")
  }
  ```
- `--bypass-uids` parsing rejects empty list, non-integer entries, duplicate UIDs (defense-in-depth against operator typos).

Behavior change consequences:

- Agent traffic to broker ClusterIP: REDIRECTed to the proxy. Proxy validator denies (no BrokerPolicy egress grant for the broker's ClusterIP/SNI). AuditEvent emitted.
- Agent traffic to kube-apiserver IP: REDIRECTed. Same denial path.
- Agent traffic to co-tenant Pod IP: REDIRECTed. Denial.
- Adapter traffic to kube-apiserver: RETURNed by adapter UID. Unchanged behavior.
- Collector traffic to kube-apiserver: RETURNed by collector UID. Unchanged behavior.
- Proxy upstream dials: RETURNed by proxy UID. Unchanged behavior.
- DNS (UDP/53): unaffected; iptables only intercepts the `--ports` list (TCP/80,443).

### F-22 — DNS rebinding, both layers

`internal/proxy/resolver.go` (new):

```go
type Resolver interface {
    LookupHost(ctx context.Context, host string) ([]net.IP, error)
}

type cachingResolver struct {
    inner *net.Resolver
    cache *lruCache[string, []net.IP]
    ttl   time.Duration
    sf    singleflight.Group
}
```

- Default impl wraps `net.DefaultResolver` (`*net.Resolver`); 30 s TTL; 256-entry LRU; `singleflight.Do(host, …)` coalesces concurrent lookups for the same name.
- IP literals short-circuit: `net.ParseIP(host) != nil` → returns `[]net.IP{ip}` without a lookup or cache touch.
- Returned IPs preserve resolver order (Go's `LookupHost` already does happy-eyeballs for IPv4/IPv6 mix); proxy dials sequentially through the slice.

`internal/proxy/deny_cidr.go` (new):

```go
type DeniedCIDRSet struct {
    nets []*net.IPNet
}

func ParseDeniedCIDRs(csv string) (*DeniedCIDRSet, error)
func (d *DeniedCIDRSet) Contains(ip net.IP) bool
```

- Parses comma-separated CIDR list; rejects malformed entries at startup.
- `Contains` walks the list (small N — ~6 entries; linear is fine).

`Server` gains two new fields populated from `cmd/proxy/main.go`:

```go
type Server struct {
    // ... existing ...
    Resolver     Resolver         // default: cachingResolver{net.DefaultResolver, 30s, 256}
    DeniedCIDRs  *DeniedCIDRSet   // nil means no denied-CIDR check (test only)
}
```

**Layer 2** (cooperative path, `internal/proxy/server.go::handleConnect`):

- Immediately after `splitConnectTarget`, check IP-literal in denied set:
  ```go
  if ip := net.ParseIP(host); ip != nil && s.DeniedCIDRs.Contains(ip) {
      // emit deny audit, 403, return
  }
  ```
- After validator allow, before dial: resolve via `s.Resolver.LookupHost(ctx, host)`, filter through denied-CIDRs:
  ```go
  resolved, err := s.Resolver.LookupHost(ctx, host)
  // ... err handling ...
  allowed := resolved[:0]
  for _, ip := range resolved {
      if !s.DeniedCIDRs.Contains(ip) { allowed = append(allowed, ip) }
  }
  if len(allowed) == 0 {
      // emit deny audit reason="denied-destination-cidr", 403
  }
  // pass `allowed[0]` (or iterate) to dialUpstreamTLS
  ```

**Layer 2** (transparent path, `internal/proxy/mode.go::HandleTransparentConn`):

- After `s.origDest(conn)` returns `origIP, origPort`:
  ```go
  if s.DeniedCIDRs.Contains(origIP) {
      // emit deny audit reason="denied-destination-cidr", abruptClose, return
  }
  ```
- Placed before the SNI peek — earlier rejection saves the handshake cost.

**Layer 1** (transparent only, `internal/proxy/mode.go`):

- After validator-allow, before `mitmTransparent`:
  ```go
  resolved, err := s.Resolver.LookupHost(ctx, sni)
  // ... err handling: deny on resolver error ...
  if !ipSliceContains(resolved, origIP) {
      // emit deny audit reason="dns-rebinding-mismatch", abruptClose
  }
  ```
- The check comes after Layer 2 so a clearly-bad `origIP` (e.g. RFC1918) is rejected with the more specific reason rather than `dns-rebinding-mismatch`.

`internal/proxy/mitm.go::dialUpstreamTLS` signature change:

```go
func (s *Server) dialUpstreamTLS(ctx context.Context, dialIP net.IP, dialPort int, serverName string) (net.Conn, error)
```

- Accepts a pre-resolved IP. `joinHostPortInt` becomes `net.JoinHostPort(dialIP.String(), strconv.Itoa(port))`.
- Cooperative caller iterates resolved IPs on dial failure (mirrors Go's happy-eyeballs but explicit).
- Transparent caller passes the SO_ORIGINAL_DST IP directly (already an IP).

AuditEvent reasons standardized:

- `denied-destination-cidr` — origIP / IP-literal CONNECT / post-resolution IP fell in the denied CIDR set.
- `dns-rebinding-mismatch` — transparent's origIP didn't match the proxy-resolved set for the SNI.
- `dns-resolution-failed` — the proxy's resolver returned an error; deny rather than fall through.

### F-26 residual — connection cap + slow-loris timeouts + metrics

`internal/proxy/limit.go` (new):

- `ConnLimiter` exposes `Acquire() (release func(), ok bool)`. Backed by a buffered channel of size N for an O(1) non-blocking acquire path.
- `LimitedListener` wraps `net.Listener.Accept`: on accept, attempts `Acquire`; on `!ok`, closes the accepted conn (with `SetLinger(0)` for transparent path) and increments the rejection counter, then re-Accepts. Caller never sees over-cap connections.
- For cooperative `http.Server`, `LimitedListener` wraps the listener handed to `httpSrv.Serve(ln)`; the over-cap rejection writes a synthetic `503 Service Unavailable` before close (small TLS overhead acceptable; the cap is a back-pressure signal not a hot path).

`cmd/proxy/main.go`:

- New `--max-connections=256` flag (`flag.IntVar`).
- Cooperative listener:
  ```go
  ln, err := net.Listen("tcp", listenAddr)
  // ...
  ln = NewLimitedListener(ln, maxConnections, "cooperative")
  httpSrv.Serve(ln)
  ```
- Transparent listener: same wrap inside `serveTransparent`.

`internal/proxy/server.go::handleConnect` (cooperative `http.Server` config):

```go
httpSrv := &http.Server{
    Addr:              listenAddr,
    Handler:           p,
    ReadTimeout:       60 * time.Second,
    WriteTimeout:      90 * time.Second,
    IdleTimeout:       60 * time.Second,
    ReadHeaderTimeout: 15 * time.Second,
    MaxHeaderBytes:    16 << 10,
}
```

(Currently set in `cmd/proxy/main.go`; values become package-private constants in `internal/proxy/server.go` and are applied via a `Server.HTTPServer(addr)` constructor so tests can read them.)

`internal/proxy/substitute.go::handleSubstituted` per-iteration deadline:

```go
for {
    _ = clientTLS.SetReadDeadline(time.Now().Add(idleTimeout))
    req, err := http.ReadRequest(bufReader)
    // ...
}
```

The existing `deadlineExtendingReader` covers byte-shuttle paths; this addition covers the request-loop path that reads framed HTTP/1.1 messages from the decrypted TLS stream.

### F-27 — refuse to start on silent-noop

`cmd/proxy/main.go` startup gates, executed in order before listener bind:

1. `--mode` empty → fatal with explicit error.
2. `--mode=cooperative` AND `--interception-acceptance-reason=""` → fatal ("cooperative mode requires --interception-acceptance-reason; controller passes the BrokerPolicy reason").
3. `--allow=""` AND `--broker-endpoint=""` AND `--disable-audit=false` → fatal ("default-deny + no audit is never intentional; either set --allow / --broker-endpoint, or set --disable-audit explicitly to acknowledge").
4. Audit-sink constructor returns the noop sink AND `--disable-audit=false` → fatal ("audit unavailable: <reason>; set --disable-audit explicitly if intentional").

`buildAuditSink` refactored to return `(AuditSink, sinkType)`:

```go
func buildAuditSink(disabled bool, runName, runNamespace string) (proxy.AuditSink, string, error)
// sinkType ∈ {"client", "noop"}; error is the unrecoverable construction error.
```

- The startup gate inspects `sinkType` to drive the refuse-to-start.
- `paddock_proxy_audit_sink{type=<sinkType>} = 1` is set after the gate passes.

### F-28 — bounded LRU on forged-leaf cache

`internal/proxy/ca.go`:

- Replace `cache map[string]*tls.Certificate` with:
  ```go
  type lruEntry struct {
      host string
      cert *tls.Certificate
      el   *list.Element
  }

  type MITMCertificateAuthority struct {
      // ... existing CA fields ...
      mu       sync.Mutex
      cache    map[string]*lruEntry
      order    *list.List // front = most recently used
      cap      int
      sf       singleflight.Group
  }
  ```
- `forge(host)` becomes:
  ```go
  func (ca *MITMCertificateAuthority) forge(host string) (*tls.Certificate, error) {
      // singleflight on key=host — concurrent forges for the same host coalesce
      v, err, _ := ca.sf.Do(host, func() (any, error) {
          ca.mu.Lock()
          if e, ok := ca.cache[host]; ok {
              ca.order.MoveToFront(e.el)
              ca.mu.Unlock()
              return e.cert, nil
          }
          ca.mu.Unlock()
          cert, err := ca.signLeaf(host)
          if err != nil { return nil, err }
          ca.mu.Lock()
          defer ca.mu.Unlock()
          // re-check after sign: another goroutine may have populated
          if e, ok := ca.cache[host]; ok {
              ca.order.MoveToFront(e.el)
              return e.cert, nil
          }
          el := ca.order.PushFront(host)
          ca.cache[host] = &lruEntry{host, cert, el}
          if ca.order.Len() > ca.cap {
              oldest := ca.order.Back()
              if oldest != nil {
                  delete(ca.cache, oldest.Value.(string))
                  ca.order.Remove(oldest)
              }
          }
          return cert, nil
      })
      // ... typed return ...
  }
  ```
- Default cap = 1024 (ample headroom for wildcard-CDN realism). Configurable via `MITMCertificateAuthority.cap` field; loaders default-set 1024.
- No flag surfaces this today; `--leaf-cache-size` follows if a deployment hits the cap.

## Risks and rollback

- **Sidecar UID pin breaking image builds.** Adapter or collector images that bake a different UID will fail container start. Verified by unit test (image-baked UID check) + e2e suite Pod startup. Rollback: revert the `RunAsUser` pin; the iptables-init falls back to a single `--bypass-uids=1337` (proxy-only) and the broad RFC1918 RETURN can be reinstated as a temporary measure if the sidecar regression is in production.
- **F-22 false positives.** A legitimate upstream's authoritative DNS may return different IPs to the proxy than to a hypothetical agent path (anycast, region-specific records). The proxy's resolution is what the validator authorized; mismatch denial is correct, but for high-volume CDN-fronted upstreams the cache TTL of 30 s may need tuning. `paddock_proxy_connections_rejected_total{reason="dns_rebinding_mismatch"}` provides the alerting surface.
- **F-26 cap too low.** A workload that legitimately holds many TLS streams (MCP servers, multi-host scrapers) may hit 256. The flag is operator-tunable. Default-256 is a starting point; observability via `paddock_proxy_active_connections` lets operators size up.
- **F-20 NetworkPolicy + iptables interaction subtleties.** Agent traffic to the cluster's own DNS service IP on TCP (rare; UDP is the default) is no longer RFC1918-RETURNed and gets REDIRECTed → proxy fails to MITM kube-dns → agent's TCP DNS fails. Mitigation: NetworkPolicy already permits TCP/53 to kube-dns from the run Pod; the proxy will see TCP/53 traffic on its `--ports=80,443` only — i.e., it doesn't, because TCP/53 isn't in `--ports`. iptables only intercepts the `--ports` list. This risk is theoretical, not real.
- **F-27 refuse-to-start blocking existing deploys.** An operator who deployed the proxy with `--allow=""` and no broker-endpoint in a one-off context gets a hard fail on upgrade. Mitigation: explicit error message instructs the operator on the `--disable-audit` opt-out path. CHANGELOG entry highlights this.

Single-commit rollback: each commit is independent except the F-20 commit depends on F-22's deny paths existing. Reverting F-20 alone is safe; reverting F-22 requires reverting F-20 first (or the e2e specs will fail).

## Out-of-scope follow-ups (tracked, not in this PR)

- **F-NN (new) — UDP / QUIC / HTTP/3 interception.** iptables REDIRECT is TCP-only; an agent using HTTP/3 (UDP/443) bypasses the proxy regardless of mode. Mitigation options: NetworkPolicy deny on UDP/443 (forces TCP fallback), UDP-aware proxy (substantial new code). To be filed against the audit findings doc as a recheck-2026-04-27 surface.
- **F-NN (new) — Generic non-TLS TCP byte-shuttle.** Today the proxy only handles HTTPS; agents that need SSH, postgres, redis, etc., have no proxy path. Allowlist + raw `io.Copy` is feasible and complements substitute-auth-style byte-shuttle for non-MITMable protocols (SSH host-key pinning).
- **Sibling-Pod proxy refactor (F-19 audit option A).** Agent and proxy in different Pods sharing nothing, communicating via Service. Closes F-19 residual structurally. Theme-sized; v1.0 candidate.
- **Agent → broker direct on TCP/8443 not closed by F-20.** iptables intercepts only `--ports=80,443`; the broker's TCP/8443 falls through. Phase 2a's NetworkPolicy egress rule permits run-Pod traffic to broker namespace+podselector (pod-level selector — admits agent traffic too). Today the defense is broker authn (audience-bound SA token). Closing the NP-level hole requires either narrowing the rule (no NP-native way to filter by source UID) or moving the proxy to a sibling Pod. Tracked alongside the sibling-Pod refactor.
- **Operator-tunable LRU cap, timeout values.** Add flags if a deployment hits the constants meaningfully.
- **F-NN (new) — `paddock_proxy_audit_sink{type="noop"}` alerting rule shipped in the chart.** The metric is in this theme; the recommended Prometheus alert rule is operator-side and follows in a chart-update PR.
