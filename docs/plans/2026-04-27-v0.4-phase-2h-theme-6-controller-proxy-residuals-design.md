## Context

Theme 6 of the v0.4 security follow-up roadmap is the catch-all for controller and proxy residuals that don't share a code surface large enough to anchor their own theme. Five findings from the 2026-04-26 recheck remain in scope:

- **F-01** — SSRF via broker endpoint (proxy → broker HTTP client).
- **F-41 (partial residual)** — `<run>-broker-creds` envFrom-injection key validation deferred from Phase 2d.
- **F-42** — `terminationGracePeriodSeconds` template-controlled with no upper bound.
- **F-44** — Source CA Secrets with missing/empty keys silently loop runs at `EgressConfigured=Pending` indefinitely.
- **F-56** — `paddock-iptables-init` built on EOL Alpine 3.20 base image.

Two of the five have been materially closed by the work that landed between the roadmap memo and this design:

- **F-01** is closed by Theme 3's `validateBrokerEndpoint` in `internal/brokerclient/brokerclient.go`. The proxy's `BrokerClient.NewBrokerClient` constructs via `brokerclient.New`, which now rejects any endpoint that isn't `https://<svc>.<ns>.svc[.cluster.local]:8443`, no userinfo, no path/query/fragment. The audit's recommendation §1 ("validate at NewBrokerClient that the endpoint is an HTTPS URL matching the expected in-cluster service pattern") is materially executed. Theme 6 only needs to add a guard test on the proxy side and update the audit-findings Status block.
- **F-44 (seed half)** was closed by Theme 1's `errSourceCAMisconfigured` + `errProxyCertPermanentFailure` plumbing in the Workspace reconciler. The HarnessRun reconciler still loops on the same misconfigurations; Theme 6 mirrors that pattern.

Pre-1.0, breaking changes are accepted in place rather than flag-aliased (CLAUDE.md). CHANGELOG entries land via individual `feat(security)!:` Conventional Commit markers picked up by `release-please`.

## Goals

- Close F-01 (verification + doc), F-41 residual, F-42, F-44 (run side), and F-56 in a single PR, commit-decomposed per finding so each Conventional Commit lands its own changelog entry and breaking-change marker for `release-please`.
- Reuse Phase 2 patterns wherever possible: Phase 2d `Owns()` + AuditEvent on tampering (F-41 residual); Phase 2e admission-rejects-out-of-range (F-42); Theme 1's terminal-condition error sentinels (F-44); Theme 1's digest-pinned third-party image precedent (F-56).
- No new admission machinery, no new shared subsystems — every change is local to its existing code path.

## Non-goals

- F-29 admission-side broker URL validation. Already shipped in Theme 3.
- F-51 workspace seed CA misconfig handling. Already shipped in Theme 1.
- F-26 connection-limit / back-pressure (Theme 4 — runtime egress residuals).
- Digest-pinning the harness/test alpine images (`evil-echo`, `harness-e2e-egress`, `harness-echo`). Separate cross-cutting cleanup; the iptables-init image is pinned here because it's security-critical (NET_ADMIN), matching the precedent set for `defaultSeedImage` in `workspace_seed_helpers.go:38–43`.
- Per-request URL re-validation for the broker client. The endpoint is set once at construction; nothing in production mutates it. Startup-time validation is the contract.
- An e2e spec for each finding. Unit + envtest is the primary verification surface; F-44's controller branches are exercised in envtest, F-56 is gated by `make trivy-images`, F-42's webhook layer by webhook unit tests.

## Open-question decisions (resolved at design time)

The Theme 6 section of the roadmap memo did not enumerate explicit deferred questions, but several decisions surfaced during design review on 2026-04-27:

1. **F-01 verification scope.** Treat F-01 as effectively resolved by Theme 3's `validateBrokerEndpoint`; do not re-do validation from a different angle. Add a single guard test on the proxy side (`TestNewBrokerClient_RejectsHostileEndpoints`) that asserts the proxy is wired to the validator. Update the audit-findings Status block from `Open` to `Resolved` with Verification: Heavy.
2. **F-42 cap value.** 300 seconds (5 minutes). Matches the audit's recommendation; tight enough to bound credential exposure (per-run MITM CA private key + broker bearer); broad enough for any realistic harness shutdown (`claude-code` and `echo` SIGTERM-clean within seconds). The speculative `proxy_tls.go:82` comment that mentioned a "24h cap" is updated in the same commit.
3. **F-42 layering.** Both admission rejection (HarnessTemplate + ClusterHarnessTemplate webhooks) and controller clamp (belt-and-braces in `buildPodSpec`). Phase 2e two-layer pattern. Cluster operators upgrading from a pre-2h release with templates carrying `terminationGracePeriodSeconds > 300` see the rejection on next Update; existing runs already in flight retain their pinned value (snapshot semantics).
4. **F-41 residual scope.** Strict-key prune in the existing `cachedBrokerCredentials` fast-path. When extras are detected, prune via a single `CreateOrUpdate` using the values already present in the Secret (no broker round-trip — values aren't being changed, only extras dropped) and emit a `broker-creds-tampered` AuditEvent. Cheaper than re-Issuing leases (which would burn PATPool slots for no security benefit).
5. **F-44 phase transition.** `Phase=Failed`, no requeue, on terminal CA misconfiguration. Mirrors Theme 1's Workspace pattern exactly. Operators see a clear terminal failure rather than the existing indefinite `Pending` loop. The trade-off — a transient operator misconfiguration becomes a Failed run — is acceptable because the misconfiguration would have manifested as Pending-forever anyway, and Failed is at least diagnosable.
6. **F-56 base image.** `alpine:3.22`, digest-pinned. 3.22 matches the audit's recommendation and the version used by every other alpine-based image in the repo (`evil-echo`, `harness-e2e-egress`, `harness-echo`). Digest pinning matches the security-critical posture set by `defaultSeedImage`. Trivy DB coverage for 3.22 is current; `make trivy-images` will give a confident clean reading rather than the "vulnerability detection may be insufficient" warning of 3.20.
7. **Commit decomposition.** One commit per finding, five commits total, each with its own `chore`/`feat`/`feat!`/`build` Conventional Commit marker. Matches the per-finding decomposition used by Themes 1, 2, and 3.

## Components

### §1 — F-01: SSRF closed by Theme 3 (`chore(security)`)

**No production code change.** Theme 3 (PR #69) added `validateBrokerEndpoint` in `internal/brokerclient/brokerclient.go::New`. The proxy's `BrokerClient` (`internal/proxy/broker_client.go::NewBrokerClient`) constructs via `brokerclient.New` at line 72, so the validator is on the proxy's only construction path.

**Guard test.** `internal/proxy/broker_client_test.go::TestNewBrokerClient_RejectsHostileEndpoints`. Table-driven; each case calls `NewBrokerClient(endpoint, "/tmp/token", "", "run-1", "ns-1")` and asserts the returned error is non-nil with a substring match on a shape-specific phrase (e.g., `"scheme must be https"`, `"endpoint must not contain userinfo"`, `"port must be 8443"`). Cases:

- `http://broker.paddock-system.svc:8443` (wrong scheme).
- `https://user:pw@broker.paddock-system.svc.cluster.local:8443` (userinfo present).
- `https://broker.paddock-system.svc.cluster.local:9443` (off-port).
- `https://evil.example.com:8443` (off-suffix; not `.svc` or `.svc.cluster.local`).
- `https://broker.paddock-system.svc.cluster.local:8443/redirect` (path component).
- `https://broker.svc.cluster.local:8443` (single-label hostname; rejected by the canonical-DNS-shape rule).

The happy-path control case (`https://paddock-broker.paddock-system.svc.cluster.local:8443`) is already exercised by existing proxy tests; not duplicated here.

**Audit-doc update.** `docs/security/2026-04-25-v0.4-audit-findings.md` F-01 status block:

- State: `Open` → `Resolved`.
- Verification: `Medium` → `Heavy`.
- Resolution paragraph: cites `validateBrokerEndpoint` callsite (`internal/brokerclient/brokerclient.go::New`) and the new proxy-side guard test, notes that recommendation §2 (compile-time pin) is intentionally not pursued because the validator already enforces shape-equivalence with a single in-cluster broker, and recommendation §3 (Pod securityContext env injection) is covered by Phase 2e's per-container PSS-restricted shape.
- Reviewer line preserved.

**Conventional commit.** `chore(security): F-01 closed by Theme 3 broker URL validation`.

### §2 — F-42: cap `terminationGracePeriodSeconds` at 300s (`feat(security)!`)

**Two-layer Phase 2e pattern.**

**Admission layer (`internal/webhook/v1alpha1/harnesstemplate_shared.go`).**

```go
const MaxTerminationGracePeriodSeconds = 300

func validateTerminationGracePeriodSeconds(v *int64, p *field.Path) field.ErrorList {
    if v == nil {
        return nil
    }
    if *v < 0 {
        return field.ErrorList{field.Invalid(p, *v, "must be non-negative")}
    }
    if *v > MaxTerminationGracePeriodSeconds {
        return field.ErrorList{field.Invalid(p, *v,
            fmt.Sprintf("must be <= %d (5 minutes); see F-42",
                MaxTerminationGracePeriodSeconds))}
    }
    return nil
}
```

The field path is `spec.defaults.terminationGracePeriodSeconds` for both `HarnessTemplate` and `ClusterHarnessTemplate`. The two webhooks (`harnesstemplate_webhook.go::ValidateCreate/Update` and `clusterharnesstemplate_webhook.go::ValidateCreate/Update`) call the helper from their Defaults validation step alongside the existing checks.

**Controller layer (`internal/controller/pod_spec.go:241–244`).**

```go
const maxPodGracePeriodSecs = 300

// in buildPodSpec:
grace := int64(defaultGracePeriodSecs)
if template.Spec.Defaults.TerminationGracePeriodSeconds != nil {
    grace = *template.Spec.Defaults.TerminationGracePeriodSeconds
}
if grace > maxPodGracePeriodSecs {
    grace = maxPodGracePeriodSecs
}
```

The clamp is belt-and-braces: an admission webhook downtime (or a templated-Run path that bypasses the webhook for any reason) cannot produce a grace period above the cap. No log emission on clamp — the operator-facing channel is admission, not run-time logs.

**Comment fix (`internal/controller/proxy_tls.go:82`).** The forward-looking comment "48h covers F-42's 24h cap on terminationGracePeriodSeconds plus margin" is updated to reference 300s; cert duration of 48h still wildly exceeds the cap, so the cert-duration constant doesn't move.

**Tests.**

- `internal/webhook/v1alpha1/harnesstemplate_webhook_test.go` and `clusterharnesstemplate_webhook_test.go` — new `TestValidateTerminationGracePeriodSeconds` table-driven cases: accept-when-unset, accept-at-0, accept-at-60 (default), accept-at-300, reject-at-301, reject-at-86400, reject-at-negative.
- `internal/controller/pod_spec_test.go` — new `TestBuildPodSpec_ClampsGracePeriod` exercising clamp behaviour: a template with `TerminationGracePeriodSeconds: 600` produces a Pod with `TerminationGracePeriodSeconds=300`.

**Conventional commit.** `feat(security)!: cap terminationGracePeriodSeconds at 300s (F-42)`. Breaking-change marker because templates with `> 300` are rejected on next write; release-please picks this up for the CHANGELOG.

```go
func (r *HarnessRunReconciler) ensureBrokerCA(ctx context.Context, run *paddockv1alpha1.HarnessRun) (bool, error) {
    if !r.brokerProxyConfigured() {
        return true, nil
    }
    src := &corev1.Secret{}
    err := r.Get(ctx, types.NamespacedName{
        Namespace: r.BrokerCASource.Namespace,
        Name:      r.BrokerCASource.Name,
    }, src)
    switch {
    case apierrors.IsNotFound(err):
        return false, nil
    case err != nil:
        return false, fmt.Errorf("reading source broker-CA Secret %s/%s: %w",
            r.BrokerCASource.Namespace, r.BrokerCASource.Name, err)
    }
    if len(src.Data[brokerCAKey]) == 0 {
        return false, errSourceCAMisconfigured
    }
    // existing copyCAToSecret + dst re-Get logic continues from here.
}
```

**Reconcile path wiring (`harnessrun_controller.go:380–420`).** The two error paths mirror `workspace_controller.go:215–305`:

For `ensureProxyTLS`:


### §4 — F-41 residual: strict-key prune of `<run>-broker-creds` (`feat(security)`)

**Tighten `cachedBrokerCredentials`.** The current logic (`broker_credentials.go:175–230`) verifies every required key is present and non-empty, then short-circuits. Add a third check: no extra keys.

```go
// after the existing per-required-key check at line 224-228:

// F-41 residual: detect and prune extra keys. The Phase 2d Owns()
// watch on the broker-creds Secret triggers a reconcile on tampering,
// but the cached fast-path returns ok=true without re-writing the
// Secret. A tenant who edits the Secret to add an extra key (e.g.,
// EXTRA_VAR=...) gets it consumed by the agent's envFrom mount until
// the next full Issue cycle.
if len(s.Data) > len(sortedReqs) {
    canonical := make(map[string][]byte, len(sortedReqs))
    pruned := make([]string, 0, len(s.Data)-len(sortedReqs))
    requiredNames := make(map[string]struct{}, len(sortedReqs))
    for _, c := range sortedReqs {
        requiredNames[c.Name] = struct{}{}
        canonical[c.Name] = s.Data[c.Name]
    }
    for k := range s.Data {
        if _, ok := requiredNames[k]; !ok {
            pruned = append(pruned, k)
        }
    }
    sort.Strings(pruned)
    secret := &corev1.Secret{
        ObjectMeta: metav1.ObjectMeta{
            Name:      brokerCredsSecretName(run.Name),
            Namespace: run.Namespace,
        },
    }
    if _, err := controllerutil.CreateOrUpdate(ctx, r.Client, secret, func() error {
        if err := controllerutil.SetControllerReference(run, secret, r.Scheme); err != nil {
            return err
        }
        secret.Type = corev1.SecretTypeOpaque
        secret.Data = canonical
        return nil
    }); err != nil && !apierrors.IsConflict(err) {
        // Transient API error — fall through to the full Issue cycle
        // rather than silently leaving extras in place.
        return false, nil, nil, false
    }
    if r.Audit != nil {
        r.Audit.EmitBrokerCredsTampered(ctx, run.Name, run.Namespace, pruned)
    }
}
```

The canonical map is built from the existing `s.Data` values (no broker round-trip; values aren't being changed, only extras dropped). On a transient API error during the prune, the code falls through to the full `Issue` cycle — which would re-write the Secret with the canonical key set — rather than silently leaving extras in place.

**Audit builder (`internal/auditing/builders.go`).** New `EmitBrokerCredsTampered(ctx, runName, runNamespace string, prunedKeys []string)`:

- Reason: `"broker-creds-tampered"`.
- Message: `"broker-creds Secret had unexpected keys; pruned: <comma-joined>"` — lists key names only, never values.
- Subject: HarnessRun (name, namespace).
- Severity: `Warning`.

**Tests (`internal/controller/broker_credentials_test.go`).** Three cases:

1. Steady state, no extras → fast-path returns `ok=true, found=true`, no `Update` call, no audit emission.
2. Extra key `EXTRA_KEY=foo` injected → fast-path detects, Secret rewritten with canonical key set, audit emission for `["EXTRA_KEY"]`, no broker `Issue` calls.
3. Required key blanked AND extra key added → falls through to full Issue cycle (existing behaviour for missing required key takes precedence; the prune branch is reached only when all required keys are populated).

**Conventional commit.** `feat(security): strict-key prune of <run>-broker-creds (F-41 residual)`. Not breaking — extras were never supposed to exist; the contract tightens but doesn't change for legitimate users.

## Data flow

Every Theme 6 change is local; no new cross-component contracts. The data-flow notes worth surfacing:

- **F-42 admission rejection** flows through the existing `field.ErrorList.ToAggregate().Error()` path. The `BrokerPolicyCustomValidator.audit` and `HarnessTemplateCustomValidator.audit` paths from Phase 2c emit `template-rejected` AuditEvents containing the aggregated field errors as the `Reason` payload. F-42's new error string ("must be <= 300 (5 minutes); see F-42") flows through unchanged.
- **F-44 terminal condition** flows through the existing `commitStatus` helper in the run reconciler. The new `Phase=Failed` transition triggers `recordPhaseTransition` for the existing phase-transition Prometheus metric.
- **F-41 residual prune** triggers an `Owns()`-watched Secret update (Phase 2d). The next reconcile cycle sees the Secret in its post-prune shape and the cached fast-path matches without re-pruning. No reconcile loop.

## Error handling

- **F-42 controller clamp.** No log emission on clamp; admission is the operator-facing channel.
- **F-44 reconcile-time errors.**
  - `errProxyCertPermanentFailure` → terminal `Failed/ProxyCAMisconfigured`.
  - `errSourceCAMisconfigured` → terminal `Failed/BrokerCAMisconfigured`.
  - Any other reconcile error from `ensureProxyTLS` / `ensureBrokerCA` → existing `return ctrl.Result{}, err` shape (transient, requeue via controller-runtime).
- **F-41 residual prune transient failure.** If the prune `CreateOrUpdate` fails with a non-conflict error, the fast-path returns `found=false` and the caller falls through to the full Issue cycle, which re-writes the Secret with the canonical key set. The extra keys are not silently left in place.
- **Audit emit failures.** Per the Phase 2c pattern, audit emit errors are non-fatal at the call site; the audit sink's broker-side write is the gate.

## Risks and trade-offs

- **F-42 cap value.** 300s is generous for any realistic harness shutdown but tight relative to legitimate use cases that wanted multi-minute graceful shutdown of long-running processes. Pre-1.0, the trade-off is acceptable; if a future harness genuinely needs more, the cap value is a single constant change. The cap's primary purpose — bounding credential exposure window — would still be best served by tightening F-11 (broker bearer revocation) so that the bearer is invalid the moment the run terminates regardless of grace period.
- **F-44 phase transition.** A transient operator misconfiguration (e.g., creating the broker-serving-cert Secret with the wrong key name and fixing it 10 seconds later) now lands runs at `Failed` rather than letting them recover when the operator fixes the source. Acceptable: the misconfiguration would have manifested as Pending-forever anyway. Operators with frequent test-cycle source-Secret churn can re-create runs after fixing the source.
- **F-41 residual prune is reactive, not preventive.** The strict-key check fires on the next reconcile after tampering — typically within 5s due to the Phase 2d `Owns()` watch, but the agent could `envFrom`-mount the extra key during that window if the Pod reads env at start. The fix is correct; the timing is bounded. A preventive admission webhook on Secret writes would close the window but adds a high-cardinality admission surface for marginal benefit.
- **F-56 digest pinning vs. operator override.** A digest-pinned base image cannot be swapped by chart values without rebuilding the iptables-init image. Theme 1 set this same constraint for `defaultSeedImage` and resolved it via `Workspace.spec.seed.image` override. iptables-init has no equivalent override path — operators who want a different base image rebuild from source. Acceptable: iptables-init is paddock-internal infrastructure, not user-pluggable.
