- Status: Draft

- Successor (planned): Phase 2c — Theme A (audit completeness), separate brainstorm.
- Target release: v0.5.0 (minor bump for new test infrastructure + image).

## 1. Summary

Phase 2a closed the three P0 findings (F-19, F-38, F-45) plus F-04 with unit-level test coverage but deferred E2E and adversarial testing. Phase 2b builds the test infrastructure that every later phase depends on:

1. A real-CNI Kind setup (Cilium) so NetworkPolicy enforcement is genuinely tested.
2. The "evil paddock-echo" hostile harness image specified in `2026-04-25-v0.4-test-gaps.md` §4.
3. Three initial E2E scenarios that adversarially validate Phase 2a's three P0 fixes (TG-2 for F-19, TG-7 for F-38, TG-24 for F-45).
4. CI integration so the new tests run on every PR.

The other 23 TG-NN scenarios from the test-gap inventory accumulate in later phases as their target findings get remediated. Phase 2b's job is to make those phases *possible*, not to cover all 26 scenarios upfront.

## 2. Goals and non-goals

**Goals:**

1. Real CNI (Cilium) installed by `make kind-up` so NetworkPolicy egress rules are actually enforced in tests. Without this, F-19 and F-45 tests would pass on `kindnet` regardless of whether the policies are correct.
2. The 15-flag hostile harness image at `images/evil-echo/`, built in CI alongside the existing reference images. Test-only; not pushed to public registries.
3. Three new E2E scenarios in `test/e2e/hostile_test.go` validating Phase 2a's three P0 fixes adversarially.
4. CI's existing `test-e2e.yml` workflow uses the Cilium-flavoured cluster and runs the new scenarios.

**Non-goals (deferred to later phases):**

- The other 23 TG-NN scenarios. They land in 2c–2g paired with the findings they validate.
- Theme A (audit completeness) and all later remediation work.
- Calico / multi-CNI / no-CNI test matrix variants. Single-CNI choice; YAGNI.
- Concurrency / lifecycle E2E tests. Already deferred per Phase 1 design §5.3 §5.
- Real-upstream tests (e.g., hitting actual `api.anthropic.com`) — opportunistic per Phase 1 plan; not needed for the three Phase 2a validations.

## 3. Per-component design

### 3.1 Real-CNI Kind setup (Cilium)

**Why Cilium over Calico:**

- **FQDN egress policies** become available — relevant to Phase 2c+ runtime-egress work where the audit's recommended fix for F-19's residual involves per-host allowlist matching (`api.anthropic.com` rather than CIDRs). With Cilium, that fix is portable; with Calico we'd need an extension or rebuild on a different CNI later.
- **Better Kind integration** — Cilium ships an official Kind quickstart and self-contained installation that doesn't require host-level kernel modules.
- **Single-source-of-truth** — broader Cilium adoption reduces CNI-divergence risk in the test suite.

**Implementation:**

- Modify `make kind-up` to install Cilium via the official Helm chart immediately after cluster creation. The Cilium install replaces `kindnet` (which Cilium will remove during installation).
- Add a `KIND_NO_CNI=1` environment variable: when set, skip the Cilium install and leave `kindnet`. Useful for users who want the default for non-NP-related testing.
- Bump `kind` cluster config to disable kindnet (`disableDefaultCNI: true`) and disable kube-proxy when `KIND_NO_CNI` is unset. Cilium replaces both.
- Document the change in `CONTRIBUTING.md` (only the developer-facing entry point — no operational docs change).
- Cluster bootstrap goes from ~10s to ~40s. Acceptable.

**No Helm-chart change to Paddock itself** — Paddock doesn't care which CNI is in use, only that NetworkPolicy is enforced. Cilium is a test-environment choice, not a runtime dependency.

### 3.2 Hostile harness (`images/evil-echo/`)

**Naming:** `paddock-evil-echo` (image) and `evil-echo` (binary). Distinct from `paddock-echo` to prevent accidental use as a real harness.

**Implementation shape:**

- `images/evil-echo/Dockerfile` — multi-stage build. Same base as `paddock-echo` (alpine + Go binary). PSS-restricted conformant: runs as UID 65532, no caps, no privilege escalation, read-only root FS where possible.
- `images/evil-echo/main.go` — single Go file. Each flag is one helper function. Flags are mutually composable (multiple flags can run in sequence in one invocation; output is one JSON object per flag executed).
- Output format per flag (matching test-gaps §4):
  ```json
  {"flag": "--connect-raw-tcp", "target": "10.0.0.1:443", "result": "denied", "error": "i/o timeout"}
  ```
- Sequential execution; each flag runs independently and reports its outcome regardless of prior failures. The harness exits 0 when all flags ran (regardless of whether attacks succeeded — success/failure is in the JSON, not the exit code; the e2e tests assert specific outcomes).

**The 15 flags (cross-reference to test-gaps §4):**

| Flag | What it does | Target finding |
|------|--------------|----------------|
| `--connect-raw-tcp <host:port>` | Open TCP socket; report success/error. | F-19, F-20, F-22, F-45 |
| `--read-secret-files <glob>` | Stat + read files matching glob; report what's readable. | F-38 |
| `--probe-broker <url>` | HTTPS POST to broker `/v1/issue` with synthetic bearer. | F-38, F-15 |
| `--bypass-proxy-env` | Unset `HTTPS_PROXY` / `HTTP_PROXY` / `NO_PROXY` before subsequent flags. | F-19, F-39 |
| `--exfil-via-dns <host>` | Send DNS queries with payload-encoded labels. | F-22, F-23 |
| `--read-other-tenant-pvc <namespace>` | Attempt to mount/read PVCs from another namespace. | T-4 |
| `--forge-ca-cert <fqdn>` | Read per-run CA private key + generate cert. | F-18 |
| `--connect-ip-literal <ip:port>` | Connect to IP literal bypassing DNS. | F-22 |
| `--probe-imds` | Reach `169.254.169.254` (cloud metadata service). | F-19 |
| `--connect-loopback <port>` | Connect to in-pod loopback ports. | T-4 |
| `--probe-env-override` | Read agent env, report `HTTPS_PROXY` etc. value; allow setting overrides. | F-39 |
| `--probe-provider-substitution-host <host:port>` | Send substitution-eligible request to non-allowlisted host. | F-09 |
| `--read-pvc-git-config` | Read `/workspace/.git/config` for embedded credentials. | F-50 |
| `--flood-connect-raw-tcp <host:port>` | Repeatedly open TCP sockets (DoS probe). | F-26, F-28 |
| `--smuggle-headers <name=value>` | Add headers to substitute-auth requests. | F-21 |

### 3.3 Three initial E2E scenarios (`test/e2e/hostile_test.go`)

A new test file with `e2e` build tag. Existing `e2e_test.go` retains the friendly-flow tests; hostile scenarios are deliberately separate so reviewers can navigate quickly.

Each scenario:

1. Creates a dedicated namespace with appropriate `BrokerPolicy` and `ClusterHarnessTemplate` (or reuses existing fixtures for `evil-echo`).
2. Submits a `HarnessRun` (or for TG-24, a `Workspace` with seed image overridden) that invokes the hostile harness with specific flags.
3. Waits for terminal state.
4. Reads the harness's JSON output from the run's output ConfigMap.
5. Asserts the specific outcome (e.g., "result: denied").
6. Optionally reads the `AuditEvent` stream and confirms the right kind was emitted (e.g., `egress-block`).

**Scenario 1 — TG-2 (F-19 cooperative-mode bypass):**

- HarnessRun: `evil-echo --bypass-proxy-env --connect-raw-tcp 10.0.0.1:443`. (`10.0.0.1` is RFC1918, expected to be excluded by Phase 2a's NP fix.)
- Run uses cooperative interception (BrokerPolicy `cooperativeAccepted: true`).
- Assert: harness output `{"flag": "--connect-raw-tcp", "result": "denied", "error": "..."}` (NP blocked).
- Assert: an `AuditEvent` with `kind: egress-block` exists in the run namespace (the proxy emitted it for the redirect path; if it doesn't, that's a separate finding for 2c-Theme-A).

**Scenario 2 — TG-7 (F-38 SA-token forgery):**

- HarnessRun: `evil-echo --read-secret-files '/var/run/secrets/**' --probe-broker https://paddock-broker.paddock-system.svc:8443/v1/issue`.
- Assert: harness output for `--read-secret-files` reports zero files matched at the standard SA-token path (Phase 2a removed it).
- Assert: harness output for `--probe-broker` reports rejection (the broker's TokenReview rejects the synthetic bearer).

**Scenario 3 — TG-24 (F-45 seed-Pod NP):**

- Workspace with `spec.seed.image: paddock-evil-echo:dev`, seed flags invoking `--connect-raw-tcp 10.96.0.1:443` (cluster service CIDR — should be excluded by the seed NP). Evil-echo replaces alpine/git as the seed Pod's main container; the seed Pod still gets the same labels (`paddock.dev/workspace=<name>`) so the seed NP selector matches.
- Workspace reaches `Failed` because evil-echo doesn't clone repos and the manifest file isn't produced. That's expected and not what we assert on.
- Assert (primary): kubectl-fetch the seed Pod's logs, parse evil-echo's JSON output, confirm `{"flag": "--connect-raw-tcp", "result": "denied"}`. This is the proof that the seed-Pod NP enforced the cluster-CIDR exclusion.
- Assert (secondary): the NetworkPolicy resource named `<workspace>-seed-egress` exists in the namespace.

Reading the seed Pod's logs (rather than the workspace status or AuditEvent stream) gives the cleanest assertion path: evil-echo's structured JSON is unambiguous about whether the egress was blocked.

**Test fixture additions:**

- `config/samples/paddock_v1alpha1_clusterharnesstemplate_evil_echo.yaml` — a ClusterHarnessTemplate pointing at `paddock-evil-echo:dev` with a permissive BrokerPolicy template (since these are adversarial-test fixtures, not user-facing samples).
- The test code lives in `test/e2e/hostile_test.go`; setup helpers reused from `test/utils/`.

### 3.4 CI integration

- The existing `test-e2e.yml` workflow already drives Kind setup and runs `make test-e2e`. With `make kind-up` updated to install Cilium, CI gets the change for free.
- The new hostile scenarios run alongside existing E2E. Add ~3-5 minutes to CI (Cilium install + 3 scenarios at 1-2 min each).
- No new CI workflow file. The change is to the existing `kind-up` invocation.

## 4. Sequencing

1. **Cilium Kind setup first.** Foundational — without it, the hostile-harness tests would pass-by-luck on `kindnet`.
2. **Hostile harness image second.** Build the binary + image; verify it runs (manual `kubectl run` invocation works) before wiring into tests.
3. **E2E scenarios third.** Each scenario lands as one commit (3 commits).
4. **CI integration last.** Verify the full pipeline from PR open → Kind cluster up with Cilium → hostile tests run → all green.

Each fix lands as one commit. Conventional Commits format. No `!:` markers — Phase 2b adds new infrastructure but doesn't break any user-facing API.

## 5. Branch and release

- Branch: `security/v0.4-review-phase-2b` off `main` (which is at v0.4.1).
- Single PR; release-please picks up the Conventional Commits and cuts `v0.5.0` after merge.

## 6. Risks

- **Cilium install reliability on Kind.** The official Cilium-on-Kind quickstart is well-documented; rare flakes possible. Mitigation: pin Cilium chart version in `make kind-up`, document upgrade procedure in `CONTRIBUTING.md`.
- **Hostile harness image proliferation.** The image is test-only but built in CI alongside reference images. Anyone who runs `make images` gets it. Document clearly in `images/evil-echo/README.md` ("DO NOT use as a real harness; this image bypasses security defences for testing").
- **CI runtime increase.** ~3-5 min added to E2E. Acceptable; `test-e2e.yml` already takes ~5-10 min.
- **`Workspace.spec.seed.image` override path is rarely-used.** Tests today pass `SeedImage` via the reconciler field rather than the spec field. Verify the spec field actually works end-to-end before relying on it for TG-24. If broken, fall back to test-only reconciler-field override (same as existing tests).
- **Synthetic bearer rejection for `--probe-broker`.** TokenReview should reject any non-API-server-issued token, but this is worth confirming early in scenario implementation — the test depends on it.

## 7. Exit criteria

1. `make kind-up` installs Cilium by default; `KIND_NO_CNI=1` preserves the old behaviour.
2. `images/evil-echo/Dockerfile` and `main.go` exist and build via `make images`.
3. All 15 flags from §3.2 work — exercise via a quick smoke test (run the binary locally with each flag against a controlled target).
4. `test/e2e/hostile_test.go` has 3 scenarios passing locally (`make test-e2e`) against a Cilium-flavoured Kind.
5. CI's `test-e2e.yml` job passes the new scenarios on PR open.
6. Documentation: `CONTRIBUTING.md` mentions Cilium; `images/evil-echo/README.md` warns against production use.
7. Each finding's `Test gap correlation` field in the audit-findings doc is updated to point at the new test for F-19 (TG-2), F-38 (TG-7), F-45 (TG-24).

## 8. What 2b explicitly does NOT do

- Build the other 23 TG-NN scenarios. They land alongside their target finding's remediation in 2c–2g.
- Touch any production code path (no controller / broker / proxy / webhook changes).
- Add operator-facing documentation. The hostile harness is test-only.
- Set up a multi-CNI test matrix.

## 9. Open questions

- **Cilium chart version pin.** Latest stable at the time of implementation; revisit if Cilium changes its Kind-quickstart contract. Decided during implementation.
- **Smoke-testing flag harness.** Whether each of the 15 flags gets a unit test (in `images/evil-echo/main_test.go`) or just the e2e smoke test. Decision: light unit tests for the flags that have non-network behaviour (`--read-secret-files`, `--read-pvc-git-config`); rely on e2e for the network flags.
