# 2026-04-26 — v0.4 security recheck (post-Phase-2g) — design

- Owner: @tjorri
- Date: 2026-04-26
- Status: Draft
- Predecessor: Phase 2g (Substitute-auth hygiene — F-09/F-10/F-21/F-25, PR #37).
- Successor (planned): TBD via the remediation roadmap memo this design produces.

## 1. Summary

Phase 1 of the v0.4 holistic security review (PR #21, 2026-04-25) produced 55 findings against the v0.4 architecture at commit `dd1e028`. Seven Phase 2 work bundles have since landed remediations across most of the high-risk surface: Phase 2a (P0 hotfix — F-04/F-19/F-38/F-45), Phase 2b (test infrastructure + adversarial coverage), Phase 2c (Theme A audit completeness — F-12/F-13/F-24/F-32/F-33/F-40), Phase 2d (carry-over fixes — F-41/F-43), Phase 2e (pod-spec hardening — F-37/F-39/F-31), Phase 2f (per-run intermediate CA — F-18), and Phase 2g (substitute-auth hygiene — F-09/F-10/F-21/F-25). Twenty findings now carry explicit `**Resolution.**` lines in the audit doc.

This design covers a recheck of the audit against post-Phase-2g state. The recheck reconciles every finding against the current code, refreshes the threat-model defence claims that Phase 2 changed, reconciles the test-gap inventory against shipped tests, opportunistically scans for new findings introduced by Phase 2's code changes, and produces a remediation roadmap memo grouping the open backlog into themes for future Phase 2h+ work.

The recheck deliberately does not re-do Phase 1's full methodology (per-component STRIDE, attack trees, the works). It is targeted reconciliation plus a thin re-scan of the areas Phase 2 changed.

## 2. Goals and non-goals

**Goals.**

1. Produce a per-finding verdict (Resolved / Partially resolved / Deferred-by-design / Won't-fix / Open) for all 55 Phase 1 findings, with verification evidence at a documented rigor level.
2. Refresh `docs/security/2026-04-25-v0.4-audit-findings.md` §1 headline counts and "areas of concern" themes over the post-Phase-2g state.
3. Refresh `docs/security/threat-model.md` defence-claim lines that Phase 2 changed — keep the threat model durable but accurate.
4. Reconcile `docs/security/2026-04-25-v0.4-test-gaps.md` TG-NN entries against shipped tests across Phase 2a-2g.
5. Surface any new findings introduced by Phase 2's code changes (targeted re-scan of areas Phase 2 touched, not a full re-audit). New findings get F-56+ IDs.
6. Produce a follow-up roadmap memo (`docs/plans/2026-04-26-v0.4-security-recheck-roadmap.md`) grouping the open backlog into themes with rough sizing — not full Phase designs.

**Non-goals (deferred):**

- Code changes. The recheck is docs-only. Remediation of any new findings is the next Phase's work.
- Full re-audit (Phase 1 v2). The Phase 1 methodology — per-component STRIDE walkthroughs, attack trees on the three high-value goals — is not re-executed. The recheck trusts Phase 1's framing and updates the verdicts.
- Full Phase 2h+ designs. The roadmap memo is a planning aid, not a phase plan; each future phase's full design happens against then-current state.
- Bridges component (`paddock-bridges`) — not yet built; out of scope per Phase 1.
- Concurrency / lifecycle E2E scenarios beyond what's already in the suite.
- Hostile-harness ("evil paddock-echo") implementation — Phase 2b shipped the test infrastructure; the per-flag F-09..F-25 mapping is reconciled in this recheck but the harness itself is not extended.

## 3. Methodology

Approach: **theme-based component sweep** (rejected alternatives: serial finding-by-finding walk; subagent fan-out — both discussed in the brainstorm and recorded in §8).

Six component sweeps in trust-root-first order, each one sitting:

1. `internal/broker` — trust root.
2. `internal/proxy` — runtime defence.
3. `internal/webhook` + `internal/policy` — admission.
4. `internal/controller` — biggest component.
5. Workspace seed paths (`internal/controller/workspace_*.go`, related webhook bits) — narrow surface.
6. CLI credential paths (`internal/cli`) — last; mostly UX.

Followed by four cross-cutting passes that don't fit per-component scope.

The methodology is the brainstorm's Approach 2 — load each component into context once, do the verification + triage + targeted re-scan in a single pass per component, commit before moving on.

## 4. Per-finding Status block

Every finding gets a uniform Status block appended after the existing prose. Existing `**Resolution.**` lines stay in place — the Status block is the structured verdict; the Resolution prose remains the narrative.

**Template:**

```
**Status (recheck 2026-04-26).**
- State: <Resolved | Partially resolved | Deferred-by-design | Won't-fix | Open>
- Verification: <Light | Medium | Heavy | N/A>
- Reviewer: @tjorri
- [Residual: …]      ← only on Partially resolved
- [Rationale: …]     ← only on Deferred-by-design or Won't-fix
- [Revisit: …]       ← only on Won't-fix
- [Regression: …]    ← only when rigor check finds the Resolution claim no longer holds
```

**State definitions:**

| State | Meaning | Goes into roadmap? |
|---|---|---|
| **Resolved** | Fix landed; code matches the Resolution claim at the chosen rigor level. | No |
| **Partially resolved** | Primary recommendation landed; named residual remains. The Residual line points at where the rest is tracked. | Residual half goes into roadmap |
| **Deferred-by-design** | Audit's recommendation reviewed and rejected (or scoped down) for documented reasons. | No |
| **Won't-fix** | Out of scope for v0.4 / pre-1.0; carries a Revisit trigger. | Listed in roadmap appendix as "deferred surface" |
| **Open** | No fix landed yet. | Yes |

**Verification rigor (Q4-D mapping):**

- **Light** — confirm cited file/symbol exists in the current tree. ~2 min/finding.
- **Medium** — Light plus read the cited unit tests, confirm they exercise the claimed behaviour. ~10 min/finding.
- **Heavy** — Medium plus run unit + relevant e2e specs locally. ~30+ min/finding plus one full e2e run.
- **N/A** — Won't-fix or Deferred-by-design where verification doesn't apply.

Mapping severity → rigor:

- Original P0s + Risk-rated High → **Heavy**.
- P1 → **Medium**.
- P2 / P3 → **Light**.

A future reader (or future recheck) can tell from the Status block what kind of evidence backs the verdict.

## 5. Component sweep template

Each of the six sweeps follows the same five steps and produces a single commit.

**Step 1 — Pull the finding subset.** Filter the §2 audit findings table by Component column. Findings tagged with two components (e.g. F-04 "proxy primary, controller secondary") are handled in the *primary* component's sweep, with a one-line forward-pointer in the secondary's sweep notes.

**Step 2 — Verify resolved findings.** For every finding currently carrying a `**Resolution.**` line in this component, apply §4 rigor (Heavy for original P0/Risk-High, Medium for P1, Light for P2/P3). Write the new Status block. If the rigor check finds the Resolution claim has regressed (no longer true after intervening commits), downgrade to **Open** with a `Regression:` note explaining what broke.

**Step 3 — Triage unresolved findings.** For each finding without a Resolution line, classify into one of the five states. **Open** is the default — only mark Deferred-by-design or Won't-fix if there's an explicit, documentable rationale (and that rationale gets recorded in the Status block).

**Step 4 — Targeted re-scan.** While the component is loaded:

- Re-run `gosec` scoped to the component subdir; reconcile any new hits.
- Walk the Phase 2 commits that touched this component (`git log dd1e028..HEAD -- internal/<component>/`) and read the diffs with security eyes — new code paths, new error returns, new unmarshal sites, new env-var reads.
- Cross-reference the threat-model trust boundary the component owns (each component owns 1–2 of the seven boundaries) — does the Phase 2 code change the boundary's STRIDE picture?

Anything new gets opened as F-56, F-57, … in the same component sweep — slotted into §2's table, full finding entry written matching the existing entry shape, Status block included from creation.

**Step 5 — Commit.** Single commit per sweep: `chore(security): recheck — <component>`. Body lists what changed (N findings verified, M state changes, K new findings opened).

Cross-cutting tools that don't fit per-component scope (`govulncheck`, `trivy`, `kube-linter`) run once globally in §6 Pass A, not per sweep.

## 6. Cross-cutting passes

After all six component sweeps, four passes in this order, each its own commit.

### 6.1 Pass A — Global static-analysis re-run

- `govulncheck ./...` against the current module graph.
- `trivy image` against the published images (same set as Phase 1: broker, controller-manager, proxy, paddock-claude-code harness, etc.).
- `kube-linter` against the Helm chart in `dist/chart/`.

Tool versions and full output captured in `docs/security/recheck-2026-04-26-tool-output/` (gitignored if large; checksum/summary committed). New hits get F-NN entries with full Status blocks. Suppressed-with-rationale items get a Status block of `Deferred-by-design` plus the suppression note.

Commit: `chore(security): recheck — global static analysis`.

### 6.2 Pass B — Threat-model defence-claim refresh

Read `docs/security/threat-model.md` end to end. For every "Defence:" line, confirm the named code path (or design pattern) still implements the defence. Phase 2 changed several:

- Per-run intermediate CA (F-18 → Phase 2f).
- Audit-write-fail-closed posture (F-12/24/40 → Phase 2c).
- Per-run NetworkPolicy hardening (F-19/45 → Phase 2a).
- Substitute-auth host scoping + revalidation (F-09/10/21/25 → Phase 2g).
- Run-pod hardening (F-37/38/39 → Phase 2a/2e).
- Seed-pod surface (F-45/47/48 → Phase 2a-2e).

Stale defence claims are rewritten in place. New defences earned in Phase 2 are added. The B-N trust boundary STRIDE tables get a "Phase 2 update (2026-04-26)" trailing column or revision marker (matching the project ADR convention of evolving via trailing "Phase N update" sections rather than rewriting in place).

A `## Revision history` block at the bottom of the doc records the recheck (date, predecessor phase, what changed at a high level).

Commit: `chore(security): recheck — refresh threat-model defence claims`.

### 6.3 Pass C — Test-gaps reconciliation

Read `docs/security/2026-04-25-v0.4-test-gaps.md`. For each TG-NN, check what shipped:

- Phase 2b explicitly delivered "test infrastructure + adversarial coverage" — many TGs trace to PR #24.
- Later phases (2a, 2c-2g) each shipped their own unit + e2e tests as cited in the corresponding Status blocks.

Mark each TG with one of:

- **Shipped** — test exists, runs in CI, exercises the original gap.
- **Partial** — some scenarios covered, others gap (carries a Residual note).
- **Open** — no test landed.

The hostile-harness ("evil paddock-echo") spec entry gets its own Status — it's the largest single deliverable from the inventory. Reconcile the per-flag F-09..F-25 mapping against `test/hostile/main.go` flags actually shipped.

A `## Recheck history` block at the top records the recheck pass.

Commit: `chore(security): recheck — reconcile TG inventory against shipped tests`.

### 6.4 Pass D — Audit doc §1 refresh

Now that every finding has a Status block, recompute:

- **Headline counts.** Total findings: 55 + m (where m = newly-opened F-56+). By state: Resolved × R, Partially × P, Deferred × D, Won't-fix × W, Open × O. By risk and priority recomputed *over the Open + Partially-resolved-residual subset only* — that's the meaningful backlog number, not the historical total. The historical total is preserved in a "Phase 1 baseline (2026-04-25)" line for reference.

- **Top three to fix.** Recomputed from current Open P0/P1. If the original P0s are all Resolved (likely — F-19 Partially Phase 2a, F-38 Resolved Phase 2a, F-45 Resolved Phase 2a), say so explicitly: "All Phase 1 P0s closed (or partially closed) in Phase 2a. New top-three is …"

- **What went well.** Append a paragraph on what Phase 2 delivered (audit story end-to-end, per-run CA isolation, substitute-auth hygiene, pod hardening, etc.). This is the positive evidence the recheck earned and explicitly worth recording.

- **Areas of concern.** Rewrite. The original three themes ("audit story broken end-to-end", "runtime egress gaps", "per-run isolation is partly fiction") are largely closed. New themes will likely emerge from the open backlog — the rewrite captures whatever the recheck data actually shows; speculation here is deferred to the roadmap memo.

A `## Recheck history` block at the top of the audit doc records the recheck pass (date, predecessor phase, headline-count delta, link to roadmap memo).

Commit: `chore(security): recheck — refresh §1 headline counts and themes`.

## 7. Roadmap memo

File: `docs/plans/2026-04-26-v0.4-security-recheck-roadmap.md`. Voice and weight mirror `docs/plans/2026-04-24-v04-followups-roadmap.md` — a planning memo, not an executable plan. Full Phase 2h+ designs get brainstormed against then-current state when each is actually started.

**Structure:**

1. **Header** — owner @tjorri, dated, "Related recheck:" → audit-findings doc, "Predecessor:" → Phase 2g.

2. **Shared context from Phase 2 (carries into every theme).** What Phase 2 established that future themes should reuse:

   - Fail-closed-on-audit-failure pattern (Phase 2c) — every new denial path emits an AuditEvent before returning.
   - Cert-manager-driven per-run intermediate CA (Phase 2f) — the model for any future per-tenant secret material.
   - Broker as single fail-closed enforcement point on substitute path (Phase 2g) — proxy is wire-shape + lifecycle, not policy.
   - PSS-baseline / PSS-restricted envelope per container (Phase 2e) — extend, don't redo.
   - Status-condition + AuditEvent dual reporting at admission (Phase 2c).

3. **Themes.** Each theme groups Open + Partially-resolved-residual findings into one work bundle. The exact themes come from the recheck output; the design commits to the **grouping rubric**:

   - Group findings by **shared code-path or shared mitigation pattern**, not by component. (E.g. F-22 DNS rebinding + F-28 forged-leaf cache both live in the proxy connection lifecycle and likely share a fix; they form one theme even though F-22 is also adjacent to F-23 / F-34 which are admission-side and form a different theme.)
   - Each theme should fit ≤ 8–10 tasks of work (Plan A's size as the calibration anchor).
   - A theme is named after its mitigation, not its findings: "broker hygiene", "workspace seed surface", "egress allow-list grammar", etc.

4. **Per theme:**

   - Findings included (ID + one-line summary).
   - Why this is one bundle (the shared mitigation or shared code surface).
   - Things to reuse from existing phases (the 2026-04-24 memo's pattern — gives the future plan-writer a head start).
   - Known complexity / unresolved spec questions.
   - Rough size in Plan A task units.

5. **Suggested execution order.** Sequencing memo — what depends on what, what's lowest-risk first, what could be bundled. Not a commitment.

6. **Open questions to decide at plan-writing time.** Same shape as the 2026-04-24 memo's §"Open questions" — defer concrete choices to when the corresponding plan is actually written, against then-current code.

7. **Deferred surface appendix.** One-paragraph recap of every Won't-fix finding plus its Revisit trigger. Not part of the active backlog; lives here so future-you can find them.

**What this memo deliberately does *not* do:**

- Does not pre-write Phase 2h, 2i, … designs. Those happen at execution time, one at a time.
- Does not commit to a sequencing. The "Suggested execution order" is a recommendation; the user may pick a different next theme based on whatever's most painful at the moment.
- Does not merge findings across themes. A finding belongs to exactly one theme — partially-resolved residuals show up under the theme that owns the residual mitigation, not the original finding's theme.

## 8. Alternatives considered

**Approach 1 — Serial finding-by-finding walk.** Walk F-01 → F-55 in numerical order, applying Q4 rigor to each. Rejected: walking by finding ID hops between broker, proxy, webhook, controller for every other finding — pays the per-component context-load cost ~5× per component instead of once. Simpler to track but materially more expensive.

**Approach 3 — Subagent fan-out.** Dispatch six parallel subagents, one per component. Rejected: subagents can't see each other's findings, so cross-component patterns (e.g., "audit story broken end-to-end" cuts across broker + proxy + controller + webhook) are easy to miss. Static-analysis re-run is global and doesn't fan out cleanly. Verification rigor is hard to enforce uniformly across six independent agents — and the original Phase 1 used subagents for the initial audit, which is precisely why we're now doing a recheck.

**Approach 2 — Theme-based component sweep (chosen).** Load each component once; the targeted re-scan piggy-backs on already-loaded context; produces natural commit boundaries (one component = one commit). Tradeoff: §1 headline counts don't refresh until the final pass; partial commits are temporarily inconsistent (counts in §1 lag the per-finding state). Acceptable — readers know a recheck is in flight.

**Status taxonomy — binary vs three-state vs five-state.** Five-state chosen so the open backlog list is honest. Binary (Resolved / Open) hides design choices like F-37's intentional non-setting of pod-level `runAsNonRoot`; three-state (adds Partially) captures F-19's residual but still conflates intentional non-fixes with backlog items. Five-state cost is small (extra two states with explicit rationale lines).

**Audit doc filename — keep `2026-04-25` prefix vs rename.** Keep the original. The date marks when the audit was *performed*; the recheck is a refresh of that audit, not a new audit. Phase 2 docs already cite the original filename and renaming would invalidate those references.

**Output as one PR vs split by phase.** One PR (12 commits). The §1 refresh in commit 11 only makes sense after the per-finding state lands in commits 2–10; the natural review unit is the whole thing. Splitting into 2–3 PRs would force §1 to refresh twice (once per partial state) or land out-of-order with the per-finding state.

**Recheck cadence — recheck-on-demand vs scheduled.** Recheck-on-demand. The recheck doc records its date in the Status blocks; future rechecks append rather than overwrite. No commitment to a scheduled cadence at this stage — pre-1.0, recheck cadence is driven by Phase work landing.

## 10. Risks and mitigations

**Drift between component sweeps.** A Phase 2 commit could land while the recheck is in flight, invalidating earlier sweeps. Mitigation: recheck is short (~1 day), branch is rebased before PR. Phase 2g just landed (PR #37); next Phase work is gated on this recheck producing the roadmap.

**Verification rigor inflation.** The mixed rigor map (§4) says P3 gets Light, but a P3 might warrant deeper attention if the static re-scan flags a related issue. Mitigation: rigor is a default, not a contract — escalate when the data warrants. Status block records actual rigor applied.

**New findings hide in unchanged code.** The targeted re-scan (§5 Step 4) only walks Phase 2 diffs — pre-Phase-2 code is trusted from Phase 1. Mitigation: this is an explicit non-goal (full re-audit deferred). The risk is acknowledged; if the static tools flag something in unchanged code it still lands as F-NN.

**Status taxonomy disagreement.** "Deferred-by-design" vs "Open" is a judgment call (e.g., F-37's pod-level `runAsNonRoot`). Mitigation: every Deferred-by-design carries a Rationale line; the mid-flight checkpoint surfaces borderline calls for user review before they lock in.

**Roadmap memo over-prescribes.** The grouping rubric says "shared mitigation pattern" but real findings cluster ambiguously. Mitigation: each theme's "Why this is one bundle" line documents the rationale; the user (or future plan-writer) can re-cluster if a different bundling makes more sense at plan-writing time.
