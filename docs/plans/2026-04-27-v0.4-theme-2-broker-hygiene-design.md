## 1. Scope

Theme 2 closes five P1–P3 broker findings from the 2026-04-25 audit recheck in a single PR. The work is mostly broker-internal; F-11 and F-14 cross into the controller via a new additive field on `HarnessRun.status` and a new finalizer.

| Finding | Priority | Resolution shape |
|---|---|---|
| **F-11** Bearer revocation on run completion not implemented | P1 | New `Revoke(ctx, leaseID)` on `Provider`, `/v1/revoke` endpoint, `paddock.dev/broker-leases-finalizer` on HarnessRun; controller writes `Status.IssuedLeases` after each successful Issue and calls Revoke for each at finalizer cleanup. |
| **F-14** PATPool lease state lost on broker restart | P2 | Broker reconstructs `PATPoolProvider` slot reservations at startup from `Status.IssuedLeases` across all non-terminal HarnessRuns. Reuses F-11's status field — no new persistent surface. |

## 2. F-11 + F-14 — Lease lifecycle

### 2.1 CRD: `HarnessRun.status.IssuedLeases`

```go
// IssuedLease records one credential lease the broker minted for this run.
// The controller appends one entry per successful broker.Issue call;
// reconcileDelete walks the slice and posts /v1/revoke for each entry
// before removing the broker-leases finalizer. Pre-1.0 evolves in place.
type IssuedLease struct {
    // Provider is the provider kind (matches BrokerPolicy
    // grant.provider.kind). The broker dispatches /v1/revoke to the
    // named provider's Revoke method.
    Provider string `json:"provider"`

    // LeaseID is the provider-supplied identifier returned from
    // IssueResult.LeaseID. Opaque to the controller; passed back unchanged.
    LeaseID string `json:"leaseID"`

    // CredentialName is the requirement name from the template's
    // spec.requires.credentials list. Used for audit correlation only —
    // never load-bearing for revocation.
    CredentialName string `json:"credentialName"`

    // ExpiresAt mirrors IssueResult.ExpiresAt. Reconstruction skips
    // entries with ExpiresAt < now (no point rebuilding state for an
    // already-dead lease). Optional: nil means "no expiry".
    ExpiresAt *metav1.Time `json:"expiresAt,omitempty"`

    // PoolRef carries PATPool-specific reconstruction metadata.
    // Populated only when Provider == "PATPool"; nil otherwise.
    // Anonymous tagged-union pattern; future providers add their own
    // optional ref alongside without breaking pre-1.0 in-place evolution.
    PoolRef *PoolLeaseRef `json:"poolRef,omitempty"`
}

type PoolLeaseRef struct {
    SecretRef SecretKeyReference `json:"secretRef"`
    SlotIndex int                `json:"slotIndex"`
}
```

`HarnessRun.status.IssuedLeases []IssuedLease` is empty for runs that haven't called Issue (e.g. templates with no `spec.requires.credentials`).

We deliberately do **not** persist bearer bytes in status — that would put credentials into a reconstructable field. The slot-index reconstruction (PATPool) plus existing `LeasedPAT` byte-equality check at substitute time (already in `patpool.go:319-323` from a prior phase) is sufficient to close F-14's "same PAT to two runs" hazard.

### 2.2 `Provider.Revoke`

```go
type Provider interface {
    Name() string
    Issue(ctx context.Context, req IssueRequest) (IssueResult, error)

    // Revoke releases the in-memory lease state for leaseID. Idempotent —
    // revoking an unknown or already-expired leaseID returns nil.
    // PATPool: marks the pool slot free + drops the bearer→lease entry.
    // Anthropic / GitHubApp: drops the bearer→lease entry.
    // UserSuppliedSecret: no-op (no in-memory state).
    Revoke(ctx context.Context, leaseID string) error
}
```

Per-provider implementation:
- Each provider's lease struct (`anthropicLease`, `githubLease`, `patLease`) gains a `LeaseID string` field populated at Issue time so the lookup is keyed on LeaseID, not bearer.
- Revoke walks the in-memory map looking for an entry whose `LeaseID` matches; if found, deletes it (and frees the pool slot for PATPool via the existing `releaseLocked` helper). O(n) per call where n is small (≤ pool size for PATPool, ≤ concurrent runs for others).
- `UserSuppliedSecret.Revoke` returns nil immediately — that provider has no per-issue state to release.

The Provider-interface change is a breaking change for out-of-tree implementors. The commit adding the method takes `feat(broker)!:` per CLAUDE.md so `release-please` picks up the marker.

### 2.3 `/v1/revoke` endpoint

```
POST /v1/revoke
Authorization: Bearer <SA token, audience=paddock-broker>
X-Paddock-Run: <run-name>
X-Paddock-Namespace: <run-namespace>
Content-Type: application/json

{"provider": "PATPool", "leaseID": "pat-abcd1234", "credentialName": "github-token"}
```

- **Authorization:** caller must be the controller (`IsController == true` in `internal/broker/auth.go`). Other callers get 403. Run pods cannot revoke their own leases — limits the blast radius of a compromised pod that might otherwise deliberately churn pool entries.
- **Run identity:** carried in `X-Paddock-Run` / `X-Paddock-Namespace` headers (same shape as `/v1/issue`). The handler uses the existing `resolveRunIdentity` helper, which already permits the controller cross-namespace access.
- **Dispatch:** handler calls `registry.Lookup(provider).Revoke(ctx, leaseID)`. Unknown provider → 400 `BadRequest`. Provider error → 500 `ProviderFailure`.
- **Audit:** writes a `credential-revoked` AuditEvent **before** the response (Phase 2c fail-closed-on-audit-failure pattern). Audit-write failure → 503 `AuditUnavailable`.
- **Response:** 204 on success; standard `ErrorResponse` on failure.
- **Idempotency:** unknown LeaseID returns 204 (provider returns nil per §2.2). Caller-side retry is safe.

New constants in `internal/broker/api/types.go`: `PathRevoke = "/v1/revoke"`, `RevokeRequest`, `RevokeResponse`.

### 2.4 Controller: `paddock.dev/broker-leases-finalizer` + status writes

**Add finalizer** alongside the existing `HarnessRunFinalizer` in `Reconcile` at the same point the existing finalizer is added (`harnessrun_controller.go:154-163`).

**Extend the `BrokerIssuer` interface** in `internal/controller/broker_client.go` so the reconciler can call Revoke alongside Issue:

```go
type BrokerIssuer interface {
    Issue(ctx context.Context, runName, runNamespace, credentialName string) (*brokerapi.IssueResponse, error)
    Revoke(ctx context.Context, runName, runNamespace string, lease IssuedLease) error

```

`BrokerHTTPClient.Revoke` POSTs to `/v1/revoke` with the run identity headers + JSON body shape from §2.3. Test fakes (existing `BrokerIssuer` mocks in `broker_credentials_test.go`) need a `Revoke` stub.

**After successful broker.Issue (in `ensureBrokerCredentials`):**

The existing call site at `internal/controller/broker_credentials.go:85` already harvests per-credential metadata into `credStatus`. Extend the harvest to also build a `[]paddockv1alpha1.IssuedLease` slice and return it alongside `credStatus`. The reconciler then writes both onto `run.Status` in the same status update:

```go
// Inside ensureBrokerCredentials, alongside the existing credStatus build:
issuedLeases = append(issuedLeases, paddockv1alpha1.IssuedLease{
    Provider:       resp.Provider,
    LeaseID:        resp.LeaseID,
    CredentialName: c.Name,
    ExpiresAt:      &metav1.Time{Time: resp.ExpiresAt},
    PoolRef:        deriveProviderRef(resp), // populated only for PATPool
})
```

`deriveProviderRef` returns a `*PoolLeaseRef` when `resp.Provider == "PATPool"`, populated from new fields on `IssueResponse` (`PoolSecretRef *SecretKeyReference`, `PoolSlotIndex *int`) that the broker already has at issue time. These fields are additive on the wire; older controllers ignore unknown JSON fields, and older brokers leave them zero — `deriveProviderRef` returns nil in that case.

**In `reconcileDelete`, BEFORE the existing HarnessRunFinalizer removal:**

```go
if controllerutil.ContainsFinalizer(run, BrokerLeasesFinalizer) {
    revokeCtx, cancel := context.WithTimeout(ctx, brokerRevokeBudget(run))
    defer cancel()
    if err := r.revokeIssuedLeases(revokeCtx, run); err != nil {
        // Loud warning + proceed: don't pin teardown on broker reachability.
        // Project memory: sequence first, wait, then force-clear.
        logger.Error(err, "broker revoke failed; force-removing finalizer to unblock teardown",
            "run", run.Name, "leases", len(run.Status.IssuedLeases))
        r.Recorder.Eventf(run, corev1.EventTypeWarning, "RevokeFailed",
            "Broker revoke failed; finalizer force-removed: %v", err)
    }
    controllerutil.RemoveFinalizer(run, BrokerLeasesFinalizer)
    if err := r.Update(ctx, run); err != nil { ... }
    return ctrl.Result{Requeue: true}, nil
}
```

- `brokerRevokeBudget(run)` returns `len(IssuedLeases) * 5s`, capped at 30s total.
- Per-call timeout: 5s with 2 retries on transient errors (network / 5xx).
- Response handling:
  - **2xx** (204 success, including idempotent unknown-LeaseID per §2.3) → success.
  - **404 NotFound** from an older broker without the `/v1/revoke` endpoint → success-equivalent. The lease is unrecoverable either way (old broker has no Revoke surface), and the loud-warning force-clear branch unblocks teardown.
  - **5xx / network errors** → retry up to 2 times with backoff, then aggregate into the returned error.
- `revokeIssuedLeases` returns nil only when **all** revokes succeed or were treated as success-equivalent; aggregated error otherwise. Caller logs and proceeds.

The loud-warning force-clear branch is the **e2e leak-prevention guarantee**: a broker pod down at teardown time cannot pin a HarnessRun. Validated by an e2e spec — see §6.3.

### 2.5 Broker startup reconstruction (F-14)

In `cmd/broker/main.go`, after `WaitForCacheSync` and before installing the API server:

```go
if err := reconstructLeases(ctx, cachedClient, registry); err != nil {
    setupLog.Error(err, "lease reconstruction had errors; broker continues with partial state")
    // Non-fatal: a reconstruction error means some pool slots may be re-leased,
    // which is exactly the F-14 hazard — log loudly, don't refuse to start.
}
```

`reconstructLeases`:

1. List HarnessRuns cluster-wide (broker already has `get;list;watch` RBAC on `harnessruns`).
2. For each run **not** in a terminal phase (`Cancelled`, `Succeeded`, `Failed`):
3. &nbsp;&nbsp;For each `IssuedLease` where `Provider == "PATPool"` AND `ExpiresAt > now` AND `PoolRef != nil`:
4. &nbsp;&nbsp;&nbsp;&nbsp;Read the pool Secret at `(run.Namespace, PoolRef.SecretRef.Name)`.
5. &nbsp;&nbsp;&nbsp;&nbsp;If `PoolRef.SlotIndex < len(entries)`: call `PATPoolProvider.ReserveSlot(secretRef, slotIndex, leaseID)` to mark the slot leased in-memory (no bearer→lease entry — see below).
6. &nbsp;&nbsp;&nbsp;&nbsp;If `SlotIndex >= len(entries)` (pool shrank since last broker uptime): skip + emit `paddock_broker_patpool_reconstruct_skipped_total{reason="slot-out-of-range"}` metric.

`ReserveSlot` is a new method on `PATPoolProvider` that takes the lock and marks `pool.leased[slotIndex] = true` without populating `pool.byBearer` — we deliberately **do not reconstruct** the bearer ↔ leaseID mapping. That mapping requires bearer bytes, which are credentials we won't put on `HarnessRun.status`.

**Consequence:** after broker restart, a dead run's bearer fails closed (the agent's TLS request errors → `BrokerReady=False` → controller drives a fresh Issue → fresh bearer). The fresh Issue picks a *different* slot from the reserved one, so the new run does not collide with the dead run's slot. F-14's "same PAT to two runs" hazard is closed.

For non-PATPool providers, no reconstruction needed. Their leases mint bearers on demand without slot collision; restart simply forces re-issuance, which is the existing fail-closed behavior.

### 2.6 Known limitation: status-write failure orphans a lease

If `broker.Issue` succeeds and the per-run Secret write succeeds but the `Status.IssuedLeases` write fails, the lease is in the broker's memory but not in the run's status. On reconcile retry the controller calls Issue again, minting a fresh lease; the original is orphaned in the broker's memory until TTL expiry (≤ 1h for default-configured providers).

The audit trail still records the issue (Phase 2c). The orphan window is bounded. Acceptable for v0.4. Closing this race fully would require either making `broker.Issue` idempotent on `(runUID, credentialName)` (broker reads `HarnessRun.status` to deduplicate) or adopting a separate persistent lease CR. Both rejected at clarifying-question time as disproportionate to the hazard.

A follow-up GitHub issue tracks this for future revisitation if real workloads hit the orphan window.

## 3. F-15 — ADR-0012 wording fix

Single edit to `docs/adr/0012-broker-architecture.md` line 23. Replace:

> **API:** HTTP/JSON over mTLS on `paddock-broker.paddock-system.svc:8443`.

with:

> **API:** HTTP/JSON over TLS (server cert from cert-manager) + SA-token Bearer authentication on `paddock-broker.paddock-system.svc:8443`. The trust boundary is the SA token validated via TokenReview against the in-cluster apiserver, with audience binding to `paddock-broker`. The TLS layer authenticates the server to clients (clients refuse to talk to a forged broker); client identity is established by the SA token, not by client certificate.

Add a "Phase 2h update (2026-04-27)" trailing section per the project's ADR convention:

> **Phase 2h update (2026-04-27).** F-15 audit recheck noted the original "mTLS" wording was aspirational — the broker's `tls.Config` ships with `ClientAuth = NoClientCert`. Wording above clarified to match the implemented trust posture (TLS server cert + SA-token). Defense-in-depth via `RequireAndVerifyClientCert` was considered and deferred — wiring a second CA hierarchy distinct from Phase 2f's per-run intermediate is disproportionate for a P3 doc/code mismatch when SA-token + TokenReview already establishes the trust boundary.

No code change. The `cmd/broker/main.go` `tls.Config` stays as-is.

## 5. F-17 — DoS hygiene

### 5.1 MaxBytesReader on every handler

`internal/broker/server.go` gains a decorator:

```go
const maxRequestBodyBytes = 64 << 10 // 64 KiB; issue/validate/substitute/revoke bodies are tiny

func limitBody(next http.HandlerFunc) http.HandlerFunc {
    return func(w http.ResponseWriter, r *http.Request) {
        r.Body = http.MaxBytesReader(w, r.Body, maxRequestBodyBytes)
        next(w, r)
    }
}
```

All four handlers wrapped in `Server.Register`:

```go
mux.HandleFunc(brokerapi.PathIssue, limitBody(s.handleIssue))
mux.HandleFunc(brokerapi.PathValidateEgress, limitBody(s.handleValidateEgress))
mux.HandleFunc(brokerapi.PathSubstituteAuth, limitBody(s.handleSubstituteAuth))
mux.HandleFunc(brokerapi.PathRevoke, limitBody(s.handleRevoke))
```

Existing `BadRequest` handling on `json.NewDecoder.Decode` covers `*http.MaxBytesError` cleanly (400 surfaces as "decoding body: http: request body too large").

### 5.2 `http.Server` timeouts

`cmd/broker/main.go`:

```go
httpSrv := &http.Server{
    Addr:              listenAddr,
    Handler:           mux,
    TLSConfig:         tlsCfg,
    ReadHeaderTimeout: 10 * time.Second,
    ReadTimeout:       30 * time.Second,  // includes body read; broker bodies tiny — slow-write is hostile
    WriteTimeout:      30 * time.Second,  // bounds slow-read response side; no streaming responses
    IdleTimeout:       120 * time.Second, // keep-alive cap; the proxy sidecar reuses connections heavily
}

probeSrv := &http.Server{
    Addr:              probeAddr,
    Handler:           probeMux,
    ReadHeaderTimeout: 5 * time.Second,
    ReadTimeout:       5 * time.Second,
    WriteTimeout:      5 * time.Second,
    IdleTimeout:       30 * time.Second,
}
```

No operator-flag override — flag surface stays minimal. Values documented inline; legitimate workloads sit far below these limits.

### 5.3 Per-run rate limiter

New file `internal/broker/ratelimit.go`:

```go
type runLimiter struct {
    issue       *rate.Limiter // 5 req/s, burst 10
    substitute  *rate.Limiter // 50 req/s, burst 100
    lastTouched time.Time
}

type RunLimiterRegistry struct {
    mu    sync.Mutex
    runs  map[runKey]*runLimiter
    clock clockSource
}

type runKey struct{ Namespace, Name string }

func (r *RunLimiterRegistry) Allow(key runKey, kind string) bool { ... }
func (r *RunLimiterRegistry) Sweep(now time.Time)                { ... }
```

Wired into the issue and substitute handlers after authentication + `resolveRunIdentity`:

```go
if !s.RunLimiter.Allow(runKey{runNamespace, runName}, "substitute") {
    audit := &CredentialAudit{
        RunName:        runName,
        Namespace:      runNamespace,
        CredentialName: "(rate-limit)",
        Reason:         "rate limit exceeded for substitute-auth",
    }
    if wErr := s.Audit.CredentialDenied(ctx, *audit); wErr != nil {
        writeError(w, http.StatusServiceUnavailable, "AuditUnavailable", "...")
        return
    }
    writeError(w, http.StatusTooManyRequests, "RateLimited", "per-run rate limit exceeded")
    return
}
```

Phase 2c fail-closed-on-audit-failure pattern: audit-write before 429 response; 503 on audit-write failure.

A periodic goroutine started in `cmd/broker/main.go` sweeps entries untouched for > 5 minutes, freeing limiter state for runs that have completed.

Metrics:
```
paddock_broker_ratelimit_denied_total{kind="substitute|issue", namespace, run}
paddock_broker_ratelimit_active_buckets   // gauge, set by Sweep
```

**Limits are not operator-configurable in v0.4.** Hardcoded values are conservative for legitimate workloads (50 req/s substitute = 3000/min, far above any real run's upstream-connection rate). Follow-up issue tracks operator overrides for future tuning.

### 5.4 Audit debounce — DEFERRED

ADR-0016 specifies emitter-side debounce + per-`{kind, decision}` summary-event collapse. Currently `internal/broker/audit.go` is the simplest possible writer: one Create per event. A co-tenant with a legitimate SA token can flood the AuditEvent stream and shift cost to etcd.

Deferred to a separate follow-up issue — getting the summary-event semantics right (window-close timing, summary-event references to suppressed events, mid-window broker restart behavior) deserves its own design pass. The Theme 1 seed-pod audit re-enable (F-52) touches the same audit emitter; that's the natural place to bundle the work.

## 7. Migration / breakage / rollout

### 7.1 CRD evolution

`HarnessRun.status.IssuedLeases` is **additive** on `v1alpha1`. Pre-1.0 evolves in place (CLAUDE.md) — no new API version, no conversion webhook. Existing HarnessRuns at upgrade time have `IssuedLeases == nil`; the controller appends as Issue calls happen post-upgrade. Old runs finish with their existing TTL-driven cleanup.

`zz_generated_deepcopy.go` regenerated via `make generate`; CRD manifest regenerated via `make manifests`. Both regen targets gate the pre-commit hook.

### 7.2 Provider interface — breaking change

Adding `Revoke(ctx, leaseID) error` to `Provider` is a breaking change for any out-of-tree implementor. The project ships four in-tree providers; no known third-party providers exist. The commit adding the method takes `feat(broker)!:` so `release-please` picks up the marker. CHANGELOG (auto-generated) will list:

> **BREAKING CHANGES**
> - **broker:** `Provider` interface gains `Revoke(ctx, leaseID) error`. In-tree providers updated; out-of-tree providers must implement the new method or fail to compile.

### 7.3 Wire-protocol additions

- New endpoint: `POST /v1/revoke`. Older controllers (without the finalizer) never call it — broker can serve mixed-version controllers. Newer controllers calling `/v1/revoke` against an older broker get 404 `NotFound`; controller treats that as transient and the loud-warning force-clear branch fires after the budget exhausts.
- `IssueResponse` gains optional `PoolSecretRef *SecretKeyReference` and `PoolSlotIndex *int`. Additive; older clients ignore. Older brokers don't populate; controller's `deriveProviderRef` returns nil for `PoolRef` in that case (PATPool reconstruction is best-effort across mixed versions — pre-upgrade leases simply aren't reconstructable, which is the existing F-14 behavior).

**Upgrade ordering:** broker upgraded *before* controller. The standard `helm upgrade` order of `paddock-broker` then `paddock-controller-manager` honors this since the controller's readiness already depends on a reachable broker. No chart change required, but the constraint is documented in §8 operator notes.

**Downgrade:** rolling back the controller while the new broker is live is safe — old controller doesn't know about the finalizer, never adds it; broker keeps serving Issue/Revoke. Rolling back the broker while the controller still expects `/v1/revoke` is mildly noisy (controller calls 404, loud-warning force-clears) but not catastrophic. No pre-upgrade hook guard needed.

### 7.4 Tenant-visible breakage

**Rate limiting (§5.3) is the only behavioral change tenants can observe.** The 50 req/s substitute / 5 req/s issue defaults are far above any legitimate workload. A tenant hitting 429 is doing something pathological. The 429 response increments `paddock_broker_ratelimit_denied_total` for operator alerting — no silent throttling.

**No other tenant-visible behavior changes.** Body cap, server timeouts, readiness gating are server-side hardening — agents and proxies see no API change.

## 8. Operator notes

- Broker pod restarts now log `lease reconstruction had errors; broker continues with partial state` if a HarnessRun's `Status.IssuedLeases` references a pool slot the Secret no longer holds. Non-fatal; correlate with `paddock_broker_patpool_reconstruct_skipped_total`.
- HarnessRun delete now waits up to 30s for broker revocation. Tenants deleting runs in tight loops should expect that latency. `KEEP_E2E_RUN=1` is unaffected (skips namespace drain, not run delete).
- The new `paddock.dev/broker-leases-finalizer` is added by the controller. `kubectl get harnessrun -o yaml` shows two finalizers in `metadata.finalizers`; both must clear before deletion completes. `kubectl patch` removal should be reserved for genuinely stuck runs (e.g., broker has been gone > 1 hour) — the loud-warning force-clear branch already handles broker outage automatically.
- Helm upgrade order: `paddock-broker` Deployment before `paddock-controller-manager` Deployment. The standard chart honors this via existing readiness ordering.
