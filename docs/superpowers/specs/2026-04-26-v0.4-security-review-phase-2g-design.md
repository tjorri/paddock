# v0.4 Security Review — Phase 2g (Substitute-auth hygiene) Design

- Owner: @tjorri
- Date: 2026-04-26
- Status: Draft
- Predecessor: Phase 2f (Per-run intermediate CA — F-18, PR #34).
- Successor (planned): TBD via separate brainstorm — likely either F-11 (bearer revocation on run completion) or one of the remaining Theme C / Theme D findings.

## 1. Summary

Phase 2g closes four audit findings that all live on the substitute-auth keep-alive request loop:

- **F-09** (P1, Risk High) — vertical providers (`AnthropicAPI`, `GitHubApp`, `PATPool`) don't host-scope `SubstituteAuth`; an Anthropic bearer can be replayed against any host that any policy allowlists.
- **F-10** (P1) — broker `SubstituteAuth` doesn't re-fetch `HarnessRun` or re-call `matchPolicyGrant`; revoking a `BrokerPolicy` mid-run has no effect until the bearer's TTL (up to 1 h).
- **F-21** (P1, Risk High) — proxy forwards arbitrary agent-supplied headers, query params, and body verbatim alongside the swapped credential; only the broker-named keys are normalised.
- **F-25** (P2) — proxy doesn't re-call `ValidateEgress` per HTTP request inside a kept-alive MITM connection; combined with F-10, a hostile agent can keep using substituted credentials until the connection's idle timeout even after the run is Cancelled.

After Phase 2g:

- Every `SubstituteAuth` call re-validates the live `HarnessRun` phase, the matching `BrokerPolicy` credential grant, the matching egress grant for `host:port`, and the lease's allowed-host list. Policy revocation takes effect on the next request, not at TTL.
- Each vertical provider's lease records its `AllowedHosts` at `Issue` time (mirroring `UserSuppliedSecretProvider`'s existing pattern) and the substitute path rejects bearers used outside that host set.
- The proxy treats every header and query parameter not on the broker-supplied allowlist as agent-controlled and strips it before forwarding. An empty allowlist fails closed (only protocol-mandatory headers + the broker's explicit `SetHeaders` keys reach upstream).
- The bytes-shuttle (non-substitute MITM) path enforces an idle timeout so revocation reaction time is bounded on opaque TLS tunnels too.

The phase touches `internal/broker/server.go` (handler re-validation), `internal/broker/providers/{anthropic,githubapp,patpool}.go` (lease shape + host-scope check), `internal/broker/providers/provider.go` and `internal/broker/api/api.go` (additive `AllowedHeaders` / `AllowedQueryParams` fields on `SubstituteResult` / `SubstituteAuthResponse`), `internal/proxy/{substitute,server,mode,broker_client}.go` (allowlist enforcement, idle timeout, new error mapping), `cmd/proxy/main.go` (new `--idle-timeout` flag), `test/e2e/hostile_test.go` (three new scenarios) plus `test/hostile/main.go` (new `--smuggle-headers` flag), plus the audit-findings doc and threat-model resolution lines.

## 2. Goals and non-goals

**Goals:**

- Close F-09, F-10, F-21, F-25 with verifiable unit + e2e coverage.
- Keep the broker as the single fail-closed enforcement point on the substitute path. Proxy becomes a wire-shape and connection-lifecycle enforcer, not a policy enforcer.
- Land the F-21 contract change (broker-declared allowlist) and the per-provider strict allowlists in the same PR so the proxy's allowlist enforcement always has a populated source.
- Preserve the existing `audit-then-respond, fail-closed-on-audit-failure` pattern from Phase 2c for every new denial reason.

**Non-goals (deferred):**

- **F-11** (bearer revocation on run completion / `Revoke` endpoint / finalizer) — separate phase. The §4 fail-closed re-check materially shrinks F-11's window (Cancelled run can't substitute even with a still-valid bearer) but doesn't replace it.
- **F-26** (connection limits, per-run rate limit, `MaxBytesReader` on bodies) — separate finding. The §6 idle timeouts close the slow-loris hole as a side benefit but don't deliver F-26's primary recommendations.
- **HTTP/2 substitute-auth.** Substitute path stays HTTP/1.1-only; F-21 enforcement is HTTP/1.1-shaped (header allowlist over `req.Header`).
- **Per-host body MIME-type allowlist** (audit F-21 bullet 4) — softer Phase 3 control; tracked as follow-up.
- **Proxy-side decision cache.** Broker informer-cache lookups are sub-millisecond; no cache needed in this phase.
- **Provider-author-facing override of `AllowedHeaders` on `UserSuppliedSecretProvider`.** Defaults shipped; per-grant override deferred until a user asks.

## 3. Architecture

The substitute-auth path collapses to a single fail-closed enforcement point in the broker. Per-request decisions live with the broker because the live `BrokerPolicy` informer cache and the `HarnessRun` watch are already there; the proxy stays a wire-shape enforcer (strip headers / query-params not on the broker's allowlist) and a connection-lifecycle enforcer (idle-timeout the bytes-shuttle path).

Three new enforcement boundaries:

1. **Broker `SubstituteAuth` handler** (`internal/broker/server.go::handleSubstituteAuth`) re-validates run phase + policy grant + egress grant + lease host-scope on every call. Returns structured denial codes the proxy maps to AuditEvent reasons.
2. **Proxy `applySubstitutionToRequest`** (`internal/proxy/substitute.go`) strips every header and query-param not on the broker-supplied `AllowedHeaders` / `AllowedQueryParams` (union with `SetHeaders` / `SetQueryParam` keys + a fixed `mustKeep` set).
3. **Proxy connection lifecycle** — substitute-loop iteration AND bytes-shuttle `io.Copy` halves enforce a manager-flag idle deadline. Substitute-loop already drops on broker error; the new fail-closed denials surface as cleanly typed errors. Bytes-shuttle gets a deadline-extending reader wrapper.

The vertical-provider lease shape gains an `AllowedHosts []string` field populated at `Issue` time from `CredentialGrant.Provider.Hosts` (with provider defaults: Anthropic → `[api.anthropic.com]`, GitHubApp → `[github.com, api.github.com]`, PATPool → required-from-grant). `UserSuppliedSecretProvider` already does this; the change is to mirror its pattern in the three vertical providers.

No CRD schema change. The proxy↔broker wire format gains optional `AllowedHeaders` / `AllowedQueryParams` fields on `SubstituteAuthResponse` — additive, pre-v1, fine to evolve in place per the project convention.

## 4. Broker — `SubstituteAuth` re-validation (F-09 + F-10)

### 4.1 Lease shape

Add `AllowedHosts []string` to `anthropicLease`, `githubAppLease`, `patPoolLease`. Populated by each provider's `Issue` from `grant.Provider.Hosts` (or the per-provider default if the grant omits it). Mirrors `UserSuppliedSecretProvider`'s existing field; reuse the package-private `hostMatchesGlobs` helper rather than duplicate it.

Per-provider defaults:

| Provider | Default when `grant.Provider.Hosts` is empty |
|---|---|
| AnthropicAPI | `["api.anthropic.com"]` |
| GitHubApp | `["github.com", "api.github.com"]` |
| PATPool | none — admission requires `grant.Provider.Hosts` (verify in `internal/webhook/v1alpha1/brokerpolicy_webhook.go`; tighten if not) |

If `grant.Provider.Hosts` is set, it **replaces** the default — no merging — to match the existing override pattern and stay surprise-free.

### 4.2 Handler flow

`handleSubstituteAuth` after authentication and request decode, before walking providers, performs the run-fetch:

```
1. Get HarnessRun by (runNamespace, runName) from cached client
   → 404                                          ⇒ 404 RunTerminated, audit "run not found"
   → run.Status.Phase ∈ {Cancelled,Succeeded,Failed} ⇒ 403 RunTerminated, audit "run terminated: <phase>"
```

Then for each provider that returns `Matched=true` from its existing bearer-prefix check, **before** calling its `SubstituteAuth` to fetch the secret, the broker performs:

```
2. matchPolicyGrant(ctx, c, runNamespace, run.Spec.TemplateRef.Name, lease.CredentialName)
   → nil grant                                    ⇒ 403 PolicyRevoked, audit "no BrokerPolicy grants this credential"
   → grant.Provider.Kind != lease.providerKind    ⇒ 403 PolicyRevoked, audit "policy provider kind changed"
3. matchEgressGrant(ctx, c, runNamespace, templateName, req.Host, req.Port)
   → nil grant                                    ⇒ 403 EgressRevoked, audit "no BrokerPolicy grants egress to host:port"
4. Provider's SubstituteAuth — existing checks + new lease.AllowedHosts host-scope via hostMatchesGlobs
   → host mismatch                                ⇒ 403 HostNotAllowed, audit "bearer not scoped for host"
   → expired / unknown / namespace mismatch       ⇒ existing paths preserved
```

### 4.3 Credential-name derivation

`SubstituteAuth` doesn't carry the credential name explicitly today — it identifies by bearer prefix. To run `matchPolicyGrant`, the broker needs the name. Each vertical provider's lease already records the originating credential name (`anthropicLease.CredentialName`, etc. — verify field names; add where missing). The reordered handler:

```
for each provider:
    if !provider.Substituter: continue
    if !provider.Owns(bearer): continue  // bearer-prefix + lease lookup; no secret read yet
    // We now know lease.CredentialName.
    re-validate (steps 2 + 3 above).
    if denied: write audit, return error, do not call SubstituteAuth.
    call provider.SubstituteAuth (step 4 above).
    write audit, return result.
```

`Owns` is a new tiny method on the `Substituter` interface (or an internal helper) that returns `(lease, ok)` without reading the secret — splits the existing `SubstituteAuth` body so re-validation can happen between bearer-recognition and secret-fetch. Keeps the secret read out of the path when policy is already revoked.

### 4.4 Failure shape

New broker-side error codes returned on the existing `writeError` path:

- `RunTerminated` (404 if not-found, 403 if terminal phase)
- `PolicyRevoked` (403)
- `EgressRevoked` (403)
- `HostNotAllowed` (403)

Each carries a deterministic `reason` string the proxy maps to a `Warning EgressDenied` or `Warning CredentialRevoked` event on the owning HarnessRun. Existing `BearerUnknown` / `SubstituteFailed` codes preserved.

### 4.5 Audit emission

Every new denial path writes a `CredentialDenied` AuditEvent before returning the error (the Phase 2c pattern: audit-then-respond, fail-closed on audit-write failure). Existing `CredentialIssued` audit on the success path stays.

### 4.6 No proxy-side cache

Each substitute request triggers one broker call as today; the new re-checks are informer-cache lookups (sub-millisecond). If profiling later shows broker CPU on the substitute path, add a per-(run, credential, host, port) 5-second LRU on the broker side — not the proxy. Out of scope for this phase.

## 5. Proxy — header/query allowlist enforcement (F-21)

### 5.1 Contract change

Additive fields on `providers.SubstituteResult` (and the wire shape `brokerapi.SubstituteAuthResponse`):

```go
type SubstituteResult struct {
    Matched            bool
    SetHeaders         map[string]string
    RemoveHeaders      []string
    SetQueryParam      map[string]string
    SetBasicAuth       *BasicAuth
    AllowedHeaders     []string  // NEW: header names permitted to forward; case-insensitive
    AllowedQueryParams []string  // NEW: query-param keys permitted to forward; case-sensitive
}
```

### 5.2 Per-provider defaults

| Provider | `AllowedHeaders` | `AllowedQueryParams` |
|---|---|---|
| AnthropicAPI | `Content-Type, Content-Length, Accept, Accept-Encoding, User-Agent, Anthropic-Version, Anthropic-Beta` | `[]` |
| GitHubApp | `Content-Type, Content-Length, Accept, Accept-Encoding, User-Agent, X-GitHub-Api-Version` | `[]` |
| PATPool | same as GitHubApp | `[]` |
| UserSuppliedSecret | `Content-Type, Content-Length, Accept, Accept-Encoding, User-Agent` | `[]` |

The exact lists are decided in the plan; the design commits to "minimal protocol-relevant set per provider, no agent-correlation headers".

### 5.3 Enforcement

`applySubstitutionToRequest` is rewritten:

```
1. Apply RemoveHeaders (existing).
2. Apply SetHeaders (existing).
3. Build allowedHeaderSet := lowercase union of:
     AllowedHeaders ∪ keys(SetHeaders) ∪ mustKeepHeaders
   where mustKeepHeaders = {"host", "content-length", "content-type", "transfer-encoding"}
4. For each header in req.Header: if !allowedHeaderSet.Has(lower(name)) ⇒ delete.
5. allowedQPSet := AllowedQueryParams ∪ keys(SetQueryParam).
   For each query param key in req.URL.Query(): if !allowedQPSet.Has(key) ⇒ delete.
6. Apply SetQueryParam, SetBasicAuth (existing).
```

`mustKeepHeaders` covers the bare HTTP/1.1 wire necessities. `Authorization` is deliberately not in `mustKeep` — the broker explicitly removes it via `RemoveHeaders` for Anthropic and overwrites it for GitHubApp / PATPool; if a future provider returns neither, the proxy strips it.

### 5.4 Defence-in-depth property

If a provider returns an empty `AllowedHeaders` (forgotten / regression), the proxy strips *everything* except `mustKeepHeaders` and `SetHeaders` keys — the request still goes upstream with the substituted credential and the bare HTTP necessities, never with agent-smuggled headers. Empty-allowlist is a fail-closed default, not a fail-open one.

### 5.5 Body forwarding (out of scope, see §2)

Body MIME-type allowlisting is the audit's "softer Phase 3 control" and needs per-provider semantics we don't have. Tracked as a follow-up; documented in non-goals.

## 6. Proxy — bytes-shuttle idle timeout (F-25 part 2)

### 6.1 Manager flag

New flag on `cmd/proxy/main.go`: `--idle-timeout duration` (default `60s`). Plumbed into `proxy.Server.IdleTimeout`.

### 6.2 Bytes-shuttle implementation

`internal/proxy/mode.go` (transparent path) and `internal/proxy/server.go` (cooperative bytes-shuttle path) wrap each `io.Copy` half (`client→upstream` and `upstream→client`) in a `deadlineExtendingReader`:

```go
type deadlineExtendingReader struct {
    conn    net.Conn
    timeout time.Duration
}

func (r *deadlineExtendingReader) Read(p []byte) (int, error) {
    _ = r.conn.SetReadDeadline(time.Now().Add(r.timeout))
    return r.conn.Read(p)
}
```

When the deadline elapses with no data, the read returns `os.ErrDeadlineExceeded`, the copy returns, the goroutine pair tears down the connection cleanly. `errors.Is(err, os.ErrDeadlineExceeded)` distinguishes idle-close from real errors in the log line ("connection idle-closed after 60s" vs "connection broken: …").

### 6.3 Substitute-loop coverage

`handleSubstituted`'s `http.ReadRequest` has no deadline today. Add the same idle-deadline pattern: `clientConn.SetReadDeadline(time.Now().Add(idleTimeout))` before each `http.ReadRequest`, cleared on success. Catches the slow-loris vector named in F-26 as a side benefit (without claiming to fix F-26 — that's a separate phase).

### 6.4 No effect on substitute correctness

Idle timeout doesn't change the per-request fail-closed semantics from §4 — those fire regardless of how long the connection has been alive. Idle timeout only bounds *how long* a stale-policy connection can live unused before the agent has to re-CONNECT and re-validate.

## 7. Failure modes & edge cases

- **HarnessRun deleted but informer cache stale.** `Get` returns NotFound; substitute returns `RunTerminated 404`. The cached client lags by at most one watch event (typically <100 ms); F-10 is closed within that window.
- **BrokerPolicy edited mid-substitute.** New informer event arrives, next `matchPolicyGrant` returns the new grant. If the credential is removed entirely, returns `PolicyRevoked`. If only the egress grant is removed, returns `EgressRevoked`. Both fail-closed.
- **Provider lease without `CredentialName` set.** Pre-2g leases don't have the field; broker restart wipes lease state for AnthropicAPI / GitHubApp and (per F-14) also for PATPool. Net effect: no lease-shape migration is needed — leases are in-memory, so the upgrade window has no leases to migrate.
- **Empty `AllowedHeaders` on the wire.** Treated as fail-closed (strip everything except `mustKeep` ∪ `SetHeaders` keys). Existing pre-2g brokers don't send the field; if a 2g+ proxy talks to a pre-2g broker, it strips agent headers — slightly stricter than the old behaviour. Tightening is the intended direction; documented in §10.
- **Bytes-shuttle long-poll workloads.** A legitimate workload doing 5-minute long-polls against `api.anthropic.com` would get idle-closed at 60s. Mitigation: the substitute path's per-request loop doesn't count as idle (each request resets the deadline). Bytes-shuttle is the non-substitute path — long-polls there are agent-to-allowlisted-host with no MITM; the timeout still terminates them. Manager flag lets operators raise the timeout if this becomes a real workload pattern.
- **Cooperative-mode CONNECT to a substitute host.** Cooperative mode's MITM flow already routes through the same `handleSubstituted`; the new idle deadline applies. No special-casing.
- **Audit write fails on a denial path.** Existing Phase 2c pattern: return 503 `AuditUnavailable`; proxy drops the connection. The agent retries; the policy is still revoked; the next attempt also denies. No silent-fail.

## 10. Migration / breaking-change posture

Pre-1.0 evolves in place per project convention. Breaking changes:

1. **`SubstituteResult` / `SubstituteAuthResponse` shape change.** Additive — pre-2g providers compile against a struct with two extra fields they leave zero-valued; pre-2g brokers don't emit the wire fields. The proxy treats absent fields as empty allowlists, which the §5.4 fail-closed semantic handles correctly. No version skew rejection needed.
2. **New broker error codes.** Proxy must learn to map `RunTerminated`, `PolicyRevoked`, `EgressRevoked`, `HostNotAllowed` to the appropriate AuditEvent reason. Old proxy + new broker: existing proxy treats unknown error codes as "drop the connection" (per existing behaviour), which is the desired fail-closed outcome — agent gets a TLS error and retries, retry gets the same denial.
3. **Stricter outbound headers.** A hypothetical existing workload that relied on the proxy forwarding e.g. `X-Custom-Tracking` to upstream now has that header stripped. Documented as a deliberate tightening; release notes call out the per-provider allowlists. No accommodation needed pre-v1.
4. **Idle timeout default 60s.** Conservative; configurable via flag. Upgrade note in CHANGELOG.

Each behaviour-changing commit takes `feat(security)!:` so `release-please` picks the breaking-change marker up.

## 11. Risks and mitigations

- **Risk: bearer→lease lookup race.** A bearer that was just issued (in-flight Issue request not yet returned) could be substituted before its lease has the new fields. Mitigation: `Issue` populates `AllowedHosts` synchronously *before* returning the bearer to the caller; lock on the lease map already serialises observers. Verify in tests.
- **Risk: empty `AllowedHosts` regression.** A future provider author forgets to populate the field; substitute calls then trivially pass the host check (empty list matches everything in `hostMatchesGlobs` semantics today — verify and tighten if so). Mitigation: require non-empty `AllowedHosts` on lease construction; panic-or-return-error at Issue time. Add a unit test that asserts each provider populates the field.
- **Risk: new error codes leak provider-internal detail.** Reason strings should be terse and not include credential names or provider-internal lease state. Mitigation: explicit format strings at the call site (audit text on the AuditEvent), not `err.Error()` propagation.
- **Risk: idle timeout breaks long-poll.** Operators with long-poll workloads (rare, but possible against e.g. `api.openai.com` even though that's not a Paddock-supplied provider) get connections idle-closed at 60s. Mitigation: configurable flag; document in release notes; the substitute-path per-request reset means SSE/streaming through the substitute path is unaffected.
- **Risk: scope creep into F-11.** Implementing F-10 on the substitute path makes a Cancelled run unable to substitute; this looks like F-11 but doesn't actually revoke the bearer. A future audit reader might think F-11 is closed by 2g — it isn't. Mitigation: explicit non-goal in §2; F-11 phase doc cross-references this design.

## 13. Open questions

1. **Provider `Owns(bearer) (lease, ok)` interface vs. handler-side bearer→lease split.** §4.3 leans on a new tiny method on `Substituter` so re-validation can sit between bearer-recognition and secret-fetch. Alternative: handler reaches into a shared `BearerRegistry` that providers register with at Issue time. Decide at plan-writing time; pick whichever requires less surgery on existing provider code.
2. **Reason-string canonicalisation.** Should the broker return a stable enum-like `code` field on the wire and let the proxy compose the human-readable AuditEvent reason locally, or return the full reason string? Phase 2c established that audit text is owned by the *emitter* of the event — argues for the proxy composing locally from the code. Confirm during plan writing.
3. **`UserSuppliedSecretProvider` `AllowedHeaders` override per grant.** Defaults shipped per §5.2; per-grant override deferred. If users immediately ask for it, fold into 2g; otherwise track as a follow-up.
4. **PATPool `grant.Provider.Hosts` admission requirement.** §4.1 assumes admission already requires it; verify, and if not, add the validation in this phase (small admission-webhook change).
5. **Idle-timeout flag scoping.** Single `--idle-timeout` covers both substitute-loop and bytes-shuttle, or separate `--substitute-idle-timeout` and `--bytes-shuttle-idle-timeout`? Default to single flag for simplicity unless a workload pattern demands the split.
