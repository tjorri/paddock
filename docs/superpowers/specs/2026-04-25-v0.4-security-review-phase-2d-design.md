# v0.4 Security Review — Phase 2d (carry-overs) Design

- Owner: Tuomo
- Date: 2026-04-25
- Status: Draft
- Predecessor: Phase 2c (Theme A audit completeness, merged in #26); doc backfill (#29).
- Successor (planned): one of the larger themes (Theme B per-run isolation, Theme C runtime egress, Theme D broker robustness, Theme E pod-spec hardening, Theme F workspace seed) — chosen via a separate brainstorm.

## 1. Summary

Phase 2d closes three small carry-overs from Phase 2c:

- **kube-apiserver NetworkPolicy egress** — Phase 2b's workaround was to skip per-run NP emission for templates with empty `requires`, because standard NetworkPolicy podSelectors can't match host-network destinations like the kube-apiserver. The fix: auto-discover the apiserver IP set from the controller's kubeconfig at manager startup and add a TCP/443 allow rule (pod-wide; security posture relies on F-38's `automountServiceAccountToken: false` to deny the agent any apiserver identity). Drop the empty-`requires` skip.
- **F-41 — `Owns()` declarations** for `Secret`, `NetworkPolicy`, `ServiceAccount`, `Role`, `RoleBinding` so mid-run `kubectl edit/delete` of these resources triggers an immediate reconcile and re-converges via the existing `CreateOrUpdate` flow. Plus a paired change: terminal-phase runs whose Job is still draining keep reconciling so the new watches do useful work during the SIGTERM window.
- **F-43 — NP-flip lifecycle fix + audit emission.** Record `status.networkPolicyEnforced` at admission; the reconciler honours that for the run's lifetime, regardless of subsequent flag flips on the manager. New `network-policy-enforcement-withdrawn` AuditKind for the residual case (operator deletes the per-run NP via `kubectl`, F-41's new watch triggers reconcile, the reconciler re-creates the NP and emits the audit).

The phase touches `internal/controller/`, `api/v1alpha1/`, `cmd/main.go`, `internal/auditing/`, plus ADR-0013 + the audit findings doc. No broker / proxy / webhook code is modified.

## 2. Goals and non-goals

**Goals:**

1. Per-run NetworkPolicy is emitted for every admitted run, with an apiserver allow rule that lets sidecars (collector, adapter) reach the kube-apiserver. The empty-`requires` skip workaround is removed.
2. Mid-run mutations of any owned resource (Secret / NP / SA / Role / RoleBinding) are reverted by the reconciler.
3. Terminal-phase runs whose Job is still draining (within `terminationGracePeriodSeconds`) keep reconciling so tampering during the SIGTERM window is reverted.
4. Per-run NP enforcement is set at admission and immutable for the run's lifetime; flag flips on the manager affect *new* runs only.
5. Operator-side withdrawal of a per-run NP (e.g., `kubectl delete networkpolicy <run>-egress`) emits a `network-policy-enforcement-withdrawn` AuditEvent and is re-converged.
6. Audit findings F-41 and F-43 have explicit Resolution lines.

**Non-goals (deferred):**

- F-42 (`terminationGracePeriodSeconds` upper bound) — separate phase.
- F-41 sub-item (3) — `<run>-broker-creds` envFrom-injection key validation; Question 2 explicitly excluded.
- IPv6 cluster support — every existing NP rule in this codebase uses IPv4 ipBlock CIDRs; dual-stack support is its own future work.
- Periodic apiserver IP refresh — manager restart picks up new IPs for new runs.
- CiliumNetworkPolicy variant for kube-apiserver — Question 1 explicitly excluded; portability across CNIs takes precedence.

## 3. Architecture

Three loosely-coupled changes, all in the controller package:

1. **Apiserver IP discovery** at controller-manager startup. New helper resolves `cfg.Host` to a set of `[]net.IP`. Stored on `HarnessRunReconciler.APIServerIPs`. Threaded through `networkPolicyConfig` to the NP builder.
2. **NP shape change.** `buildRunNetworkPolicy` and `buildSeedNetworkPolicy` gain an apiserver allow rule (TCP/443 to each resolved IP via ipBlock). The empty-`requires` skip in `harnessrun_controller.go` is removed.
3. **F-43 lifecycle fix.** New `Status.NetworkPolicyEnforced *bool` field on HarnessRun; pinned at admission time; the reconciler reads from status, never the live flag. Operator-side NP deletion (caught by F-41's new watch) re-creates the NP and emits a new `network-policy-enforcement-withdrawn` AuditEvent.

`SetupWithManager` adds five new `Owns()` declarations. The main `Reconcile` short-circuit on `isTerminal` is qualified with "AND no Job exists" so terminal-while-draining runs keep reconciling.

## 4. Apiserver IP discovery

A new helper in `internal/controller/apiserver_ips.go`:

```go
// apiserverIPsFromConfig returns the IP set the controller's kubeconfig
// resolves the apiserver to. Used to seed a per-run NetworkPolicy
// allow rule on TCP/443 from sidecar pods.
//
// Behaviour:
//   - cfg.Host is parsed as a URL. The port is stripped.
//   - If the host is an IP literal, returns that single IP.
//   - If the host is a hostname, net.LookupIP is called and all returned
//     IPv4 addresses are returned.
//   - IPv6 addresses are filtered out — every existing NP rule in the
//     codebase uses IPv4 ipBlock CIDRs. Dual-stack support is future work.
//   - An empty result is treated as a hard error; manager startup fails
//     fast rather than starting with a permissively-configured controller.
func apiserverIPsFromConfig(cfg *rest.Config) ([]net.IP, error)
```

`HarnessRunReconciler` gains a field:

```go
type HarnessRunReconciler struct {
    // ... existing fields ...

    // APIServerIPs is the set of IPs the controller's kubeconfig resolves
    // the kube-apiserver to. Set once at manager startup; per-run
    // NetworkPolicies include a TCP/443 allow rule for each entry so
    // sidecars (collector for AuditEvent emission, adapter for status
    // writes) can reach the apiserver. F-41 / Phase 2d.
    APIServerIPs []net.IP
}
```

`networkPolicyConfig` gains the same field, populated from the reconciler.

`cmd/main.go` calls `apiserverIPsFromConfig` immediately after `ctrl.GetConfig()` returns and before the reconciler is constructed. Failure exits with `setupLog.Error` and `os.Exit(1)`.

**HA caveat documented in code comments:** if the apiserver moves IPs (control-plane node replacement), existing per-run NPs hold the old IPs until the run terminates. Acceptable for the run's lifetime budget. New runs after the move pick up the new IPs from a fresh `apiserverIPsFromConfig` call — but that requires a manager restart. Periodic refresh would add complexity without a known need; revisit when an operator hits it.

## 5. Per-run NP shape change

`buildRunNetworkPolicy` (`internal/controller/network_policy.go:170`) gains a new egress rule when `cfg.APIServerIPs` is non-empty:

```go
// Apiserver egress allow rule. Sidecars (collector for AuditEvent
// emission, adapter for status writes) need TCP/443 to the kube-
// apiserver. Pod-wide because NetworkPolicy operates at pod level;
// the agent container shares the network namespace with sidecars.
// F-38 (Phase 2a) ensures the agent has automountServiceAccountToken=
// false, so the apiserver rejects any request the agent might forge.
{
    To: []networkingv1.NetworkPolicyPeer{
        // One peer per resolved apiserver IP (HA control planes resolve to N IPs).
        {IPBlock: &networkingv1.IPBlock{CIDR: ip + "/32"}}, // for each ip in cfg.APIServerIPs
    },
    Ports: []networkingv1.NetworkPolicyPort{
        {Protocol: &tcp, Port: &httpsPort},
    },
},
```

The same rule shape is added to `buildSeedNetworkPolicy` (workspace seed pods need apiserver access too).

**The empty-`requires` skip in `harnessrun_controller.go:495` is removed.** All admitted runs now get a per-run NP because the new apiserver rule fixes the actual problem the skip was working around. The dead `deleteRunNetworkPolicy` cleanup branch (the `else` arm of that if) is also removed — it's only reachable via flag-flip, which F-43 handles separately.

The long parenthetical comment in `network_policy.go:142–154` (about why there's no apiserver rule) is deleted since it's now wrong.

**Security trade-off acknowledged.** Adding an apiserver allow rule pod-wide means the agent container also gets a network path to the apiserver. Mitigation:

- Agent has `automountServiceAccountToken: false` (F-38) — no projected SA token to authenticate with.
- The apiserver rejects unauthenticated requests via standard RBAC.
- Even if the agent obtained a forged bearer, the broker's TokenReview audience-check would reject it.

The /32 ipBlock leak is small; the threat model from F-38 + apiserver RBAC + broker re-validation already covers the abuse path. Two-NetworkPolicy alternatives (one for sidecars only) were considered and rejected as disproportionate complexity for marginal defence-in-depth gain.

## 6. F-43 lifecycle fix + audit emission

### 6.1 Status field on HarnessRun

Add to `api/v1alpha1/harnessrun_types.go` `HarnessRunStatus`:

```go
// NetworkPolicyEnforced records whether per-run NetworkPolicy enforcement
// was active when this run was admitted. Immutable after admission. The
// reconciler honours this for the run's lifetime, so a flag flip on the
// controller-manager (--networkpolicy-enforce=on → off) does not weaken
// running pods. F-43.
// +optional
NetworkPolicyEnforced *bool `json:"networkPolicyEnforced,omitempty"`
```

The field is set by the reconciler on the *first* reconcile (when `status.NetworkPolicyEnforced == nil`) from the current value of `r.networkPolicyEnforced()`. Subsequent reconciles read the stored value, never the live flag.

`ensureRunNetworkPolicy` is rewritten to consume `run.Status.NetworkPolicyEnforced` rather than calling `r.networkPolicyEnforced()` directly. When `*run.Status.NetworkPolicyEnforced == true`, the reconciler ensures the NP exists; when `false`, it never creates one. The current "delete on flag-flip" path in `network_policy.go:144–148` is removed.

**Migration for existing runs.** Runs admitted before this phase have `status.NetworkPolicyEnforced == nil`. The reconciler reads `nil` as "use live flag this once, then pin" — i.e., the first post-upgrade reconcile sets the field from `r.networkPolicyEnforced()` and persists it. From that point forward the live flag is irrelevant for this run.

### 6.2 New AuditKind + emission

Add to `api/v1alpha1/auditevent_types.go`:

- `AuditKindNetworkPolicyEnforcementWithdrawn AuditKind = "network-policy-enforcement-withdrawn"`.
- Extend the kubebuilder `+kubebuilder:validation:Enum=...` annotation.

Add a builder in `internal/auditing/builders.go`:

```go
type NetworkPolicyEnforcementWithdrawnInput struct {
    RunName, Namespace, Reason string
    When time.Time
}

func NewNetworkPolicyEnforcementWithdrawn(in NetworkPolicyEnforcementWithdrawnInput) *paddockv1alpha1.AuditEvent
```

GenerateName prefix: `ae-np-` (new family).

Add an emit method to `ControllerAudit` (`internal/controller/audit.go`):

```go
func (c *ControllerAudit) EmitNetworkPolicyEnforcementWithdrawn(ctx context.Context, runName, namespace, reason string)
```

**Emit site.** Inside `ensureRunNetworkPolicy`, when `controllerutil.CreateOrUpdate` returns `OperationResultCreated` AND `status.NetworkPolicyEnforced == true` AND the run is past its first reconcile (status was already pinned). That sequence means "the NP didn't exist on this reconcile, but it should — something deleted it." First reconcile sets up the NP normally and does NOT emit a withdrawal audit.

### 6.3 ADR-0013 §7.4 update

Adds a paragraph documenting the new immutable-at-admission semantics:

> "Enforcement decision is captured in `HarnessRun.status.networkPolicyEnforced` at admission and persists for the run's lifetime; flag flips on the manager affect new runs only. The `network-policy-enforcement-withdrawn` AuditEvent surfaces operator-side deletions of the per-run NP so they are not silently ignored. Combined with F-41's `Owns(&networkingv1.NetworkPolicy{})` watch, an operator who manually deletes the per-run NP triggers an immediate reconcile that re-creates it and records the withdrawal."

## 7. F-41 Owns() + terminal-while-draining

### 7.1 SetupWithManager additions

Edit `internal/controller/harnessrun_controller.go` `SetupWithManager` (~lines 1295–1310):

```go
return ctrl.NewControllerManagedBy(mgr).
    For(&paddockv1alpha1.HarnessRun{}).
    Owns(&batchv1.Job{}).
    Owns(&corev1.ConfigMap{}).
    Owns(&paddockv1alpha1.Workspace{}).
    Owns(&corev1.Secret{}).
    Owns(&networkingv1.NetworkPolicy{}).
    Owns(&corev1.ServiceAccount{}).
    Owns(&rbacv1.Role{}).
    Owns(&rbacv1.RoleBinding{}).
    Named("harnessrun").
    Complete(r)
```

`rbacv1` is already imported via the RBAC marker; no new import needed.

Behaviour: `kubectl edit secret <run>-broker-creds`, `kubectl delete networkpolicy <run>-egress`, etc., generate a watch event keyed by the owner reference and trigger a reconcile. The existing `CreateOrUpdate` flow inside each `ensure*` function re-converges the resource on the next pass.

### 7.2 Terminal-while-draining reconvergence

Today's short-circuit (`harnessrun_controller.go:196–198`) becomes:

```go
// Terminal phase: short-circuit unless the Job still exists. While
// the Pod is draining within terminationGracePeriodSeconds, an edit
// to a mounted Secret / SA / Role would be read by the agent on next
// access; keep reconciling so the new Owns() watches can revert
// tampering. F-41.
if isTerminal(run.Status.Phase) {
    if run.Status.JobName == "" {
        return ctrl.Result{}, nil
    }
    var job batchv1.Job
    err := r.Get(ctx, client.ObjectKey{Namespace: run.Namespace, Name: run.Status.JobName}, &job)
    if apierrors.IsNotFound(err) {
        return ctrl.Result{}, nil
    }
    if err != nil {
        return ctrl.Result{}, err
    }
    // Job still exists; fall through to re-converge owned resources.
}
```

Falling through is safe because each `ensure*` step is idempotent. The reconciler re-runs `ensurePromptSecret`, `ensureBrokerCredentials`, `ensureCollectorRBAC`, `ensureProxyTLS`, `ensureBrokerCA`, `ensureRunNetworkPolicy` — all `CreateOrUpdate`-based, all revert tampering. It will *not* re-create the Job (the existing reconcile flow doesn't recreate a deleted Job for a terminal run).

**`commitStatus` interaction.** The Phase 2c `emitTerminalAudit` is gated by `!isTerminal(orig.Phase) && isTerminal(run.Status.Phase)`. During terminal-while-draining reconciles both `orig` and `run.Status.Phase` are already terminal, so the gate keeps duplicate `run-completed` events from being emitted. No change needed.

## 10. Risks and mitigations

1. **HA / IP rotation.** Existing per-run NPs hold old apiserver IPs after a control-plane node replacement. Mitigation: bounded by run lifetime; restart picks up new IPs for new runs. Document; revisit when an operator hits it.
2. **Startup auto-detection failure.** If `apiserverIPsFromConfig` returns empty / errors, manager startup fails fast. Preferred over silent permissive NPs.
3. **Owns() watch volume.** All five new owned resources are per-run and stable; controller-runtime deduplicates by key. Should be invisible.
4. **Terminal-while-draining reconcile pressure.** Bounded by `terminationGracePeriodSeconds`; F-42's known upper-bound gap means a malicious template could in theory reconcile for up to 24h after deletion. Acceptable for this phase; F-42 is its own fix.
5. **Status migration for runs from before this phase.** The reconciler reads `Status.NetworkPolicyEnforced == nil` as "use live flag once, then pin." First post-upgrade reconcile pins. Documented in code.
6. **NP-shape change for empty-`requires` test fixtures.** Existing e2e specs (TG-2, TG-7, TG-24) use `evil-echo` templates with non-empty `requires`, so they were already getting an NP. Templates with empty `requires` (smoke fixtures, dev-only) now also get an NP — should be transparent because the new apiserver rule lets sidecars work.
