## Context

Theme 3 of the v0.4 security follow-up roadmap closes seven Open admission-side findings from the 2026-04-26 recheck. All seven share one code surface: the `BrokerPolicy` and `HarnessTemplate` admission webhooks plus chart-level rollout shape. None require runtime changes to the proxy hot path or the broker hot path.

F-23 spans both runtime and admission halves; only the admission half (overlap with F-34) is in scope here. Its runtime halves — `splitConnectTarget` and `splitHostPort` cluster-internal/IP-literal hardening — defer to Theme 4 (runtime egress residuals) where they sit alongside F-22, F-20, F-19 residual.

F-29 is technically a proxy startup-time validator rather than an admission webhook. It belongs to Theme 3 because its fix shape (parse-time refusal of bad strings, fail before entering the hot path) matches the webhook tightening pattern, and it pairs naturally with F-01's controller-side admission complement.

Pre-1.0, breaking changes are accepted in place rather than flag-aliased (CLAUDE.md). Existing `BrokerPolicy` / `HarnessTemplate` objects in etcd remain (Kubernetes admission only validates on write); operators discover the rejections on first edit. CHANGELOG entries land via individual `feat(security)!:` Conventional Commit markers picked up by `release-please`.

## Goals

- Close F-08, F-23 (admission half), F-29, F-30, F-34, F-35, F-36 in a single PR, commit-decomposed per finding so each `feat(security)!:` Conventional Commit lands its own breaking-change marker for `release-please`.
- Reuse Phase 2 patterns wherever possible (`field.ErrorList` composition, `BrokerPolicyCustomValidator.audit`'s policy-applied/policy-rejected emission, the existing wildcard-position rules in `validateHosts`/`validateRequireSpec`). No new admission machinery.
- Pull the per-finding host validation rules into one shared `validateExternalHost` helper so `BrokerPolicy` and `HarnessTemplate` admission stay in lockstep, and any future host-bearing field (proxy endpoint, future admission objects) can call into the same checker.

## Non-goals

- Theme 4 work (proxy `splitConnectTarget` runtime rejection, F-22 DNS rebinding, F-20 iptables RFC1918, F-26/27/28). Different code surface.
- Theme 1 workspace-seed admission tightening — already shipped in Phase 2h Theme 1.
- F-44 (run-Pod source-Secret bug pattern). Tracked under Theme 6.
- An e2e spec exercising the new admission rejections. The unit + envtest webhook layer is the primary verification surface; e2e end-to-end audit-trail confirmation is a follow-up.
- Operator-tunable broker port. Today hardcoded to 8443 in the F-29 validator.
- A first-party glob library for F-30. Stdlib `path.Match` covers the cited use cases.

## Open-question decisions (resolved at design time)

Four open questions were deferred from the roadmap memo. Decisions taken at design review on 2026-04-27:

1. **F-30 glob dialect.** Shell-style via `path.Match` (`*`, `?`, `[abc]` character classes). Stdlib, well-understood, gives users `claude-code-*`, `*-prod`, `*-?` for free; admission validates pattern syntax via `path.Match("pattern", "")` which returns `ErrBadPattern` for malformed input.
2. **F-23/F-34 deny-list scope.** Apply uniformly across all three host surfaces: `BrokerPolicy.spec.grants[].egress[].host`, `BrokerPolicy.spec.grants[].credentials[].provider.hosts` + `proxyInjected.hosts`, and `HarnessTemplate.spec.requires.egress[].host`. No host-surface carve-out for credential injection — there is no realistic legitimate use of `kubernetes.default.svc` as a credential-injection target.
3. **F-29 acceptable broker-endpoint shape.** Tight: scheme `https://`, host ending in `.svc` or `.svc.cluster.local`, port equal to `8443`. Matches the only deployment shape we ship. Operators on a custom port will need to revisit; cheap to relax later.
4. **F-08 RollingUpdate values.** `paddock-broker` uses `strategy: Recreate` (single replica, can't run two in parallel due to TLS + per-run audit/issue paths). `paddock-controller-manager` uses `strategy: RollingUpdate` with `maxSurge: 1` / `maxUnavailable: 0` so the leader-elected manager rolls without webhook downtime.

## Architecture

Files touched and the role each plays:

- `internal/policy/resolver.go` — `AppliesToTemplate` switches to `path.Match`; new sibling `ValidateAppliesToTemplate(selector string) error` returns the `ErrBadPattern` for admission to surface.
- `internal/webhook/v1alpha1/host_validation.go` — new file. Holds `validateExternalHost(p *field.Path, host string) field.ErrorList` consolidating cluster-internal, IP-literal, port-in-host, RFC 1123 hostname, and wildcard-position rules.
- `internal/webhook/v1alpha1/brokerpolicy_webhook.go` — `validateHosts` + `validateEgressGrants` delegate per-host checks to `validateExternalHost`. `validateBrokerPolicySpec` calls `policy.ValidateAppliesToTemplate` per selector. `validateProviderConfig` adds numeric-positive-integer + ≤20-digit checks on `GitHubApp.AppID` and `GitHubApp.InstallationID`.
- `internal/webhook/v1alpha1/harnesstemplate_shared.go` — `validateRequireSpec` delegates to `validateExternalHost` for `spec.requires.egress[].host`.
- `api/v1alpha1/brokerpolicy_types.go` — `+kubebuilder:validation:Pattern="^[1-9][0-9]{0,19}$"` annotations on `AppID` and `InstallationID`. Regenerated CRD via `make manifests`.
- `internal/brokerclient/brokerclient.go` — `New` validates the endpoint string (https scheme; host suffix `.svc` or `.svc.cluster.local`; port 8443). Returns a typed error so callers can fatal at startup.
- `internal/proxy/broker_client.go` — propagates the new validator's error path verbatim. No structural change.
- `cmd/proxy/main.go` — surfaces the new validator's error class with an explicit setup-log line (`"broker-endpoint validation failed: <reason>"`) before `os.Exit(1)`.
- `config/broker/deployment.yaml` — adds `spec.strategy: { type: Recreate }`.
- `config/manager/manager.yaml` — adds `spec.strategy: { type: RollingUpdate, rollingUpdate: { maxSurge: 1, maxUnavailable: 0 } }`.
- `charts/paddock/templates/paddock.yaml` — regenerated via `make helm-chart`.
- `docs/internal/security-audits/2026-04-25-v0.4-audit-findings.md` — Resolution + Status (recheck) update lines for the seven findings.
- `CHANGELOG.md` — owned by `release-please`; not edited manually. Conventional Commit messages provide the entries.

Pattern reuse from Phase 2:

- **`field.ErrorList` composition** (Phase 2e/2g). New helpers (`validateExternalHost`, the F-36 GitHub App ID checks) follow the same shape as the existing `validateDeliveryMode` / `validateProxyInjected` / `validateInterception`.
- **Status-condition + AuditEvent dual reporting** (Phase 2c). The existing `BrokerPolicyCustomValidator.audit` already emits `policy-applied` / `policy-rejected` AuditEvents on every admission outcome; new rejection codepaths inherit this for free with the aggregated `field.ErrorList.ToAggregate().Error()` string in `Reason`. Same applies to the existing HarnessTemplate validators.
- **`hostCoveredByAnyEgress` glob helper** — already uses suffix matching for `*.example.com`-shape patterns; `path.Match` for F-30 is structurally similar (and its lowercase normalization stays as defense-in-depth even after F-35 rejects mixed-case at admission).
- **Broker-as-single-enforcement-point** (Phase 2g). F-29's runtime re-validation of broker URL stays at the broker (no admission requirement on cluster operators); the proxy's startup-time check is parse-time hygiene, not policy.

## Components

### F-08 — Explicit Helm rollout strategy (chart)

- `config/broker/deployment.yaml` adds `spec.strategy: { type: Recreate }` directly under `spec`.
- `config/manager/manager.yaml` adds `spec.strategy: { type: RollingUpdate, rollingUpdate: { maxSurge: 1, maxUnavailable: 0 } }`.
- `make helm-chart` regenerates `charts/paddock/templates/paddock.yaml`. The kustomize-to-Helm pipeline (`hack/gen-helm-chart.sh`) doesn't need changes — `strategy:` flows through verbatim.
- `kube-linter`'s `no-rolling-update-strategy` rule, today the standing regression gate via the existing kube-linter CI step, flips green for both Deployments.

### F-23 (admission half) / F-34 — Cluster-internal and IP-literal host rejection

- New file `internal/webhook/v1alpha1/host_validation.go` holds:
  ```go
  func validateExternalHost(p *field.Path, raw string) field.ErrorList
  ```
  The function consolidates:
  - Empty / leading-or-trailing whitespace rejection (already in callers; pulled in here).
  - Wildcard position rules (already in callers; pulled in here).
  - Bare `*` rejection: `*` alone is the catch-all and is denied; the recommendation is to use a TLD-shape pattern.
  - Bare `*.` rejection: `host[2:] == ""` is the empty suffix; denied.
  - Cluster-internal rejection. Define a closed predicate `isClusterInternal(s string) bool` that returns true when `s` is exactly one of `localhost`, `kubernetes`, `kubernetes.default`, `kubernetes.default.svc`, `kubernetes.default.svc.cluster.local`, `svc`, `cluster.local`, `svc.cluster.local`, OR ends in `.svc`, `.svc.cluster.local`, `.cluster.local` (with a non-empty prefix before the dot). The validator denies the host when `isClusterInternal(host)` is true; for wildcard hosts it ALSO denies when `isClusterInternal(host[2:])` is true. This catches `*.svc.cluster.local`, `*.svc`, `*.cluster.local`, `*.com.svc`, etc.
  - IP-literal rejection: `net.ParseIP(host) != nil` denied. Catches `127.0.0.1`, `::1`, `192.0.2.1`, `169.254.169.254`. IPv6 in brackets (`[::1]`) gets the brackets stripped first. Wildcard hosts with an IP-literal suffix (`*.0.0.1` is technically not an IP) don't trip this; they fall through to the RFC 1123 / cluster-internal checks.
- `validateHosts` (`brokerpolicy_webhook.go`) and `validateEgressGrants` (`brokerpolicy_webhook.go`) replace their inline wildcard checks with a per-host call to `validateExternalHost`.
- `validateRequireSpec` (`harnesstemplate_shared.go`) makes the same delegation for `spec.requires.egress[].host`.
- The existing wildcard-position rules in callers move into `validateExternalHost`; the original inline checks delete to avoid duplication.

### F-29 — Broker endpoint shape validator at construction

- `internal/brokerclient/brokerclient.go::New` adds an early check after the `Endpoint == ""` guard:
  ```go
  if err := validateBrokerEndpoint(opts.Endpoint); err != nil {
      return nil, fmt.Errorf("brokerclient: %w", err)
  }
  ```
  with a sibling `func validateBrokerEndpoint(endpoint string) error` doing:
  - `url.Parse` of the trimmed endpoint.
  - Scheme MUST be `"https"`.
  - Hostname MUST end in `.svc` or `.svc.cluster.local` (lowercase, no trailing dot).
  - Port MUST be `"8443"`.
  - No path / no query / no fragment beyond the empty.
- `internal/proxy/broker_client.go::NewBrokerClient` already propagates `brokerclient.New`'s error; the change there is a one-line wrap to keep the error message readable.
- `cmd/proxy/main.go` updates its setup-log to surface the new error class explicitly (`setupLog.Error(err, "broker-endpoint validation failed")`) before `os.Exit(1)`. This matches the existing `cmd/proxy/main.go::140-156` shape.
- The controller's broker-client construction (`internal/controller/brokerclient_setup.go` or equivalent) inherits the same validator since both use `brokerclient.New`. Today's chart-shipped value (`https://paddock-broker.paddock-system.svc:8443`) passes; no operator-visible change there.

### F-30 — Shell-style globs in `appliesToTemplates`

- `internal/policy/resolver.go::AppliesToTemplate` switches its inner loop:
  ```go
  func AppliesToTemplate(selectors []string, templateName string) bool {
      for _, sel := range selectors {
          if sel == "*" {
              return true
          }
          ok, err := path.Match(sel, templateName)
          if err == nil && ok {
              return true
          }
      }
      return false
  }
  ```
  The `sel == "*"` short-circuit is preserved because `path.Match("*", "anything")` is `true` already, but the branch makes the catch-all visible.
- New sibling `func ValidateAppliesToTemplate(selector string) error` calls `path.Match(selector, "")` and returns `ErrBadPattern` to admission. Empty match against the empty string is defined as `false` for valid patterns; admission cares only that `err == nil`.
- `internal/webhook/v1alpha1/brokerpolicy_webhook.go::validateBrokerPolicySpec` calls `ValidateAppliesToTemplate` per selector after the existing empty/whitespace check, surfacing `field.Invalid` with the malformed pattern in the value field.
- The two existing callers in `internal/broker/matching.go:65,97` and the one in `internal/policy/intersect.go:89` work transparently — they already call `AppliesToTemplate` and gain glob support without modification.

### F-35 — Host-field normalization (lowercase, no port, RFC 1123)

- Folded into `validateExternalHost` (above) so the rules live in one place:
  - Reject any host that contains `":"` (port-in-host) with the message "ports go in the egress grant's `ports` field". The `[::1]` IPv6-bracketed case is caught by the IP-literal check before this rule fires.
  - Reject any host that doesn't match RFC 1123 DNS subdomain via `k8s.io/apimachinery/pkg/util/validation.IsDNS1123Subdomain`. This covers the lowercase requirement (RFC 1123 is lowercase-only), label-length limits, and label format. For wildcard hosts (`*.example.com`), strip the leading `*.` before the check so the predicate sees `example.com`.
- The existing lowercase normalization at match time (`hostCoveredByAnyEgress` in `brokerpolicy_webhook.go:506-508`) stays — cheap defense-in-depth that protects future fields not routed through `validateExternalHost`.

### F-36 — Numeric `appId` and `installationId` for GitHubApp

- `api/v1alpha1/brokerpolicy_types.go::ProviderConfig` adds:
  ```go
  // +kubebuilder:validation:Pattern=`^[1-9][0-9]{0,19}$`
  AppID string `json:"appId,omitempty"`
  // +kubebuilder:validation:Pattern=`^[1-9][0-9]{0,19}$`
  InstallationID string `json:"installationId,omitempty"`
  ```
  Twenty-digit cap exceeds GitHub's actual ID range comfortably.
- `make manifests` regenerates `charts/paddock/crds/brokerpolicies.paddock.dev.yaml`.
- `internal/webhook/v1alpha1/brokerpolicy_webhook.go::validateProviderConfig` adds an explicit webhook check for the same pattern in the GitHubApp branch — the CRD pattern is the first line of defense; the webhook gives the friendlier error message ("appId must be a positive integer (1-20 digits); see kubectl get installation -o yaml | grep id or the GitHub App settings page").

## Behaviour changes (operator-visible)

All are pre-1.0 hard breaks. CHANGELOG entries land via individual Conventional Commit `feat(security)!:` markers.

1. **F-30** — `appliesToTemplates: ["claude-code-*"]` previously matched no templates; now matches every template whose name starts with `claude-code-`. CHANGELOG note: "audit your `appliesToTemplates` selectors after upgrade — globs now match per the documented behaviour." Operators who widened to `["*"]` to compensate for the silent-no-match should narrow back to the intended pattern; otherwise they're now over-granting on both entries.
2. **F-34/F-23/F-35** — Existing `BrokerPolicy.spec.grants[].egress[].host`, `…credentials[].provider.hosts`, `…credentials[].provider.deliveryMode.proxyInjected.hosts`, and `HarnessTemplate.spec.requires.egress[].host` containing the deny-list patterns are rejected on next update. Existing objects in etcd remain (Kubernetes admission only validates on write); operators discover rejection on first edit. Mitigation: kubectl get -o yaml dumps the values, and `kubectl apply --dry-run=server` surfaces the rejection in CI without writing.
3. **F-36** — `GitHubApp.appId` or `installationId` containing non-numeric characters or longer than 20 digits is rejected on next update. Today these would fail at first JWT-signing call with a less helpful error.
4. **F-29** — Proxy `--broker-endpoint` values that don't match `https://*.svc[.cluster.local]:8443` cause the proxy to refuse to start. The chart-shipped value passes; any operator who manually overrode it to a different shape will need to align.
5. **F-08** — Helm upgrades trigger a different rollout shape: `paddock-broker` now has a brief Recreate window (Pod stops, Pod starts) instead of the prior implicit RollingUpdate; `paddock-controller-manager` still rolls without downtime but with explicit `maxSurge:1, maxUnavailable:0` (effectively the same as the implicit default for a single-replica leader-elected Deployment).

## Error-message shape

All admission errors land via `field.Invalid` / `field.Required` / `field.Forbidden` and surface in `kubectl apply` as a list. Concrete messages:

- F-30 (bad glob): `spec.appliesToTemplates[i]: Invalid value: "claude-code-[": selector is not a valid glob pattern (path.Match: syntax error in pattern)`
- F-34/F-23 (cluster-internal): `spec.grants.egress[i].host: Invalid value: "*.svc.cluster.local": cluster-internal hosts are not permitted in egress grants; use a public DNS name instead`
- F-34 (IP literal): `spec.grants.egress[i].host: Invalid value: "192.0.2.1": IP literals are not permitted; use a DNS name so the proxy can match the SNI`
- F-34 (bare wildcard): `spec.grants.egress[i].host: Invalid value: "*": catch-all host is not permitted; restrict to a specific TLD or apex (e.g. "*.example.com")`
- F-35 (port-in-host): `spec.grants.credentials[i].provider.hosts[j]: Invalid value: "api.example.com:443": host must not contain a port; ports go in the egress grant's "ports" field`
- F-35 (RFC 1123): `spec.grants.credentials[i].provider.hosts[j]: Invalid value: "Api.Example.com": host must be a lowercase RFC 1123 DNS name`
- F-36 (non-numeric appId): `spec.grants.credentials[i].provider.appId: Invalid value: "broken": appId must be a positive integer (1-20 digits); see kubectl get installation -o yaml | grep id or the GitHub App settings page`
- F-29 (proxy startup): `broker-endpoint validation failed: scheme must be https; got "http://..."` — emitted at `Error()` level before `os.Exit(1)` so it's visible in default zap config.

## Audit emission

`BrokerPolicyCustomValidator.audit` (Phase 2c) already emits `policy-applied` / `policy-rejected` AuditEvents on every admission outcome with the aggregated `field.ErrorList.ToAggregate().Error()` string in `Reason`. New rejection codepaths inherit this for free — no new wiring. Same applies to the existing HarnessTemplate validators.

The proxy startup-time F-29 rejection happens *before* the audit sink is constructed (`buildAuditSink` runs after the broker client) so it logs only — no AuditEvent. That's the right shape: it's a misconfiguration during Pod startup, not a tenant-policy decision, and the proxy never enters its hot path.

- `internal/webhook/v1alpha1/brokerpolicy_webhook_test.go` — extends with: `appliesToTemplates` malformed-glob rejection; F-36 GitHubApp `appId`/`installationId` non-numeric + over-length rejection; deny-list rejection cases for `egress[].host`, `provider.hosts`, `proxyInjected.hosts`.
- `internal/webhook/v1alpha1/harnesstemplate_webhook_test.go` (and `clusterharnesstemplate_webhook_test.go`) — extends with the deny-list cases on `spec.requires.egress[].host`.
- `internal/brokerclient/brokerclient_test.go` — `New` rejects non-https endpoint, non-cluster-internal host, wrong port, missing port. Accepts canonical `https://paddock-broker.paddock-system.svc:8443` and `https://paddock-broker.paddock-system.svc.cluster.local:8443`.

Envtest webhook coverage — the existing `webhook_suite_test.go` already loads `BrokerPolicyCustomValidator` and the HarnessTemplate validators; new rejection cases land as additional `It()` blocks under existing `Describe(...)` ordered contexts.

E2E — no new e2e spec required. The unit + envtest layers are the primary verification surface; an end-to-end audit-trail confirmation in `hostile_test.go` is overhead disproportionate to the finding's threat surface and is logged as a follow-up.

`kube-linter` regression gate — F-08 fix flips the existing `no-rolling-update-strategy` failures green; CI's existing kube-linter step is the standing gate.

## Risks and trade-offs

- **F-30 silent matching change** is the highest-impact behaviour change. An operator with `appliesToTemplates: ["claude-code-*"]` who's been silently ungoverned now suddenly *is* governed — net safer, but if they widened to `["*"]` to compensate, they're now over-granting on both entries. CHANGELOG callout is the mitigation; we accept the risk pre-1.0.
- **F-34 deny list breaking existing intra-cluster egress policies** — operators who deliberately allowed the agent to reach an in-cluster Service (e.g. an internal embeddings server at `embeddings.ml.svc.cluster.local`) hit admission rejection on next edit. They must use a public DNS name or accept that intra-cluster egress is exactly the attack surface F-23/F-34 close. Pre-1.0, accept the break.
- **F-29 startup-time validator regressing on a future custom port** — if a future operator legitimately runs the broker on a non-8443 port, the proxy refuses to start. Mitigation: the constant is one line of code; a future PR can swap `8443` for a small allowlist or accept a flag override. Today's chart hardcodes 8443.
- **F-36 numeric pattern rejecting GitHub App IDs near the int64 boundary** — at 20 digits we cap at 99,999,999,999,999,999,999 which exceeds GitHub's actual ID range comfortably. No real risk.
- **Validator placement in `brokerclient.New`** — pushing F-29 down into the shared brokerclient package means the controller's broker client gets the same validator. The controller's only call passes the same chart-shipped string, so it passes. Acceptable; better than divergence.
- **F-08 broker Recreate vs RollingUpdate** — the broker has a brief no-replica window during rollout. Combined with retries on the controller side and the broker's stateless-issue path, this is acceptable; in-flight TLS handshakes drop and the agent retries. Multi-replica broker (post-F-14 PATPool durability) would flip to RollingUpdate then.

## Commit decomposition

Each commit is a `feat(security)!:` or `fix(chart)!:` so `release-please` picks up the breaking-change marker. Order chosen so each commit's tests pass against the prior commit's state.

1. `chore(webhook): extract host_validation helper` — pure refactor moving the existing wildcard-position rules out of `brokerpolicy_webhook.go` and `harnesstemplate_shared.go` into `host_validation.go`. Non-breaking. Keeps the F-34 commit smaller.
2. `feat(webhook)!: F-34/F-23 — reject cluster-internal and IP-literal hosts in admission` — extends `validateExternalHost` with the deny list; new tests; the previous helper extraction makes this a clean delta.
3. `feat(webhook)!: F-35 — normalize host fields (lowercase, no port, RFC 1123)` — adds the normalization rules to `validateExternalHost`. Folds together with F-34 because they share the helper, but the commit is its own to surface in the changelog.
4. `feat(policy)!: F-30 — appliesToTemplates supports shell-style globs` — `path.Match`-based matcher in `internal/policy/resolver.go` + admission `path.Match("p", "")` syntax check + tests.
5. `feat(webhook)!: F-36 — GitHubApp appId/installationId must be positive integers` — kubebuilder annotation on the API types, regenerated CRD, webhook double-check, tests.
6. `feat(brokerclient)!: F-29 — validate broker endpoint shape at construction` — new validator in `brokerclient.New` + proxy startup error log update + tests.
7. `fix(chart)!: F-08 — explicit rollout strategy on broker (Recreate) and manager (RollingUpdate)` — `config/{broker,manager}/*.yaml` + regenerated `charts/paddock/templates/paddock.yaml`.
8. `docs(security): mark F-08, F-23, F-29, F-30, F-34, F-35, F-36 Resolved (Phase 2h Theme 3)` — Resolution + recheck-status updates in `docs/internal/security-audits/2026-04-25-v0.4-audit-findings.md`.

Eight commits, one PR. CI runs the existing pre-commit hook (`go vet -tags=e2e ./...` + `golangci-lint`) on each.
