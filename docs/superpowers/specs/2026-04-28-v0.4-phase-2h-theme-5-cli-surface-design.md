- Owner: @tjorri

- Tracks: GitHub issue #46.
- Predecessors: Phase 2e (reserved-env allowlist pattern reused for F-55), Phase 2h Themes 1/2/3/6 (commit-shape + audit-status-block pattern).
- Related: `docs/superpowers/plans/2026-04-26-v0.4-security-recheck-roadmap.md` (Theme 5 section); `docs/internal/security-audits/2026-04-25-v0.4-audit-findings.md` (F-53, F-54, F-55 detail blocks).

## Context

Theme 5 of the v0.4 security follow-up roadmap closes three Open P3 UX-hardening findings from the 2026-04-26 recheck. All three live in `internal/cli/` and never touch any server-side component (broker, proxy, controller, webhooks). A single small PR addresses all three with a contained diff.

The findings:

- **F-53** — `--prompt-file` / stdin reads via `io.ReadAll` with no client-side size cap; a multi-GiB file silently OOMs the CLI process before the server-side 256 KiB admission cap (`MaxInlinePromptBytes` in `internal/webhook/v1alpha1/harnessrun_webhook.go:210`) triggers.
- **F-54** — `logs --file <path>` passes the user-supplied path verbatim into `cat -- <path>` inside the reader Pod with no normalisation or workspace-prefix enforcement.
- **F-55** — `logs --reader-image` accepts an arbitrary image string with no registry allowlist or digest requirement; the default `busybox:1.37` is tag-pinned, not digest-pinned.

Pre-1.0, breaking changes are accepted in place rather than flag-aliased (CLAUDE.md). F-54 and F-55 are user-visible behaviour changes for callers that today rely on the unconstrained surface (paths outside `/workspace`, arbitrary image references). All three behaviour-changing commits carry the `feat(security)!:` Conventional Commit marker matching the Theme-1/2/3/6 cadence — see Rollout for the full commit shape.

## Goals

- Close F-53, F-54, F-55 in a single PR, commit-decomposed per finding so each behaviour-changing commit lands its own breaking-change marker for `release-please`.
- Reuse Phase 2 patterns wherever possible — promote `MaxInlinePromptBytes` to a shared API location so both the webhook and the CLI reference one constant rather than duplicating the literal; the F-55 allowlist mirrors the "explicit allowlist + reject everything else at entry" shape from Phase 2e's `reservedExtraEnvLiterals`.
- Leave the existing `--raw`/`--result`/default selector behaviour unchanged. Only `--file` and `--reader-image` (and `--prompt-file`) gain validation.

## Non-goals

- **Symlink resolution inside the reader pod.** A pod-side `realpath`-and-recheck before `cat` would close the residual where a tenant drops `/workspace/escape` → `/etc/passwd`. Out of scope: the same user can mount the PVC from any other Pod they have `pods.create` for, so the trust boundary doesn't shift; deferred as defence-in-depth.
- **Server-side enforcement of reader-image policy.** F-55 is CLI-side only. An admission webhook on tenant Pod creation would give the policy server-side teeth, but that's a separate finding and would expand the diff well beyond Theme 5's "small bundled PR" intent.
- **Per-org config file for the CLI.** The override flag (`--allow-reader-image-prefix`) is per-invocation. A future `~/.config/paddock/cli.yaml` is a separate enhancement; deciding now would gate Theme 5 on a file-format design.
- **An e2e spec.** All three findings are validated entirely client-side before any cluster contact. Unit tests in `internal/cli/{submit,logs}_test.go` give full coverage.

## Open-question decisions (resolved at design time)

Three open questions were deferred from the roadmap memo. Decisions taken at design review on 2026-04-28:

1. **F-53 cap value.** Match the server cap exactly (`MaxInlinePromptBytes = 256 * 1024`). Diverging would either silently truncate server-accepted payloads (cap < server) or let through payloads the server will reject (cap > server). Tunability deferred until a per-org config-file mechanism exists.
2. **F-54 path-prefix shape.** Hard reject anything that doesn't `path.Clean` to under `/workspace/`. The existing `--raw`/`--result`/default selectors cover the friendly cases; `--file` is the explicit "I know what I'm doing" knob, so strict-and-explicit matches intent. No relative-path auto-prepend.
3. **F-55 allowlist shape.** Compile-time prefix list + per-invocation `--allow-reader-image-prefix` flag override (option A from the brainstorm), combined with a digest-pinned default `busybox` reference. A future config-file override is out of scope.

## Architecture

Files touched and the role each plays:

- `api/v1alpha1/harnessrun_types.go` — owns `const MaxInlinePromptBytes = 256 * 1024` after the move. Single source of truth for the cap.
- `internal/webhook/v1alpha1/harnessrun_webhook.go` — its local `const MaxInlinePromptBytes` is deleted; the two existing call sites at `harnessrun_webhook.go:250–251` rewrite as `paddockv1alpha1.MaxInlinePromptBytes` (the package alias is already imported at file scope).
- `internal/cli/submit.go` — `readPromptFile` factored into a thin file/stdin selector plus an unexported `readPromptCapped(r io.Reader) (string, error)` helper that wraps the read in `io.LimitReader(r, MaxInlinePromptBytes+1)`; the helper takes an `io.Reader` so unit tests can drive the cap-detection branch without spawning a child process for stdin.
- `internal/cli/util.go` — `readAll` has only one consumer (`readPromptFile`) today; it is removed once that consumer migrates to `readPromptCapped`. Verified at implementation time with a `grep`.
- `internal/cli/logs.go` — four changes:
  - The bare `readerImage: "busybox:1.37"` default in `newLogsCmd` (`logs.go:50`) is replaced with a new package-level `defaultReaderImage = "busybox:1.37@sha256:<digest>"` constant.
  - New `defaultReaderImageAllowlist` package-private slice.
  - `logsOptions.allowedImagePrefixes []string` bound to a new `--allow-reader-image-prefix` `StringSliceVar` flag.
  - New `validateReaderImage(image string, extra []string) error` — boundary-aware prefix match; called from `runLogs` before `newReaderPod`.
  - `resolvedPath` signature changes to `(*paddockv1alpha1.HarnessRun) (string, error)`; `path.Clean` + `HasPrefix("/workspace/")` guard, with a friendly error naming the allowed prefix.
  - `--file` and `--reader-image` flag help strings updated to surface the new constraints.
- `Makefile` — new `update-reader-image-digest` target.
- `hack/update-reader-image-digest.sh` — new helper invoking `crane digest` (with `docker buildx imagetools inspect` fallback) to refresh the digest constant in `internal/cli/logs.go` via `sed`.
- `internal/cli/submit_test.go` — adds size-cap unit tests (file-path and stdin paths).
- `internal/cli/logs_test.go` — adds `resolvedPath` prefix-enforcement and `validateReaderImage` allowlist tests.
- `docs/internal/security-audits/2026-04-25-v0.4-audit-findings.md` — appends `Status (recheck 2026-04-28). State: Closed.` blocks to F-53, F-54, F-55.

## Detailed designs

### F-53 — `--prompt-file` / stdin size cap

**Constant promotion.** `MaxInlinePromptBytes = 256 * 1024` lives in the webhook package today. Step 1 of the work moves the declaration to `api/v1alpha1` (the package the CLI already imports for `HarnessRun` constructions). The webhook's existing call site updates to import from the API package; no behaviour change. This avoids two divergent copies of the constant for all time.

**Read shape.** `readPromptFile` collapses to a thin selector plus a testable cap-applying helper:

```go
func readPromptFile(path string) (string, error) {
    if path == "-" {
        return readPromptCapped(os.Stdin)
    }
    f, err := os.Open(path)
    if err != nil {
        return "", fmt.Errorf("opening --prompt-file %s: %w", path, err)
    }
    defer f.Close()
    return readPromptCapped(f)
}

// readPromptCapped reads at most MaxInlinePromptBytes from r and rejects
// inputs that exceed the cap. Takes an io.Reader (rather than reading
// os.Stdin directly) so unit tests can drive the cap-detection branch.
func readPromptCapped(r io.Reader) (string, error) {
    data, err := io.ReadAll(io.LimitReader(r, paddockv1alpha1.MaxInlinePromptBytes+1))
    if err != nil {
        return "", fmt.Errorf("reading prompt: %w", err)
    }
    if len(data) > paddockv1alpha1.MaxInlinePromptBytes {
        return "", fmt.Errorf(
            "prompt exceeds %d-byte limit (set by the admission webhook); "+
                "use a smaller --prompt-file or split the prompt",
            paddockv1alpha1.MaxInlinePromptBytes)
    }
    return string(data), nil
}
```

The `+1` is the standard cap-detection trick: if the read produces exactly `cap+1` bytes, the source had at least `cap+1` and we error. Anything `<= cap` reads fully.

**Behaviour.** `<= 256 KiB` payloads succeed unchanged. `> 256 KiB` payloads error client-side with the byte cap named explicitly, so the user can see the limit and tune.

**Util cleanup.** `internal/cli/util.go::readAll` is currently only consumed from `readPromptFile`. After this change it has no consumers and is removed. Verified at implementation time with a `grep` pass.

### F-54 — `logs --file` path-prefix enforcement

**Validator.**

```go
const workspaceMount = "/workspace"

func (o *logsOptions) resolvedPath(run *paddockv1alpha1.HarnessRun) (string, error) {
    if o.file != "" {
        clean := path.Clean(o.file)
        if clean != workspaceMount && !strings.HasPrefix(clean, workspaceMount+"/") {
            return "", fmt.Errorf(
                "--file must be an absolute path rooted under %s/ (got %q after path.Clean)",
                workspaceMount, clean)
        }
        return clean, nil
    }
    base := fmt.Sprintf("/workspace/.paddock/runs/%s", run.Name)
    switch {
    case o.raw:
        return base + "/raw.jsonl", nil
    case o.result:
        return base + "/result.json", nil
    default:
        return base + "/events.jsonl", nil
    }
}
```

`path.Clean` (not `filepath.Clean`) is correct: the path is interpreted by `cat` *inside the Linux reader pod*, and Windows-host CLI invocations must still produce `/`-separated paths and POSIX `..` semantics.

`path.Clean("/workspace/../etc/passwd")` returns `/etc/passwd`, which fails the prefix check. Relative paths (`raw.jsonl`, `../foo`) also fail because they don't start with `/workspace`. The cleaned path (not the raw input) is what flows into the pod's `cat`, normalising e.g. `/workspace//foo/./bar` to `/workspace/foo/bar`.

**Caller change.** `runLogs` now propagates the `error`:

```go
filePath, err := opts.resolvedPath(run)
if err != nil {
    return err
}
```

The error returns *before* the reader pod is created, so a rejected path never produces a Pod object. Existing shell-injection safety (separate argv element + `--`) is preserved on the cleaned path.

**Help-text update.** `--file` flag help string changes from:

> Arbitrary absolute path inside the workspace (overrides the selectors above)

to:

> Absolute path rooted under /workspace (overrides the selectors above; relative paths and paths outside the workspace are rejected)

### F-55 — `--reader-image` allowlist + digest-pinned default

**Default image.**

```go
// Refreshed by `make update-reader-image-digest`. Format is
// <ref>:<tag>@sha256:<digest> — Kubernetes accepts both tag and
// digest in a single image reference and pulls by digest.
const defaultReaderImage = "busybox:1.37@sha256:<64-char-hex>"
```

The `<64-char-hex>` placeholder is filled at implementation time with the current digest from `crane digest busybox:1.37` (or the docker-buildx fallback). Tag and digest both stay in the string so `--help` output and pod logs stay readable; k8s pulls by digest when both are present, so the tag is purely cosmetic for the user.

The bare `readerImage: "busybox:1.37"` initialisation in `newLogsCmd` (`logs.go:50`) is replaced with `readerImage: defaultReaderImage` so the digest-pinned value is the user-facing default.

**Allowlist.**

```go
// defaultReaderImageAllowlist holds trusted registry/repo prefixes
// for --reader-image. Operators running air-gapped clusters with
// private registries can extend the allowlist at invocation time
// via --allow-reader-image-prefix.
var defaultReaderImageAllowlist = []string{
    "busybox",            // docker shorthand: busybox, busybox:tag, busybox:tag@sha256:...
    "docker.io/library/", // qualified Docker Hub library namespace
    "registry.k8s.io/",   // SIG-blessed Kubernetes registry
}
```

**Override flag.**

```go
cmd.Flags().StringSliceVar(&opts.allowedImagePrefixes, "allow-reader-image-prefix", nil,
    "Additional trusted registry/repo prefixes for --reader-image (repeatable, comma-separated)")
```

**Validation.**

```go
func validateReaderImage(image string, extra []string) error {
    if image == "" {
        return fmt.Errorf("--reader-image must not be empty")
    }
    allowed := append([]string(nil), defaultReaderImageAllowlist...)
    allowed = append(allowed, extra...)
    for _, p := range allowed {
        if image == p ||
            strings.HasPrefix(image, p+":") ||
            strings.HasPrefix(image, p+"@") ||
            strings.HasPrefix(image, p+"/") {
            return nil
        }
    }
    return fmt.Errorf(
        "--reader-image %q is not on the allowlist; trusted prefixes: %s. "+
            "Use --allow-reader-image-prefix=<prefix> to extend.",
        image, strings.Join(allowed, ", "))
}
```

The match is prefix-aware-with-boundary: `busybox` matches `busybox:1.37@sha256:...` (`:` boundary) and `busybox/foo` (`/` boundary) but **not** `busybox-evil:tag` (no boundary char). The boundary characters `:`, `@`, `/` cover all valid OCI image-ref shapes.

**Caller change.** `runLogs` calls `validateReaderImage(opts.readerImage, opts.allowedImagePrefixes)` before `newReaderPod`. Rejection happens client-side before any pod is created. Existing happy-path behaviour for the default image is preserved (the default satisfies the `busybox` prefix).

**Makefile target.**

```makefile
.PHONY: update-reader-image-digest
update-reader-image-digest:
	@./hack/update-reader-image-digest.sh
```

`hack/update-reader-image-digest.sh` resolves the current digest of `busybox:1.37` (preferring `crane`, falling back to `docker buildx imagetools inspect`) and patches the `defaultReaderImage` constant in `internal/cli/logs.go` via `sed`. Header comment in the script names the cadence: re-run when busybox publishes a new build under tag `1.37` (rare; CI does not gate on this).

## Testing

All three changes are pure CLI-side validations that happen before any cluster contact. Unit tests in `internal/cli/{submit,logs}_test.go` give full coverage. No e2e changes.

**F-53 — `submit_test.go`:**

- `readPromptCapped(bytes.NewReader)` at exactly `MaxInlinePromptBytes` → success, content matches.
- `readPromptCapped(bytes.NewReader)` at `MaxInlinePromptBytes + 1` → error string contains the byte cap.
- `readPromptCapped` driven from a hostile reader that returns more bytes than the cap → error (verifies `LimitReader` boundary).
- `readPromptFile` over a tmp file at exactly the cap → success (covers the file-open path end-to-end).
- `readPromptFile` over a tmp file at `cap + 1` → error.
- Existing happy-path `buildRunFromOptions` tests stay unchanged.

The stdin path (`readPromptFile("-")`) is exercised indirectly by the `readPromptCapped(io.Reader)` cases — driving `os.Stdin` from a unit test would require a child-process harness for negligible additional coverage. The cap logic lives in the helper, not the dispatcher.

**F-54 — `logs_test.go`:**

- `resolvedPath` with `--file=/workspace/foo.log` → returns `/workspace/foo.log`.
- `resolvedPath` with `--file=/workspace/../etc/passwd` → error.
- `resolvedPath` with `--file=/etc/passwd` → error.
- `resolvedPath` with `--file=raw.jsonl` (relative) → error.
- `resolvedPath` with `--file=/workspace//foo/./bar` → returns cleaned `/workspace/foo/bar`.
- `resolvedPath` with `--file=""` falls through to the existing selectors (unchanged).

**F-55 — `logs_test.go`:**

- `validateReaderImage("busybox:1.37@sha256:...")` → ok.
- `validateReaderImage("busybox")` → ok.
- `validateReaderImage("docker.io/library/busybox:1.37")` → ok.
- `validateReaderImage("registry.k8s.io/pause:3.10")` → ok.
- `validateReaderImage("evil.example.com/img:latest")` → error.
- `validateReaderImage("busybox-evil:tag")` → error (boundary check).
- `validateReaderImage("ghcr.io/myorg/foo:v1", extra=["ghcr.io/myorg/"])` → ok.
- `validateReaderImage("ghcr.io/other/foo:v1", extra=["ghcr.io/myorg/"])` → error.
- `validateReaderImage("")` → error.

## Rollout

Single PR. Per-finding commits matching the Theme-1/2/3/6 pattern (scope `security` for the v0.4-recheck-tracked changes; `refactor`/`docs` scopes for the supporting commits):

1. `refactor(api): hoist MaxInlinePromptBytes to api/v1alpha1` — non-breaking constant relocation; webhook updates its call sites.
2. `feat(security)!: cap CLI prompt input at MaxInlinePromptBytes (F-53)` — wraps reads with `io.LimitReader`; surfaces the limit client-side.
3. `feat(security)!: enforce /workspace prefix on logs --file (F-54)` — `path.Clean` + `HasPrefix` guard; `resolvedPath` now returns an error.
4. `feat(security)!: allowlist --reader-image and digest-pin default (F-55)` — validation, override flag, digest constant, Makefile target.
5. `docs(security): close F-53/F-54/F-55 status blocks` — audit findings + roadmap status update.

Commits 2/3/4 each carry the `!` breaking-change marker so `release-please` picks per-finding markers. F-53 is technically non-breaking by semver (anything that previously succeeded continues to succeed; oversized payloads change which side surfaces the error and what the message says), but the marker is kept for consistency with the Theme-1/2/3/6 cadence — every user-visible change in a recheck-tracked PR carries `!`. Commit 1 is a pure non-breaking refactor and intentionally separate to keep its diff easy to review.

PR squash subject (matching the established theme pattern): `feat(security)!: v0.4 Phase 2h — Theme 5 — CLI surface (closes #46)`.

## Risks and rollback

- **F-53.** A user pipes a file that is exactly at the cap and previously round-tripped successfully — still works (cap is `MaxInlinePromptBytes`, server cap is `MaxInlinePromptBytes`, equal). A user pipes a file that the server already rejects — they now see a friendlier error sooner. No workflow that previously worked will newly fail.
- **F-54.** Any caller today passing `--file` outside `/workspace` newly fails. Rollback is a one-line revert of the prefix check in `resolvedPath`. The CHANGELOG entry names the change.
- **F-55.** Any CI script today passing `--reader-image=<non-allowlisted>` newly fails. Mitigation: ship the override flag in the same commit so callers can adopt `--allow-reader-image-prefix=...` without code change. Rollback is a one-line revert of the `validateReaderImage` call in `runLogs`.
