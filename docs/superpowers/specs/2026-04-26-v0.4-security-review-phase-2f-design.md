# v0.4 Security Review — Phase 2f (Per-run intermediate CA) Design

- Owner: @tjorri
- Date: 2026-04-26
- Status: Draft
- Predecessor: Phase 2e (Pod-spec hardening — F-37 + F-39 + F-31, PR #31).
- Successor (planned): one of the remaining themes (Theme C runtime egress, Theme D broker robustness, Theme F workspace seed) — chosen via a separate brainstorm.

## 1. Summary

Phase 2f closes audit finding F-18 (P1) — the MITM CA private key is currently shared across every run and tenant, materially weakening the threat model's "per-run MITM CA (not shared)" defence claim. The fix is to issue a real per-run intermediate CA (and per-Workspace intermediate for the seed Pod path) using cert-manager's `kind: CA` `ClusterIssuer` against the existing cluster-wide root.

After the change:

- The cluster root private key is read only by cert-manager. The paddock-controller never reads it; tenant namespaces never receive a copy.
- Each run gets a unique intermediate (private key + cert) issued by cert-manager directly into the run's namespace.
- The agent's CA-trust env vars point at the **per-run intermediate cert** as the trust anchor — not the cluster root. Compromise of any tenant's intermediate key cannot mint leaves trusted by other tenants' agents.
- The seed Pod path (Workspaces) gets the analogous treatment with per-Workspace intermediates.

The phase touches `config/proxy/`, `internal/controller/proxy_tls.go`, `internal/controller/workspace_broker.go`, `internal/controller/pod_spec.go` (one subPath constant change), `cmd/main.go` (flag deprecation), plus go.mod/go.sum (new cert-manager API import), plus ADR-0013, threat-model.md, spec 0002 §16, and the audit findings doc.

## 2. Goals and non-goals

**Goals:**

1. The cluster root MITM CA private key never leaves cert-manager's signing path. Specifically, the paddock-controller does not read it; tenant namespaces never contain a copy.
2. Each `HarnessRun` gets its own intermediate CA (`isCA: true`) issued by cert-manager via a `ClusterIssuer` of `kind: CA` against the existing cluster root.
3. Each `Workspace` gets its own intermediate CA on the same model (per-Workspace, not per-seed-Job — see §4 for the lifetime decision).
4. The agent's trust anchor is the per-run/per-Workspace intermediate cert only. The cluster root cert is **not** mounted into the agent. Tenant A's agent does not trust leaves signed by tenant B's intermediate.
5. The threat-model claim "per-run MITM CA (not shared across runs)" becomes factually accurate. ADR-0013 §7.3 wording is updated to match.
6. F-18 has an explicit Resolution line in the audit findings doc.
7. Anti-regression test (`TestEnsureProxyTLS_PerRunIsolation`) asserts that two HarnessRuns in different namespaces produce distinct `Certificate` resources and distinct intermediate keys.

**Non-goals (deferred):**

- Cert revocation infrastructure. cert-manager handles renewal; revocation of a leaked intermediate is "rotate the root + re-issue all intermediates" via standard cert-manager flows. Acceptable pre-v1.
- Name constraints on intermediates (option C from brainstorm Q3 — `x509.NameConstraints` to limit forgery scope per-run). Pure trust-anchor isolation is the F-18 fix; name constraints are a future hardening.
- Hot-reload of the proxy sidecar's in-memory CA when cert-manager rotates the per-Workspace intermediate mid-Job. Operationally rare (1y duration / 30d renewBefore vs. seed Jobs that complete in minutes); documented as a known limitation.
- Per-seed-Job intermediates for Workspaces (option B from Q4). Per-Workspace already delivers the F-18 property; per-seed-Job is incremental hardening.
- Migration of in-flight pre-2f runs. Pre-1.0 evolve in place — old runs continue using their existing mounted Secrets; new runs get the new shape; no data migration.
- Backwards-compatibility shims on the controller flags. The `--proxy-ca-secret-name` / `--proxy-ca-secret-namespace` flags are removed (not aliased) in favor of `--proxy-ca-cluster-issuer-name`.
- Tightening the paddock-controller's `secrets get/list` RBAC on `paddock-system` (the controller no longer needs to read the source Secret after this phase). Tracked as a future cleanup.

## 3. Architecture

**Components added:**

1. **`ClusterIssuer` of `kind: CA`** — new manifest at `config/proxy/cluster-issuer.yaml`. Names the existing `paddock-proxy-ca` Secret as the signing root. Cluster-scoped because it has to be referenceable from per-tenant `Certificate` resources in arbitrary namespaces.

2. **Per-run cert-manager `Certificate`** — created by the controller in the run's namespace via `controllerutil.CreateOrUpdate`. `isCA: true`, `issuerRef: { kind: ClusterIssuer, name: paddock-proxy-ca-issuer }`. cert-manager produces the per-run Secret (`<run>-proxy-tls`) directly in the tenant namespace; the controller never reads or writes the intermediate's private key.

3. **Per-Workspace cert-manager `Certificate`** — symmetric to the per-run case but lives for the Workspace's lifetime (long-duration intermediate, cert-manager auto-renews via `renewBefore`).

**Components removed/changed:**

- `ensureProxyTLS` (`internal/controller/proxy_tls.go`): no longer reads the source Secret and copies bytes. Becomes "ensure a cert-manager `Certificate` exists for this run; check its readiness; flip `EgressConfigured` accordingly." The function shrinks substantially.
- `ensureWorkspaceProxyTLS` (`internal/controller/workspace_broker.go`): same transformation, scoped per-Workspace.
- A new shared helper `ensureProxyCACertificate(ctx, owner, name, ns, duration, renewBefore)` factors the common cert-manager Certificate-create logic between the run and Workspace paths. Both call sites become thin wrappers.
- `--proxy-ca-secret-name` / `--proxy-ca-secret-namespace` controller flags: removed in favor of `--proxy-ca-cluster-issuer-name` (default `paddock-proxy-ca-issuer`). Pre-v1 evolve in place — old flags removed, not aliased.
- `ProxyCASource` struct: replaced by `ProxyCAClusterIssuer string` on the reconciler (the controller no longer needs source-Secret coordinates because it doesn't read the Secret).
- `proxy_tls.go::caBundleKey = "ca.crt"` constant: deleted. The agent no longer reads a controller-authored `ca.crt` key — it reads `tls.crt` directly (the per-run intermediate cert).

**Operational requirements documented for the operator:**

- cert-manager must run with `--cluster-resource-namespace=paddock-system` so `ClusterIssuer kind: CA` reads `paddock-proxy-ca` from the right namespace. **This is a behavior change for every other ClusterIssuer in the cluster** — they all start looking in `paddock-system` for backing Secrets. Documented as an install requirement; flagged loudly in upgrade notes.
- Cluster operators must NOT broadly grant `certificates.cert-manager.io: create` to tenant ServiceAccounts. The paddock-controller's RBAC includes this; the ClusterIssuer is meant only for the controller's use. Sample Kyverno policy shipped (`config/samples/kyverno-cluster-issuer-restriction.yaml`) restricting Certificate-create on this specific ClusterIssuer to the paddock-controller SA.
- The `paddock-proxy-ca` source Secret in `paddock-system` becomes effectively read-only-by-cert-manager. The paddock-controller's `secrets get/list` RBAC on `paddock-system` can be tightened in a follow-up phase (out of scope here).

## 4. Per-run flow (HarnessRun)

**Trigger:** `HarnessRunReconciler.Reconcile` reaches the proxy/CA wiring step — currently `ensureProxyTLS` at `harnessrun_controller.go:421`.

**New flow:**

1. **Ensure `Certificate` resource exists.** Controller creates `<run>-proxy-tls` `cert-manager.io/v1 Certificate` in the run's namespace with:
   - `spec.isCA: true`
   - `spec.commonName: paddock-proxy-<run-name>`
   - `spec.secretName: <run>-proxy-tls` (matches existing Secret name; volume-mount paths in `pod_spec.go` unchanged for the proxy)
   - `spec.duration: 48h` — safe upper bound on run wall-clock time including grace period. F-42 (separate finding) caps `terminationGracePeriodSeconds` at 24h once resolved; 48h is the bound-plus-margin choice that does not require renewal during a run.
   - `spec.renewBefore`: omitted. No renewal — runs are bounded by `ActiveDeadlineSeconds` + grace; the intermediate outlives the run.
   - `spec.privateKey: { algorithm: ECDSA, size: 256 }` — matches the cluster root.
   - `spec.usages: [digital signature, key encipherment, cert sign]`
   - `spec.issuerRef: { kind: ClusterIssuer, name: <controller-flag-value> }`
   - `metadata.ownerReferences: [HarnessRun]` — cascades on run deletion → cert-manager garbage-collects the backing Secret too.

2. **Check `Certificate.status.conditions[Ready]==True`.** If not Ready:
   - `setCondition(EgressConfigured, False, Reason=ProxyCAPending, Message=<cert-manager-error or "waiting on cert-manager to issue per-run intermediate">)`
   - `Phase=Pending`, `RequeueAfter: 10s` (matches current cadence for `ProxyCAPending`).
   - The cert-manager-managed Secret exists once cert-manager creates it (post-Ready); attempting to mount it before then would crash-loop the run pod, which the Ready check prevents by gating Pod creation on `EgressConfigured=True`.

3. **When Ready,** the cert-manager-managed `<run>-proxy-tls` Secret has `tls.crt` (intermediate cert), `tls.key` (intermediate private key), and `ca.crt` (the cluster root cert that signed the intermediate — populated by cert-manager by default). **The agent's CA-trust env vars must point at the intermediate cert, not the root.** The change in `pod_spec.go` is one constant: `agentCABundleSubPath` flips from `"ca.crt"` to `"tls.crt"`. The mount remains a `subPath` projection of a single file — only the source key changes. An explanatory comment is added at the constant's declaration noting that `tls.crt` here is the per-run intermediate (which IS the agent's trust anchor — not the conventional "ca.crt = chain root" file convention).

4. **Audit emission.** The existing `EmitCAProjected` audit event continues to fire once on the first reconcile pass when the `Certificate` is freshly created (`controllerutil.OperationResultCreated`). The audit-kind enum (`AuditKindCAProjected`) is unchanged — same operator-visible audit semantics, just a different K8s resource (`Certificate` instead of the old byte-copied `Secret`). The same emitter is also still used by the broker-ca Secret-copy path (`ensureBrokerCA`), which Phase 2f does not change. Originally the design proposed renaming to `EmitCAIssued`, but the kept-name approach was chosen during implementation to avoid churn on the audit-kind enum and to keep the broker-ca emitter consistent.

5. **Removed code.** `ensureProxyTLS`'s source-Secret read, byte-copy, and the `proxyTLSSecretKeyCACert` / `proxyTLSSecretKeyCAKey` / `caBundleKey` constants (last one deleted; first two repurposed since cert-manager uses the same key names in its managed Secret). The function shrinks to "create-or-get cert-manager Certificate, check Ready, return ok/err."

## 5. Per-Workspace flow + Pod-spec / proxy-side adjustments

**Per-Workspace flow** (mirrors per-run with two differences):

- `Certificate` resource lives in the workspace's namespace, owned by the Workspace CR via OwnerReference; `secretName: <workspace>-proxy-tls`.
- `spec.duration: 8760h` (1y, matches the cluster root cert), `spec.renewBefore: 720h` (30d). cert-manager auto-renews. The seed Pod's mount is via projected Secret — when cert-manager rotates the Secret content, kubelet's projection updates within ~minutes. Long-lived Workspaces with infrequent re-seeding tolerate this without disruption.
- `commonName: paddock-proxy-ws-<workspace-name>`
- Otherwise identical (isCA, ECDSA P-256, ClusterIssuer ref, key-only-trust-anchor projection to seed Pod's agent-equivalent).
- `ensureWorkspaceProxyTLS` shrinks the same way as `ensureProxyTLS`. The shared helper `ensureProxyCACertificate(ctx, owner, name, ns, duration, renewBefore)` is the natural extraction point — both call sites become thin wrappers that supply the lifetime parameters.

**Pod-spec change** (`internal/controller/pod_spec.go`):

The agent currently mounts `ca.crt` from the per-run Secret via:
```go
{Name: proxyCAVolumeName, MountPath: agentCABundleMountPath, SubPath: agentCABundleSubPath, ReadOnly: true}
```
where `agentCABundleSubPath = "ca.crt"`. The new value is:

```go
// agentCABundleSubPath is the per-run intermediate cert — the agent's
// TLS trust anchor. Despite the conventional "tls.crt = serving cert"
// naming in cert-manager-issued Secrets, here that file IS the trust
// root from the agent's POV (we issue an intermediate isCA Certificate;
// the proxy serves [leaf, intermediate] chains; the agent's ca-bundle
// = the intermediate). The cluster root cert is deliberately NOT
// mounted into the agent. F-18 / Phase 2f.
agentCABundleSubPath = "tls.crt"
```

The proxy sidecar continues to mount `tls.crt` + `tls.key` as the signing keypair — those are now the intermediate's keypair, but the file names and mount path are unchanged. The proxy code in `internal/proxy/ca.go` works unchanged (the `forge` function builds `[der, ca.caCert.Raw]` — `ca.caCert.Raw` is now the intermediate's bytes, exactly what the agent needs in the chain to validate against its intermediate trust anchor).

## 6. Failure modes & edge cases

1. **Cert-manager unavailable / Certificate stuck.** Per-run `Certificate` exists but never reaches `Ready`. Controller sees `Ready=False`, sets `EgressConfigured=False, Reason=ProxyCAPending`, requeues 10s. After enough delay, the run's `ActiveDeadlineSeconds` (Job-level) terminates it. No new fail-closed path needed — matches today's behavior when cert-manager hasn't populated the source Secret.

2. **`ClusterIssuer` not Ready** (root Secret missing, cert-manager misconfigured, etc.). Per-run `Certificate` reports `IssuingFailed`. Same `ProxyCAPending` condition; the cert-manager error message is threaded through (read from `Certificate.status.conditions[Issuing]` or `[Ready]` Message field). Operator sees clear diagnostic in `kubectl describe harnessrun`.

3. **Run deleted while Certificate pending.** OwnerReference cascades — cert-manager's Certificate is deleted, the not-yet-issued Secret never appears. No leaked resources.

4. **cert-manager rotates Workspace intermediate while seed Job is running.** Rare (1y duration / 30d renewBefore vs. seed Jobs that complete in minutes). When kubelet refreshes the projected Secret mid-Job, the proxy sidecar continues running with its in-memory `caCert` from startup; new connections fail to forge correctly (signed by old intermediate that is no longer in the projected Secret). **Mitigation:** the seed Job's `terminationGracePeriodSeconds` is short (default 60s); operationally rare; not worth a hot-reload watcher in scope. Documented as a known limitation. Per-run intermediates have no renewal so this doesn't apply to runs.

5. **Operator forgets to set `--cluster-resource-namespace=paddock-system` on cert-manager.** `ClusterIssuer` reports `Ready=False, Reason=NotFound` (root Secret not found). Per-run Certificates fail at issuance. Surfaced via the `ProxyCAPending` chain (clear error). **Install docs** call this out prominently with the exact cert-manager flag value.

6. **Tenant attempts to create their own `Certificate` referencing our ClusterIssuer.** Their tenant SA does not have `certificates.cert-manager.io: create` RBAC by default. **Install docs** call this out as an RBAC discipline requirement; sample Kyverno policy shipped restricting Certificate-create on this specific ClusterIssuer to the paddock-controller SA.

7. **Existing pre-2f run pods continue running** with the old shape (root key in their `<run>-proxy-tls` Secret). They keep working — pre-1.0 evolve in place; no migration of in-flight runs. New runs pick up the new shape on the next reconcile pass.

8. **`ca.crt` populated by cert-manager in the per-run Secret contains the ROOT cert.** This is cert-manager's default behavior — it adds the chain root for client convenience. We deliberately do NOT mount this key into the agent (only `tls.crt`). The fact that the root cert appears in tenant namespace Secrets is information disclosure (not key disclosure) — the root cert is public anyway (it's a CA certificate, not a secret). Documented.

## 7. Tests

### 7.1 Controller (`internal/controller/proxy_tls_test.go`, `internal/controller/workspace_broker_test.go`)

- `TestEnsureProxyTLS_CreatesCertificate` — envtest: reconciles a HarnessRun, asserts a `cert-manager.io/v1 Certificate` resource is created in the run namespace with the right spec shape (isCA=true, ECDSA P-256, ClusterIssuer ref, OwnerReference to the HarnessRun, `secretName == "<run>-proxy-tls"`, duration=48h, no renewBefore).

- `TestEnsureProxyTLS_PendingWhenCertNotReady` — envtest: reconciles before cert-manager would have processed; asserts `EgressConfigured=False, Reason=ProxyCAPending`.

- `TestEnsureProxyTLS_ReadyAfterCertIssued` — envtest: stub cert-manager by setting `Certificate.status.conditions[Ready]=True` directly + creating the backing Secret with stub bytes; asserts `EgressConfigured=True, Reason=Cooperative` (or transparent).

- `TestEnsureProxyTLS_PerRunIsolation` — envtest: reconciles two HarnessRuns in different namespaces; asserts each gets its OWN `Certificate` resource (not the same one); asserts the resulting Secrets contain DIFFERENT `tls.key` bytes (the F-18 anti-regression check). Achievable with stubbed cert-manager because the test controls what bytes go into each Secret — the assertion is about *resource separation*, not cert content.

- Parallel set under `internal/controller/workspace_broker_test.go` for the per-Workspace shape: `TestEnsureWorkspaceProxyTLS_CreatesCertificate`, `TestEnsureWorkspaceProxyTLS_PendingWhenCertNotReady`, `TestEnsureWorkspaceProxyTLS_PerWorkspaceIsolation`.

### 7.2 No new e2e

- The hostile suite covers run-time security posture; cert-manager integration is configuration. envtest with stubbed cert-manager-status is sufficient.
- A real-cert-manager e2e would test cert-manager itself, which is upstream's responsibility.
- TG-1 (per-run CA key unreachable from agent) and TG-6 (per-run CA key not actually per-run) — both addressed by the unit tests above. The TG-6 anti-regression test (`TestEnsureProxyTLS_PerRunIsolation`) is the load-bearing one; without it, a future refactor that goes back to byte-copying could silently re-introduce F-18.

### 7.3 New dependency

- `github.com/cert-manager/cert-manager` added to `go.mod`. The transitive dep set is non-trivial but contained — cert-manager has its own k8s.io vendoring constraints. Pinning to a version in the v1.x series compatible with our k8s.io v0.35.4 stack.

## 10. Migration / breaking-change posture

- Pre-1.0 evolve in place. Existing in-flight runs continue using their currently-mounted `<run>-proxy-tls` Secret content (root keypair) — no migration of running pods.
- New runs after the rollout get the per-run intermediate.
- Operator must:
  1. Set cert-manager's `--cluster-resource-namespace=paddock-system`.
  2. Apply the new `ClusterIssuer` manifest (ships in `config/proxy/`).
  3. Update controller flags: remove `--proxy-ca-secret-name` / `--proxy-ca-secret-namespace`; add `--proxy-ca-cluster-issuer-name=paddock-proxy-ca-issuer` (or accept the default).
- Each behaviour-changing commit gets `feat(security)!:` per the breaking-change-commits convention. The combined `BREAKING CHANGE:` footer language: "Controller flags `--proxy-ca-secret-name` and `--proxy-ca-secret-namespace` removed; `--proxy-ca-cluster-issuer-name` added (default `paddock-proxy-ca-issuer`). Operators must apply `config/proxy/cluster-issuer.yaml` and set cert-manager's `--cluster-resource-namespace=paddock-system`."

## 11. Risks and mitigations

1. **`--cluster-resource-namespace` is a cluster-wide cert-manager flag.** Setting it to `paddock-system` affects every other ClusterIssuer of `kind: CA` in the cluster — they all start looking in `paddock-system` for their backing Secrets. **Mitigation:** loud install-doc callout. For operators with pre-existing CA-type ClusterIssuers using a different namespace, this is a coordination burden — they need to either (a) move their backing Secrets to `paddock-system` or (b) defer the Paddock install until they can. Documented as an upgrade prerequisite.

2. **Operator misconfig grants tenants `Certificate`-create RBAC.** A tenant with that RBAC could create a `Certificate` referencing our ClusterIssuer and request a CA-signed intermediate of their own, then mint leaves for arbitrary hostnames. **Mitigation:** sample Kyverno policy restricts Certificate-create on this specific ClusterIssuer to the paddock-controller SA. Documented in install README. The risk surface is identical to "operator broadly grants any privileged RBAC" — Paddock is not introducing a new risk shape, just amplifying the importance of one existing principle.

3. **cert-manager dependency on the controller hot path.** Per-run reconciliation now depends on cert-manager being healthy. Today's flow already does — `ensureProxyTLS` reads a cert-manager-managed Secret. The change moves the dependency from "Secret presence" to "Certificate Ready," which is cert-manager-internal-state-aware. Same operational footprint; clearer error surfacing via Certificate.status.

4. **Anti-regression coverage.** The F-18 root cause is a single byte-copy in 14 lines of code. A future refactor that "simplifies" the cert-manager Certificate flow back to copying the source Secret would silently re-introduce F-18. **Mitigation:** `TestEnsureProxyTLS_PerRunIsolation` explicitly asserts that two runs produce distinct `tls.key` bytes. As long as that test exists and passes, F-18 cannot regress without explicit and visible removal of the test.

5. **Workspace intermediate rotation mid-seed.** Documented as a known limitation in §6 case 4. Operationally rare; no hot-reload in scope.

6. **`ca.crt` populated by cert-manager in tenant Secrets.** Cert-manager's default behavior writes the cluster root cert into the per-tenant Secret as `ca.crt`. We don't mount it; it's information disclosure of a public CA cert (not key material). Explicitly documented in §6 case 8.

## 13. Open questions

None at design time. All scope decisions resolved in the brainstorming Q1–Q4:

- Q1: issuer architecture = cert-manager `ClusterIssuer` of `kind: CA`.
- Q2: scope = `ClusterIssuer` (cluster-scoped) with operator-side documentation about `--cluster-resource-namespace=paddock-system` and Certificate-create RBAC discipline.
- Q3: agent trust anchor = per-run intermediate cert ONLY (no cluster root in the agent's bundle).
- Q4: seed-Pod granularity = per-Workspace intermediate (lifecycle matches existing `<workspace>-proxy-tls` Secret).
