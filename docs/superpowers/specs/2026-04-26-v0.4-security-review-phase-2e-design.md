## 1. Summary

Phase 2e closes the three Pod-spec-shaped findings from the v0.4 audit:

- **F-37 (P1)** ŌĆö Run-pod containers (agent / adapter / collector) carry no `SecurityContext`, so PSS `restricted` is not enforceable. The controller starts authoring per-container `SecurityContext` matching each container's image-trust tier, plus a pod-level `SeccompProfile: RuntimeDefault` that all containers inherit.
- **F-39 (P1)** ŌĆö `run.Spec.ExtraEnv` is appended after controller-emitted env, letting a tenant override `HTTPS_PROXY` / `SSL_CERT_FILE` / `NODE_EXTRA_CA_CERTS` and silently bypass cooperative-mode interception. The webhook gains a name-collision check against a Paddock-reserved set; the controller reverses the env-append order as belt-and-braces.
- **F-31 (P2)** ŌĆö `extraEnv.valueFrom` only blocks `secretKeyRef`; the other three shapes (`fieldRef`, `resourceFieldRef`, `configMapKeyRef`) reach the agent un-audited. The webhook tightens the existing check to "any `valueFrom` shape."

The phase touches `internal/controller/pod_spec.go`, `internal/webhook/v1alpha1/harnessrun_webhook.go`, paired tests, plus ADR-0010, spec 0002, and the audit findings doc. No broker / proxy / API CRD code is modified.

## 2. Goals and non-goals

**Goals:**

1. Every controller-emitted run-pod container declares a `SecurityContext` matching its image-trust tier; first-party containers (collector, proxy, iptables-init) individually satisfy PSS `restricted`. Operators no longer need to label run namespaces with PSA to obtain first-party container hardening (PSA still recommended for the tenant agent image).
2. `extraEnv` cannot override Paddock-reserved env names. Webhook rejects at admission; controller is defense-in-depth via reversed append order.
3. `extraEnv.valueFrom` is a closed set in any shape, not just `secretKeyRef`.
4. Audit findings F-31, F-37, F-39 each have explicit Resolution lines.
5. Test posture: `pod-security-admission` policy package validates the built pod spec at PSS `baseline`; per-first-party-container restricted compliance is asserted in isolation.

**Non-goals (deferred):**

- New ADRs. ADR-0010 (PSS) gets a backfill paragraph; the controller-authors-SecurityContext stance was already implicit in ADR-0010 plus the F-37 audit recommendation. No new ADR needed.
- A "relaxed envelope" API for the adapter (per Q1: adapter gets the same baseline envelope as the agent ŌĆö no opt-in/opt-out flag).
- Other Pod-spec carry-overs ŌĆö e.g., `<run>-broker-creds` envFrom-injection key validation deferred from F-41 (Phase 2d). One phase, three findings; the broker-creds work is its own future phase.
- Feature-flagging or gradual rollout. Pre-1.0 evolve in place; the breaking-change behaviours (Sections 4.3 + 5.3) ship outright with `feat(security)!:` commits.
- Forcing `RunAsNonRoot: true` on the tenant agent image ŌĆö see Section 3.1 for the reasoning. PSS `restricted` at the *pod* level is not achievable as long as the agent is a tenant-supplied image.

## 3. F-37 SecurityContext envelope

### 3.1 Pod-level

`buildPodSpec` (`internal/controller/pod_spec.go:227ŌĆō273`) gains a pod-level `SecurityContext`:

```go
SecurityContext: &corev1.PodSecurityContext{
    SeccompProfile: &corev1.SeccompProfile{
        Type: corev1.SeccompProfileTypeRuntimeDefault,
    },
}
```

We **do not** set pod-level `RunAsNonRoot: true`. Per K8s semantics, pod-level `RunAsNonRoot:true` would force the agent (tenant image) to run as non-root or fail to start at the kubelet runtime check. Container-level overrides on the agent would then have to set `RunAsNonRoot:false`, which violates PSS `restricted` at the container level too. This is the structural reason the audit's "PSS `restricted` is enforceable" framing is partially overstated for the *pod* ŌĆö what we actually offer is PSS `baseline` at the pod level, with each first-party container individually satisfying `restricted`.

Setting seccomp at the pod level satisfies the PSS-restricted seccomp rule for *all* containers in one place; per-container `runAsNonRoot` and `readOnlyRootFilesystem` are set only on containers whose image we control.

### 3.2 Per-container envelopes

Mirrors the existing `seedContainerSecurityContext` helper shape (`internal/controller/workspace_seed.go:603ŌĆō608`). New shape per container:

| Container | Trust tier | Envelope target | RunAsNonRoot | RO root FS | Drop ALL caps | No priv-esc | Seccomp |
|---|---|---|---|---|---|---|---|
| **agent** | tenant image | PSS-baseline | unset | unset | yes | yes | inherits pod RuntimeDefault |
| **adapter** | template-author image | PSS-baseline | unset | unset | yes | yes | inherits pod RuntimeDefault |
| **collector** | first-party | PSS-restricted | true | true | yes | yes | inherits pod RuntimeDefault |
| **proxy** | first-party (already set; parity addition) | PSS-restricted | implicit via `RunAsUser:1337` | true (existing) | yes (existing) | yes (existing) | add explicit `RuntimeDefault` for parity |
| **iptables-init** | first-party (unchanged) | restricted-with-NET_ADMIN | UID 0 + NET_ADMIN/NET_RAW | true (existing) | drop ALL except NET_ADMIN/NET_RAW (existing) | yes (existing) | unchanged |

Three implementation notes:

1. **Adapter is template-author-supplied** (per `template.Spec.EventAdapter.Image`). The audit's wording calls it "first-party," but in code reality the image is operator/template-author-controlled. Per Q1 (answer B), the adapter gets the same baseline envelope as the agent ŌĆö drop caps, no priv-esc, but neither `RunAsNonRoot:true` nor `ReadOnlyRootFilesystem:true`. This is the minimum hardening that does not break a template author who picks an adapter image that needs a writable rootfs or runs as UID 0.
2. **Collector with `ReadOnlyRootFilesystem:true`** is verified safe. The collector writes only to `/paddock` (shared emptyDir), the workspace PVC, and the projected SA-token volume. No rootfs writes. If a future change introduces a tmp-write requirement, the standard remediation is a tmpfs `emptyDir` at `/tmp`.
3. **Proxy seccomp parity addition.** The proxy already sets the rest of the restricted envelope (`RunAsUser:1337`, `AllowPrivilegeEscalation:false`, `ReadOnlyRootFilesystem:true`, `Capabilities.Drop:[ALL]`). The pod-level seccomp will cover it, but adding an explicit `SeccompProfile: RuntimeDefault` at the proxy container level matches the "every first-party container declares its full envelope explicitly" convention.

### 3.3 Helper organisation

A new helper at the top of `pod_spec.go` matching `seedContainerSecurityContext`'s shape ŌĆö call it `runContainerSecurityContextBaseline()` ŌĆö returns the baseline envelope used by agent + adapter. A second `runContainerSecurityContextRestricted()` returns the restricted envelope used by collector. Inlining at the call sites (`buildAgentContainer`, `buildAdapterContainer`, `buildCollectorContainer`) is also acceptable; the helpers exist to make the intent self-documenting and avoid copy-paste drift between agent and adapter.

The TODO comment at `pod_spec.go:284ŌĆō290` is removed.

## 4. F-39 webhook + controller changes

### 4.1 Webhook reserved-name set

`internal/webhook/v1alpha1/harnessrun_webhook.go` gets a package-level reserved set:

```go
// reservedExtraEnvLiterals are env var names the controller authors itself
// on the agent container. Tenant overrides via spec.extraEnv are rejected
// at admission ŌĆö HTTPS_PROXY / SSL_CERT_FILE in particular are load-bearing
// for cooperative-mode interception (F-39).
var reservedExtraEnvLiterals = map[string]struct{}{
    "HTTPS_PROXY":         {},
    "HTTP_PROXY":          {},
    "NO_PROXY":            {},
    "SSL_CERT_FILE":       {},
    "NODE_EXTRA_CA_CERTS": {},
    "REQUESTS_CA_BUNDLE":  {},
    "GIT_SSL_CAINFO":      {},
}

// reservedExtraEnvPrefix reserves the entire PADDOCK_ namespace for the
// controller. New PADDOCK_* envs added to buildEnv inherit this protection
// without requiring the literal set above to be updated.
const reservedExtraEnvPrefix = "PADDOCK_"
```

The existing `for i, e := range spec.ExtraEnv` loop in `validateHarnessRunSpec` (line 251) gains a name-collision check that runs whether or not `e.ValueFrom` is set:

```go
if _, reserved := reservedExtraEnvLiterals[e.Name]; reserved ||
    strings.HasPrefix(e.Name, reservedExtraEnvPrefix) {
    errs = append(errs, field.Forbidden(
        specPath.Child("extraEnv").Index(i).Child("name"),
        "env name is reserved by the controller; "+
            "see docs/specs/0002-broker-proxy-v0.3.md ┬¦5.4"))
}
```

### 4.2 Controller defense-in-depth

`buildEnv` (`internal/controller/pod_spec.go:636ŌĆō677`) reverses its append order: `run.Spec.ExtraEnv` is appended *first*, then the eight `PADDOCK_*` standard set, then the proxy/CA fan-out. Under K8s last-wins resolution, controller-authored env wins on any duplicate key, so even if the webhook ever has a bug or a controller-internal path bypasses admission validation, the resulting Pod spec carries the controller's authoritative values.

One-line conceptual change; the rewritten function preserves all existing env composition logic verbatim.

### 4.3 Migration / breaking-change posture

- Existing HarnessRuns in a cluster that *currently* set a reserved key (e.g., a HarnessTemplate emitting `extraEnv: [{name: HTTPS_PROXY, value: "http://corp-proxy:3128"}]` for legitimate operator reasons) will fail webhook validation on next CREATE / UPDATE.
- The audit's framing is correct: such templates were *already* a security regression, silently disabling Paddock's MITM cert trust path. Rejecting them at admission is the intended posture.
- Pre-1.0 evolve-in-place (per project memory). The commit gets `feat(security)!:` per the breaking-change-commits convention. Migration note in the commit body: "Operators using `extraEnv` to set HTTPS_PROXY / SSL_CERT_FILE etc. must remove these ŌĆö Paddock now manages these envs authoritatively. If a custom proxy is needed, propose an explicit spec field rather than an `extraEnv` collision."
- No new CRD version, no conversion webhook.

## 5. F-31 webhook tightening

### 5.1 Generalise the existing valueFrom check

The existing `extraEnv` loop already checks `e.ValueFrom.SecretKeyRef`. Generalise to any non-nil `valueFrom`:

```go
if e.ValueFrom != nil {
    errs = append(errs, field.Forbidden(
        specPath.Child("extraEnv").Index(i).Child("valueFrom"),
        "valueFrom is not permitted on extraEnv; use a literal value, "+
            "or declare a credential on the template's requires and grant "+
            "via a BrokerPolicy (see spec 0002 ┬¦5.4)"))
}
```

This subsumes the existing `SecretKeyRef`-only branch (no longer reachable as a separate check), and closes `fieldRef`, `resourceFieldRef`, and `configMapKeyRef`.

### 5.2 Loop ordering

Two checks per `extraEnv` entry, accumulating into the same `field.ErrorList`:

1. Reserved-name check (F-39) ŌĆö fires regardless of `valueFrom`.
2. `valueFrom != nil` check (F-31) ŌĆö independent.

A single CREATE that includes both shapes (e.g., `extraEnv: [{name: HTTPS_PROXY, valueFrom: {fieldRef: ...}}]`) gets all errors back at once.

### 5.3 Migration / breaking-change posture

- Tenants currently using `fieldRef` for innocuous purposes (pod name / namespace passthrough) will fail validation. Per the audit's recommendation: if a future use case needs `fieldRef`, add it as an explicit named field on the HarnessRun spec ŌĆö explicitly out of scope for this phase.
- The audit notes these shapes are rare in practice (no test exercises them today; no first-party template uses them).
- Same `feat(security)!:` commit type and migration-note approach as F-39.

## 9. Risks and mitigations

1. **Pod fails to start under `ReadOnlyRootFilesystem:true` for collector.** Verified safe ŌĆö collector writes only to `/paddock`, the workspace PVC, and the projected SA-token volume. Mitigation if a regression appears: add a tmpfs `emptyDir` at `/tmp` and mount it on the collector. Caught by existing e2e harness runs that exercise the collector's output path.
2. **Operator templates that previously injected `HTTPS_PROXY` via `extraEnv` will now fail validation.** Acknowledged breaking change; commit and PR body call this out. Pre-1.0 posture: this was already a silent security regression.
3. **PSS-admission package upgrade breaks the validator test on a future K8s minor.** This is the *intended* signal ŌĆö if K8s tightens the baseline profile in a future version, our test surfaces it on dependency bump (which is grouped via the existing dependabot `k8s.io` group, so reviewer attention is concentrated).
4. **Adapter image authors who relied on running as root or with mounted caps.** No breakage today (adapter envelope drops caps + no priv-esc, but doesn't force non-root or RO root). If a future adapter image needs a cap, the contract is "declare it explicitly via the template" (not in scope; would be a separate spec change).
5. **`buildEnv` reversal subtly changes the order of env vars in the resulting Pod manifest.** Tools that reason about Pod manifests by index (rare but possible ŌĆö e.g., debug tooling) might surface the change. Functionally invisible ŌĆö last-wins semantics are unchanged for any non-colliding key set, and colliding keys now resolve to the controller's value (the intended fix).
