# Paddock Core — v0.1 Spec

Status: draft
Scope: the three core CRDs (`HarnessTemplate`/`ClusterHarnessTemplate`, `HarnessRun`, `Workspace`), their controllers, the pod runtime shape, the adapter/collector contract, and the local dev loop. This is the substrate every later milestone (bridges, broker, egress, archive) assumes.

Out of scope: the broker, bridges (including Linear), egress policy, git-proxy sidecar, archive-to-S3, workspace snapshots/forking, multi-repo support on runs, long-lived credential issuance.

## 1. Guiding decisions

| Decision | Choice | Rationale |
| --- | --- | --- |
| v0.1 scope | All three CRDs (Template + Run + Workspace) | Every future feature assumes Workspace; adding it later means rewriting volume plumbing. |
| Language / framework | Go + kubebuilder | Default for K8s operators; largest contributor surface. |
| Repo layout | Monorepo, permanent | Explicit choice (kubernetes/kubernetes-style). Simplifies contract evolution; documented as deliberate. |
| Template contract | Typed fields + optional `podTemplateOverlay` escape hatch | Paddock reasons about the fields it needs (image, resources, credentials, adapter), users escape-hatch the rest. |
| `harness` field | Free-form label only | No per-harness code in the core controller. Adapters carry harness-specific logic. |
| Template scope | `ClusterHarnessTemplate` + namespaced `HarnessTemplate` | Platform team publishes the cluster catalog; teams override locally (mirrors `ClusterRole`/`Role`). |
| Command variable substitution | Env-var based (`$PADDOCK_PROMPT_PATH`, etc.) | Standard shell expansion; no new templating language. |
| Prompt passing | File-mounted via owned Secret per run | Prompts can carry sensitive material; a ConfigMap exposes them to anyone with `configmaps get` on the namespace. See ADR-0011. |
| Prompt source | Inline (`spec.prompt`) or ref (`spec.promptFrom`) | One of. Bridges use inline; ops pipelines use ref. |
| Credentials (pre-broker) | Typed `credentials[]` with Secret refs | Same schema the broker will later satisfy; no migration. |
| Ownership | Workspace → PVC; HarnessRun → Job. No cross-ownership. | Runs terminate, workspaces persist. Finalizers enforce "workspace can't be deleted while referenced." |
| Workspace access mode | RWO default | Works on Kind and every cloud; one run at a time is a feature, not a limitation. |
| Concurrency gate | `Workspace.status.activeRunRef` + controller gate | Declarative, visible, no PVC attach errors. |
| Run status shape | Phase enum + Conditions (Job-style) | Readable at a glance (`kubectl get`) with structured detail. |
| Cancellation | Finalizer-driven graceful shutdown; Job delete via Background propagation | Harness gets SIGTERM from the kubelet, commits WIP, exits. Background (rather than Foreground) avoids a dependency on the garbage-collection controller, which is absent from envtest; PVC RWO serialises successor pods so we don't need GC ordering. |
| Retries | `backoffLimit=0` default, user-settable `spec.retries` | Agent failures rarely self-heal; retries are opt-in. |
| Output extraction | Collector sidecar reads `/workspace/.paddock/runs/<name>/result.json` → ConfigMap → status | Pod log pulls are an anti-pattern; sidecar is already there. |
| Workspace seeding | Dedicated seed Job spawned by Workspace controller | Clean phase transition; seed failures don't look like run failures. |
| Run overrides | Typed fields only (`timeout`, `resources`, `extraEnv`, `model`) | Constrains what a run can change; policy stays legible. |
| Event parsing | Pluggable per-harness adapter sidecar + generic collector sidecar | Upstream images stay untouched; core stays harness-agnostic. |
| Event transport | CRD `status.recentEvents` ring buffer + K8s watch | No streaming infra in v0.1; full history persists in workspace. |
| Log/event persistence | Sidecar writes `raw.jsonl`, `events.jsonl`, `result.json` to Workspace PVC | Follows workspace lifecycle; future archive ships logs for free. |
| Ephemeral runs (no workspace) | Auto-provision a throwaway Workspace + size-bounded status/ConfigMap fallback | One-shot runs still have inspectable output after pod is GC'd. |
| Run TTL | No default; `spec.ttlSecondsAfterFinished` optional | Matches `Job` convention; platform teams add policy later. |
| Admission webhook | Validation webhook for cross-object references and immutability | Cheap to add; prevents invalid state. |
| Observability | Prometheus metrics + structured JSON logs | Kubebuilder gives `/metrics` for free; traces deferred. |
| Reference harnesses | Claude Code (demo) + `paddock-echo` mock (CI/e2e) | Real use-case + reproducible test harness. |
| CLI | `kubectl-paddock` plugin, minimal commands | `submit`, `logs`, `events -f`, `status`. |
| API version | `paddock.dev/v1alpha1` | Breaking changes allowed while the API matures. |
| Dev loop | Tilt + Kind | Sub-10s feedback; kubebuilder-compatible. |
| Tests | envtest (reconciler) + Ginkgo + Kind-based e2e | Kubebuilder-default stack. |

## 2. CRDs

### 2.1 `ClusterHarnessTemplate` and `HarnessTemplate`

Two kinds sharing one schema (`HarnessTemplateSpec`). `ClusterHarnessTemplate` is cluster-scoped and typically maintained by the platform team; `HarnessTemplate` is namespaced and may reference-and-override a cluster template. A `HarnessRun` resolves its template name by preferring a namespaced `HarnessTemplate` with that name, falling back to the cluster one. (Override semantics: v0.1 starts with a single `baseTemplateRef` field for namespaced templates that inherits spec and allows narrow field overrides; full overlay semantics are a v0.2 concern.)

```yaml
apiVersion: paddock.dev/v1alpha1
kind: ClusterHarnessTemplate
metadata:
  name: codex-default
spec:
  harness: codex                       # free-form label, observability only
  image: ghcr.io/paddock/harness-codex:v1
  command: ["sh", "-c", "codex exec --json \"$(cat $PADDOCK_PROMPT_PATH)\" | tee $PADDOCK_RAW_PATH"]
  # Optional args merged after command; env-var expansion via K8s $(VAR) is supported.
  args: []
  defaults:
    model: gpt-5-codex                 # exported as PADDOCK_MODEL
    timeout: 30m
    resources:
      requests: { cpu: 500m, memory: 2Gi }
      limits:   { cpu: 2,    memory: 4Gi }
  eventAdapter:                        # OPTIONAL. If omitted, no fine-grained events.
    image: ghcr.io/paddock/adapter-codex:v1
    # adapter reads $PADDOCK_RAW_PATH, writes $PADDOCK_EVENTS_PATH
  credentials:
    - name: openai
      secretRef: { name: openai-creds, key: api-key }
      envKey: OPENAI_API_KEY
  workspace:
    required: true                     # this template must run against a workspace
    mountPath: /workspace
  podTemplateOverlay:                  # OPTIONAL merge into generated PodSpec
    spec:
      nodeSelector: {}
      tolerations: []
status:
  observedGeneration: 1
  conditions:
    - type: Ready
      status: "True"
```

Key schema notes:
- `command`/`args` are mandatory at the template level. There are no per-harness defaults inside the controller.
- `eventAdapter.image` is how a template declares "I want fine-grained events parsed." Without it, `status.recentEvents` carries only lifecycle events; `raw.jsonl` is still persisted.
- `credentials[].secretRef` is a standard `SecretKeySelector`. When the broker arrives (v0.4), it fulfils the same field by synthesizing short-lived Secrets; templates don't change.
- `workspace.required` lets the template declare whether a run must reference a workspace or can run ephemerally.

### 2.2 `HarnessRun`

```yaml
apiVersion: paddock.dev/v1alpha1
kind: HarnessRun
metadata:
  name: refactor-auth-042226
  namespace: team-platform
spec:
  templateRef:                         # exactly one of
    name: codex-default
    # kind: ClusterHarnessTemplate     # optional; auto-resolve if omitted
  workspaceRef: ws-app                 # required if template.workspace.required
  prompt: |                            # exactly one of prompt / promptFrom
    Refactor auth.py to use async patterns. Keep tests green.
  # promptFrom:
  #   configMapKeyRef: { name: prompts, key: refactor-async }
  timeout: 45m                         # override of template.defaults.timeout
  retries: 0
  resources:                           # override of template.defaults.resources
    requests: { memory: 4Gi }
  extraEnv:
    - { name: DEBUG, value: "1" }
  model: gpt-5-codex                   # override of template.defaults.model
  ttlSecondsAfterFinished: 86400       # optional
status:
  phase: Running                       # Pending | Running | Succeeded | Failed | Cancelled
  conditions:
    - type: TemplateResolved
      status: "True"
    - type: WorkspaceBound
      status: "True"
    - type: JobCreated
      status: "True"
    - type: PodReady
      status: "True"
    - type: Completed
      status: "False"
      reason: InProgress
  jobName: refactor-auth-042226-xk7d
  workspaceRef: ws-app
  startTime: "2026-04-22T14:03:11Z"
  completionTime: null
  recentEvents:                        # ring buffer, last ~100
    - ts: "2026-04-22T14:03:22Z"
      type: ToolUse
      tool: read
      path: auth.py
    - ts: "2026-04-22T14:03:25Z"
      type: Message
      role: assistant
      content: "I see the issue..."
  outputs:                             # populated from result.json on completion
    pullRequests: []
    filesChanged: 0
    summary: ""
```

Spec is immutable after creation (enforced by admission webhook). Mutating a run is not a supported operation — submit a new one. Status is the only thing that changes.

### 2.3 `Workspace`

```yaml
apiVersion: paddock.dev/v1alpha1
kind: Workspace
metadata:
  name: ws-app
  namespace: team-platform
spec:
  storage:
    size: 20Gi
    storageClass: standard             # optional; cluster default if unset
    accessMode: ReadWriteOnce          # v0.1 default; RWX is v0.3+
  seed:                                # exactly one of seed / fromArchive (v0.2+)
    git:
      url: https://github.com/me/app
      branch: main
      depth: 1

  phase: Seeding                       # Seeding | Active | Failed | Terminating
  pvcName: pvc-ws-app
  seedJobName: ws-app-seed-2xk7
  activeRunRef: refactor-auth-042226
  totalRuns: 3
  conditions:
    - type: PVCBound
      status: "True"
    - type: Seeded
      status: "True"
    - type: Ready
      status: "True"
  lastActivity: "2026-04-22T14:03:22Z"
```

## 3. Controllers

Three reconcilers, one controller-manager binary.

### 3.1 `HarnessTemplate` / `ClusterHarnessTemplate` controller

Near-passive. Validates the spec, computes a resolved spec for namespaced templates that inherit from cluster templates, maintains `status.Ready`. No pod-shaped side effects.

### 3.2 `Workspace` controller

Reconcile loop (simplified):

1. Ensure `PVC` exists (owned by Workspace). If creating, set `PVCBound=False` until bound.
2. If `spec.seed.git` is set and `Seeded=False`:
   - Spawn a short-lived seed `Job` (owned by Workspace) that mounts the PVC and clones.
   - On seed Job completion: set `Seeded=True` or `Seeded=False, reason=...`.
3. Maintain `phase`: `Seeding` → `Active` → (later) `Terminating`.
4. Maintain `status.activeRunRef` by watching `HarnessRun` objects that reference this workspace (via a field index + informer). `activeRunRef` is the single `Pending→Running` run; others stay `Pending`.
5. Finalizer: on delete, refuse to finalize while `activeRunRef` is set. When clear, let cascade delete PVC.

### 3.3 `HarnessRun` controller

Reconcile loop (simplified):

1. **Resolve template**. Namespaced first, cluster second. If unresolved, set `Failed/TemplateNotFound`.
2. **Resolve workspace**. If `spec.workspaceRef` set, ensure Workspace is `Active`. If its `activeRunRef` is another run, stay `Pending` with `reason=WorkspaceBusy`. If this run is the active one, proceed. If `spec.workspaceRef` unset and template allows, auto-provision an ephemeral workspace (owned by this run only, deleted on TTL).
3. **Bind workspace**. Patch `Workspace.status.activeRunRef = <this run>`.
4. **Materialise the prompt Secret**. Copy inline `spec.prompt` or resolve `spec.promptFrom`. The Secret (`<run>-prompt`, Opaque, key `prompt.txt`) is owned by the run. See ADR-0011 for why Secret rather than ConfigMap.
5. **Generate the PodSpec**. Merge typed template fields + run overrides + `podTemplateOverlay`. Inject:
   - Main container (image, command, args as declared by template).
   - Adapter sidecar iff `template.spec.eventAdapter.image` is set.
   - Collector sidecar (core image, always injected).
   - Shared `emptyDir` volume mounted at `/paddock` in all three.
   - Prompt ConfigMap mounted at `/paddock/prompt/prompt.txt`.
   - Workspace PVC mounted at `template.spec.workspace.mountPath` (default `/workspace`).
   - Env vars: `PADDOCK_PROMPT_PATH`, `PADDOCK_RAW_PATH`, `PADDOCK_EVENTS_PATH`, `PADDOCK_RESULT_PATH`, `PADDOCK_WORKSPACE`, `PADDOCK_RUN_NAME`, `PADDOCK_MODEL`.
   - Credentials wired as `env` from Secret refs.
   - `terminationGracePeriodSeconds` set from template or a conservative default (60s).
6. **Create the Job**. Owned by the run. `backoffLimit=spec.retries` (default 0).
7. **Watch**: Job status, Pod status, the run's output ConfigMap (populated by the collector).
8. **Emit K8s Events** on meaningful transitions.
9. **Update `status.recentEvents`** from the collector-written ConfigMap (size-bounded; full events in workspace).
10. **On Job completion**: parse `result.json` (from collector's ConfigMap) into `status.outputs`; set `phase` and `Completed` condition.
11. **On deletion**: finalizer sends cancellation. Delete the backing Job with Background propagation; the kubelet then drives the Pod through SIGTERM → `terminationGracePeriodSeconds` → SIGKILL. Clear `Workspace.status.activeRunRef`. Remove finalizer. (Background rather than Foreground because envtest lacks the GC controller; the PVC RWO access mode already serialises the next run against the previous one.)

## 4. Pod runtime shape

Every HarnessRun materializes a Pod with this shape:

```
Pod
├─ volumes
│  ├─ shared    (emptyDir)         mounted at /paddock in all containers
│  ├─ prompt    (secret)           mounted at /paddock/prompt
│  └─ workspace (pvc)              mounted at /workspace (template-configurable)
│
├─ containers
│  ├─ agent                        # upstream image, unmodified
│  │  image: <template.image>
│  │  command/args: <template>     # expected to tee raw output to $PADDOCK_RAW_PATH
│  │  env: PADDOCK_*, creds
│  │  volumeMounts: shared, prompt, workspace
│  │
│  ├─ adapter                      # iff template.eventAdapter.image
│  │  image: <template.eventAdapter.image>
│  │  # reads $PADDOCK_RAW_PATH → writes $PADDOCK_EVENTS_PATH
│  │  volumeMounts: shared
│  │
│  └─ collector                    # core image, always present
│     # tails raw.jsonl and events.jsonl → persists to workspace
│     # watches for result.json → writes output ConfigMap
│     # writes ring buffer to a "status" ConfigMap polled by controller
│     volumeMounts: shared, workspace
│
├─ terminationGracePeriodSeconds: 60
└─ restartPolicy: Never             # Jobs control retries
```

### 4.1 Sidecar lifecycle

Adapter and collector must be declared as **native sidecar containers** (`restartPolicy: Always` on an init container slot — the K8s 1.29+ sidecar pattern). This guarantees:
- They start before the main container.
- They stop **after** the main container exits (so they can drain final bytes and write `result.json`).
- Pod completion is gated on the main container, not the sidecars.

### 4.2 Env-var contract

| Variable | Value | Purpose |
| --- | --- | --- |
| `PADDOCK_PROMPT_PATH` | `/paddock/prompt/prompt.txt` | File containing the prompt |
| `PADDOCK_RAW_PATH` | `/paddock/raw/out` | Where the agent tees raw output |
| `PADDOCK_EVENTS_PATH` | `/paddock/events/events.jsonl` | Where the adapter writes PaddockEvents |
| `PADDOCK_RESULT_PATH` | `/workspace/.paddock/runs/<run>/result.json` | Where the harness writes final structured output (optional) |
| `PADDOCK_WORKSPACE` | `/workspace` (or template override) | Workspace mount point |
| `PADDOCK_RUN_NAME` | `<run name>` | For logging, filenames, branch names |
| `PADDOCK_MODEL` | `<spec.model or defaults.model>` | Passed through to harness invocation |

## 5. Event and output schemas

### 5.1 `PaddockEvent`

JSON-lines format (one event per line), written by adapter to `/paddock/events/events.jsonl`:

```json
{"ts":"2026-04-22T14:03:22Z","type":"ToolUse","tool":"read","path":"auth.py"}
{"ts":"2026-04-22T14:03:25Z","type":"Message","role":"assistant","content":"..."}
{"ts":"2026-04-22T14:04:01Z","type":"FileEdit","path":"auth.py","linesAdded":42,"linesRemoved":8}
{"ts":"2026-04-22T14:04:30Z","type":"Commit","sha":"abc123","message":"async auth"}
{"ts":"2026-04-22T14:05:00Z","type":"Elicitation","question":"Retry should be exponential?"}
{"ts":"2026-04-22T14:05:30Z","type":"Error","message":"..."}
{"ts":"2026-04-22T14:06:00Z","type":"Result","pullRequests":["..."],"summary":"..."}
```

Initial v0.1 event types: `ToolUse`, `Message`, `FileEdit`, `Commit`, `Elicitation`, `Error`, `Result`. The schema is intentionally minimal — adapters should emit what they have, not invent data.

### 5.2 `result.json`

Optional file written by the harness (or adapter as a fallback from the `Result` event) to `/workspace/.paddock/runs/<run>/result.json`:

```json
{
  "pullRequests": ["https://github.com/me/app/pull/4471"],
  "filesChanged": 14,
  "summary": "Converted auth to async; added tests.",
  "artifacts": []
}
```

Controller reads this at Job completion and writes it to `HarnessRun.status.outputs`.

## 6. Persistence layout (on the workspace PVC)

```
/workspace/                              # the seeded repo lives here
/workspace/.paddock/
  workspace.json                         # metadata (seed source, created, etc.)
  runs/
    <run-name-1>/
      raw.jsonl                          # verbatim harness output
      events.jsonl                       # parsed PaddockEvents
      result.json                        # final structured output
      meta.json                          # run metadata (start, end, exit code)
    <run-name-2>/
      ...
```

Follow-up runs (in a future Linear bridge) read `/workspace/.paddock/runs/` to build prior-run context. Archive-to-S3 (v0.2+) tars the whole PVC; logs come along for free.

## 7. Admission webhook

Validating webhook for CRDs (no mutation in v0.1).

- `HarnessTemplate` / `ClusterHarnessTemplate`
  - `image` required.
  - `command` required, non-empty.
  - `eventAdapter.image` must be a valid image reference when set.
- `HarnessRun`
  - Exactly one of `prompt`, `promptFrom`.
  - Referenced template must exist (resolved at admission time).
  - If referenced template has `workspace.required=true`, `workspaceRef` must be set or auto-provision must be permitted.
  - `spec` is immutable after creation (all fields except `metadata.labels`/`annotations`).
- `Workspace`
  - Exactly one of `spec.seed`, `spec.fromArchive` (the latter is a v0.2 field; webhook tolerates absence).
  - `spec.storage.size` required.
  - Immutable fields: `spec.storage`, `spec.seed` (once bound).

Webhook cert management via cert-manager (or kubebuilder's built-in dev scaffolding during local iteration).

## 8. Observability

Metrics (Prometheus, `/metrics` via controller-runtime) shipped in v0.1:

- `paddock_workspace_seed_duration_seconds{outcome}` histogram — seed Job latency from start to terminal state.
- `paddock_workspace_phase_transitions_total{from,to}` counter — Workspace phase flips; flags flapping Seeding↔Failed.
- `paddock_harnessrun_phase_transitions_total{from,to}` counter — HarnessRun phase flips.
- `paddock_harnessrun_duration_seconds{phase}` histogram — HarnessRun wall-clock from StartTime to CompletionTime.

`controller_runtime_*` covers the standard reconcile counts/latencies for free.

Deferred metrics (may land in v0.2 once we have more operational experience):

- `paddock_runs_total{phase,template,namespace}` counter
- `paddock_active_runs{template,namespace}` gauge
- `paddock_workspace_bytes{name,namespace}` gauge (best-effort from PVC `used`)
- `paddock_reconcile_errors_total{controller,kind,reason}` counter (partially covered by `controller_runtime_reconcile_errors_total`)

Logs: JSON, structured, with `run`, `workspace`, `template`, `reconcile_id` fields.

Traces: deferred.

## 9. CLI — `kubectl-paddock`

Shipped in v0.1, minimal surface:

```
kubectl paddock submit -t <template> --prompt "..." [--workspace <ws>] [--wait]
kubectl paddock list [runs|workspaces|templates]
kubectl paddock status <run>                 # human-friendly phase + conditions
kubectl paddock events <run> [-f]            # tails recentEvents; -f follows via watch
kubectl paddock logs <run> [--raw]           # reads workspace/.paddock/runs/<run>/{events,raw}.jsonl
kubectl paddock cancel <run>                 # graceful delete
```

Implementation: single Go binary, `cobra`, installed via `krew` plugin index. Shares `api/v1alpha1` types with the controller.

## 10. Reference harnesses

Two shipped in v0.1, each as its own image + adapter + sample template:

### 10.1 `paddock-harness-claude-code` (+ `paddock-adapter-claude-code`)

- Image: Debian slim + `claude` CLI + `git` + `gh` + `ripgrep`.
- Command template: `sh -c 'claude -p --output-format stream-json "$(cat $PADDOCK_PROMPT_PATH)" | tee $PADDOCK_RAW_PATH'`.
- Adapter: parses Claude Code stream-JSON → PaddockEvents.

### 10.2 `paddock-harness-echo` (+ `paddock-adapter-echo`)

- Image: tiny, no model dependency.
- Emits deterministic fake PaddockEvents on a timer, writes a `result.json`.
- Used by CI/e2e tests; the golden-path check that the whole pipeline works without a model provider.

(Codex CLI is a fast-follow for v0.2 — adapter is straightforward but not needed to validate the core.)

## 13. Acceptance criteria for v0.1 "done"

A fresh contributor can:

1. Clone the repo and run `hack/kind-up.sh && tilt up`, and within 2 minutes have a running controller-manager on Kind.
2. Apply `examples/template-echo.yaml` (ClusterHarnessTemplate for the mock), `examples/workspace-demo.yaml`, and `examples/run-hello.yaml`.
3. Observe `kubectl paddock events hello -f` streaming typed PaddockEvents.
4. Observe `kubectl paddock status hello` reporting `Succeeded`, with `status.outputs.summary` populated.
5. Re-submit against the same workspace and observe the workspace has `totalRuns: 2` and the prior run's `raw.jsonl` still readable from the PVC.
6. Apply `examples/template-claude-code.yaml` and `examples/run-claude-code.yaml` (requires an `Anthropic` API key Secret), observe a real agent run end to end.
7. `kubectl delete harnessrun <running-run>` triggers graceful shutdown within the grace period, workspace's `activeRunRef` clears, next Pending run starts.
8. `make test` and `make test-e2e` both pass from a clean checkout.

## 14. Deferred decisions (to be resolved before or during implementation)

- **PaddockEvent schema versioning.** Should events carry an explicit `schemaVersion`? (Likely yes; trivial to add.)
- **How does the collector surface events to the controller?** Polling an owned ConfigMap is proposed above; a projected `/status` writer Service is an alternative. Decide during implementation based on etcd write pressure.
- **Cluster-vs-namespaced template override semantics.** v0.1 supports a namespaced template inheriting from a cluster one with narrow overrides; the exact overridable fields need a short ADR.
- **Auto-provisioned ephemeral workspace lifetime.** When a run has no `workspaceRef`, is the auto-provisioned Workspace visible as a real CR? Probably yes (debuggability), with a TTL finalizer.
- **Pod security standards.** Enforce `restricted` PSS on the run namespace by default in Helm chart?
- **Git clone credentials in the Workspace seed Job.** v0.1 uses a plain Secret ref for private repos. Broker integration later.
- **Ring-buffer size on `status.recentEvents`.** Default 100; needs empirical tuning once reference harnesses are running.
- **Whether `ClusterHarnessTemplate` needs a per-namespace `allowedNamespaces` allowlist.** Probably yes, but can be added via admission policy rather than spec field.

## 15. Monorepo note (explicit)

The vision document described "each a separate repo, each independently versionable." v0.1 intentionally lives in a single monorepo (kubernetes/kubernetes-style). The contracts between the run platform, future bridges, and the broker are still stabilizing; splitting is cheap once they've proven out, and expensive before.
