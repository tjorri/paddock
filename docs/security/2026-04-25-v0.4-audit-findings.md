### F-01 — SSRF via broker endpoint (proxy → broker HTTP client)

| Field          | Value                                                                   |
|----------------|-------------------------------------------------------------------------|
| Component      | proxy                                                                   |
| Threats        | T-? (to be assigned during component review)                            |
| Severity       | Likelihood: Low · Impact: High · Risk: Medium                           |
| Priority       | P2                                                                      |
| Test coverage  | Pending — to be assessed in Stage 5                                     |

**Description.**
`BrokerClient.do()` constructs an HTTP request using `c.Endpoint + path` where `Endpoint` is set at client construction time from configuration. Gosec's taint analysis (G704) flags this as a potential SSRF vector: if an attacker can influence the `Endpoint` value (e.g., via a compromised Pod env injection, or a future misconfiguration path), they could direct the proxy to POST authenticated tokens to an arbitrary URL.

The `TokenPath` (G703) is read fresh on every call and sent as the `Authorization` header — meaning every request leaks the projected SA token to whoever controls the endpoint.

**Evidence.**
- `internal/proxy/broker_client.go:157` — `os.ReadFile(c.TokenPath)` (G703)
- `internal/proxy/broker_client.go:161` — `http.NewRequestWithContext(ctx, ..., c.Endpoint+path, ...)` (G704)
- `internal/proxy/broker_client.go:171` — `c.hc.Do(req)` (G704)
- gosec rules: G703, G704

**Recommendation.**
Phase 2 should:
1. Validate at `NewBrokerClient` that the endpoint is an HTTPS URL matching the expected in-cluster service pattern (e.g., `https://paddock-broker.<ns>.svc.cluster.local`).
2. Consider pinning the endpoint to a compile-time constant or a validated config field with an allowlist.
3. Ensure the proxy Pod's securityContext prevents env variable injection from workspace containers.

**Test gap correlation.**
Pending — assessed in Stage 5.

### F-02 — Context-less `net.Listen` in transparent proxy mode

| Field          | Value                                                                   |
|----------------|-------------------------------------------------------------------------|
| Component      | proxy                                                                   |
| Threats        | T-? (to be assigned during component review)                            |
| Severity       | Likelihood: Low · Impact: Low · Risk: Low                               |
| Priority       | P3                                                                      |
| Test coverage  | Pending — to be assessed in Stage 5                                     |

**Description.**
`cmd/proxy/main.go:204` calls `net.Listen("tcp", listenAddr)` directly rather than `(*net.ListenConfig).Listen(ctx, ...)`. Without context propagation, the `Listen` call itself cannot be cancelled; if the address is already in use (e.g., during a race between proxy restart and kernel TIME_WAIT), there is no context deadline. This is a resilience concern in a proxy that binds privileged ports.

**Evidence.**
- `cmd/proxy/main.go:204`
- noctx rule

**Recommendation.**
Phase 2 should replace `net.Listen(...)` with `(&net.ListenConfig{}).Listen(ctx, "tcp", listenAddr)` so the listener inherits the process-level context and can be cancelled during shutdown races.

**Test gap correlation.**
Pending — assessed in Stage 5.

### F-03 — TLS handshake without context in SNI sniffer

| Field          | Value                                                                   |
|----------------|-------------------------------------------------------------------------|
| Component      | proxy                                                                   |
| Threats        | T-? (to be assigned during component review)                            |
| Severity       | Likelihood: Medium · Impact: Medium · Risk: Medium                      |
| Priority       | P1                                                                      |
| Test coverage  | Pending — to be assessed in Stage 5                                     |

**Description.**
`internal/proxy/sniffer.go:64` calls `tls.Server(dummy, cfg).Handshake()` without using `HandshakeContext(ctx)`. The sniffer's purpose is to peek at the TLS ClientHello to extract the SNI hostname before deciding whether to allow the connection. A slow or malicious egress target could hold the connection open by sending a partial ClientHello, causing the goroutine to block indefinitely without any timeout or cancellation.

Since each intercepted outbound TCP connection spawns a goroutine that calls into the sniffer, a burst of slow-TLS connections could exhaust the proxy's goroutine pool.

**Evidence.**
- `internal/proxy/sniffer.go:64`
- noctx rule: `(*crypto/tls.Conn).Handshake must not be called. use (*crypto/tls.Conn).HandshakeContext`

**Recommendation.**
Phase 2 should:
1. Replace `.Handshake()` with `.HandshakeContext(ctx)` where `ctx` carries a short deadline (e.g., 5s, matching the broker timeout budget).
2. Consider whether the `peekConn`/`teeReader` infrastructure correctly propagates deadlines down to the underlying `net.Conn`.

**Test gap correlation.**
Pending — assessed in Stage 5.

### F-04 — Go 1.26 stdlib TLS/x509 vulnerabilities (crypto/tls, crypto/x509)

| Field          | Value                                                                                 |
|----------------|---------------------------------------------------------------------------------------|
| Component      | proxy (primary), controller (secondary)                                               |
| Threats        | T-? (to be assigned during component review)                                          |
| Severity       | Likelihood: Medium · Impact: High · Risk: High                                        |
| Priority       | P1                                                                                    |
| Test coverage  | Pending — to be assessed in Stage 5                                                   |

**Description.**
Five advisories affect `crypto/tls` and `crypto/x509` in go1.26, all fixed in go1.26.1 or go1.26.2. All are reachable via Paddock's call graph through `internal/proxy/mode.go` (MITM TLS termination) and `internal/proxy/sniffer.go` (SNI peek):

- **GO-2026-4947** — Unexpected work during chain building (`x509.Certificate.Verify`). CPU amplification risk.
- **GO-2026-4946** — Inefficient policy validation (`x509.Certificate.Verify`). CPU amplification risk.
- **GO-2026-4870** — Unauthenticated TLS 1.3 KeyUpdate record causes persistent connection retention and DoS (`tls.Conn.Handshake`/`HandshakeContext`/`Read`/`Write`). Reachable via proxy, controller HTTP client, and policy package.
- **GO-2026-4866** — Case-sensitive `excludedSubtrees` name constraints bypass auth in `x509.Certificate.Verify`. TLS certificate validation bypass.
- **GO-2026-4600** — Panic in name constraint checking for malformed certificates (`x509.Certificate.Verify`). Crash/DoS via crafted cert.
- **GO-2026-4599** — Incorrect enforcement of email constraints in `x509.Certificate.Verify`. Auth bypass.

The proxy performs MITM TLS termination for all egress connections from workspace containers, making these vulnerabilities directly relevant to Paddock's core trust boundary.

**Evidence.**
- govulncheck advisories: GO-2026-4947, GO-2026-4946, GO-2026-4870, GO-2026-4866, GO-2026-4600, GO-2026-4599
- Affected symbols: `x509.Certificate.Verify`, `tls.Conn.HandshakeContext`, `tls.Conn.Handshake`, `tls.Conn.Read`, `tls.Conn.Write`, `tls.Dialer.DialContext`
- Paddock callsites: `internal/proxy/mode.go:127`, `internal/proxy/sniffer.go:64`, `internal/proxy/server.go:320`, `internal/policy/intersect.go:164`, `internal/controller/broker_client.go:106`

**Recommendation.**
Upgrade the Go toolchain to go1.26.2 or later. This is a single toolchain bump; no Paddock source changes required. Track in go.toolchain directive.

**Test gap correlation.**
Pending — assessed in Stage 5.

### F-07 — picomatch ReDoS in bundled npm dep of claude-code harness (CVE-2026-33671)

| Field          | Value                                                                                                    |
|----------------|----------------------------------------------------------------------------------------------------------|
| Component      | paddock-claude-code harness image                                                                        |
| Threats        | T-? (to be assigned during component review)                                                             |
| Severity       | Likelihood: Low · Impact: Low · Risk: Low                                                                |
| Priority       | P3                                                                                                       |
| Test coverage  | Pending — to be assessed in Stage 5                                                                      |

**Description.**
Trivy reports CVE-2026-33671 (HIGH) in `picomatch` 4.0.3, a transitive dependency of the `@anthropic-ai/claude-code` npm package bundled inside `paddock-claude-code:dev`. The vulnerability is a Regular Expression Denial of Service (ReDoS) triggered by crafted extglob patterns passed to picomatch's matching functions. A fixed version (picomatch 4.0.4 / 3.0.2 / 2.3.2) exists upstream.

Because `picomatch` is a transitive dependency embedded inside the `@anthropic-ai/claude-code` CLI package (inside `tinyglobby`), Paddock cannot patch it directly — the fix requires either a new `@anthropic-ai/claude-code` release that updates its bundled deps, or pinning a post-fix npm version.

The `paddock-claude-code` image is a demo/harness image, not a production network-facing service. The ReDoS attack vector requires the ability to supply crafted glob patterns to the picomatch call site within the Claude CLI, which is not an externally exposed API surface in Paddock deployments. Impact is therefore Low.

**Evidence.**
- Trivy scan of `paddock-claude-code:dev` (run 2026-04-25)
- Affected package: `picomatch` 4.0.3 (in `usr/local/lib/node_modules/npm/node_modules/tinyglobby/node_modules/picomatch/package.json`)
- CVE: CVE-2026-33671 — picomatch ReDoS via crafted extglob patterns
- Fixed versions: picomatch 4.0.4, 3.0.2, 2.3.2
- Reference: https://avd.aquasec.com/nvd/cve-2026-33671

**Recommendation.**
1. Monitor `@anthropic-ai/claude-code` releases for a version that bumps picomatch to ≥4.0.4. Update `CLAUDE_VERSION` in `images/harness-claude-code/Dockerfile` accordingly and re-run `make trivy-images` to verify.
2. Remove `CVE-2026-33671` from `.trivyignore` once the upstream release incorporating the fix is pinned.

**Test gap correlation.**
Pending — assessed in Stage 5.

### F-08 — Helm Deployments missing explicit rollout strategy (paddock-broker, paddock-controller-manager)

| Field          | Value                                                                                                              |
|----------------|---------------------------------------------------------------------------------------------------------------------|
| Component      | paddock-broker Deployment, paddock-controller-manager Deployment (Helm chart)                                      |
| Threats        | T-? (to be assigned during component review)                                                                       |
| Severity       | Likelihood: Low · Impact: Low · Risk: Low                                                                          |
| Priority       | P3                                                                                                                 |
| Test coverage  | Pending — to be assessed in Stage 5                                                                                |

**Description.**
Neither the `paddock-broker` nor the `paddock-controller-manager` Deployment in the Helm chart has an explicit `spec.strategy:` field. Kubernetes defaults to `RollingUpdate` with `maxSurge: 25%` / `maxUnavailable: 25%`, but the absence of an explicit strategy means:

1. The intent is not captured in source control; future chart changes could silently alter rollout behaviour.
2. For a single-replica broker, a 25% maxUnavailable during rollout means a brief period with zero ready replicas, causing auth failures for any agent activity in flight.

kube-linter rule `no-rolling-update-strategy` flags this on both Deployments.

**Evidence.**
- kube-linter scan of `helm template paddock charts/paddock --namespace paddock-system` (run 2026-04-25)
- Rule: `no-rolling-update-strategy`
- Objects: `paddock-system/paddock-broker apps/v1, Kind=Deployment` and `paddock-system/paddock-controller-manager apps/v1, Kind=Deployment`

**Recommendation.**
Add explicit `strategy:` stanzas to both Deployments in the Helm chart (or its kustomize source):

```yaml
spec:
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1
      maxUnavailable: 0  # zero-downtime rollout for single-replica broker
```

Consider `maxUnavailable: 0` for the broker (network-facing) to avoid any gap in auth coverage during updates.

**Test gap correlation.**
Pending — assessed in Stage 5.

## 3. Severity scheme reference

OWASP-style risk = Likelihood × Impact, plus a separate Priority field.

**Likelihood:**

- **High** — an attacker who already has tenant-namespace access can trigger this with default Paddock config.
- **Medium** — requires a misconfiguration, a co-tenant, or non-default settings.
- **Low** — requires a specific chain of conditions or actor capabilities (e.g., compromised paddock-system component).

**Impact:**

- **High** — credential disclosure, cross-tenant data leak, audit suppression, or root-equivalent escalation.
- **Medium** — single-tenant data leak, denial of service against one run, audit completeness gap.
- **Low** — informational, hardening, defence-in-depth.

**Risk = quadrant of Likelihood × Impact** (3×3 matrix; H×H = High Risk, L×L = Low Risk, mixed = Medium Risk).

**Priority (separate from severity):**

- **P0** — drop everything; ship a fix or workaround for the next release.
- **P1** — must fix before v1.0.
- **P2** — should fix; schedule into a milestone.
- **P3** — nice-to-have / hardening / track-but-don't-block.

## 4. Static-analysis tooling summary

### 4.1 golangci-lint (with gosec, bodyclose, noctx)

**Run:** `make lint` (locally) / `golangci-lint (security linters enabled)` job in `.github/workflows/security.yml` (CI).

**Configuration:** `.golangci.yml`.

**Baseline counts:**

- Findings before triage: 14 raw gosec/noctx diagnostics across the codebase.
- Suppressions applied: 9 categories (see below).
- Genuine findings extracted: 3 → F-01, F-02, F-03.

**Suppressions (with rationale):**

- gosec G101 (codebase-wide) — matched constants are taxonomy labels (`"credential-issued"` AuditKind values), routing markers (`anthropicBearerPrefix`, `githubAppBearerPrefix`), and volume mount paths (`brokerTokenMountPath`). None are actual credential values; gosec matches on string content.
- gosec G104 in `internal/cli/.*` — `tabwriter.Flush()` to a CLI stdout/stderr writer; write errors are not actionable in CLI tools, consistent with the existing `errcheck` exclusion for `fmt.Fprintf`.
- gosec G301/G302 in `cmd/(adapter-claude-code|adapter-echo|collector)/.*` — `0o755` directories and `0o644` files in the Pod-local workspace (`/paddock/…`) are intentional; sibling containers in the same Pod must be able to read the audit trail, and the files are never exposed outside the Pod.
- gosec G304 in `cmd/(adapter-claude-code|adapter-echo|collector)/.*` — file paths come from flags/env set by the controller in the Pod spec (trusted control plane, not user input).
- gosec G304 in `internal/cli/submit.go` — `--prompt-file` flag is an intentional user-supplied path in a CLI tool; analogous to `cat <file>`.
- gosec G304 in `internal/(controller|proxy)/broker_client.go` — CA cert paths from operator configuration, not user-tainted input.
- gosec G304 in `internal/proxy/ca.go` and `cmd/proxy/main.go` — cert-manager-generated TLS cert/key paths and optional extra CA bundle path, both from operator configuration (mounted Secret volume).
- gosec G204 + noctx in `cmd/iptables-init/.*` — compile-time constant binary (`"iptables"`) with validated config flag arguments; init container runs to completion so context cancellation via `CommandContext` is not meaningful.
- gosec G204 in `test/.*` — hardcoded `"kubectl"` in e2e scaffold; noctx in `test/.*` — e2e helpers with no production DoS surface.

**Genuine findings:** F-01 (SSRF via broker endpoint), F-02 (context-less `net.Listen`), F-03 (TLS handshake without context in SNI sniffer).

**Notes:** bodyclose reported zero findings — the codebase consistently closes HTTP response bodies at all call sites. gosec G703 (`internal/proxy/broker_client.go`) was examined but suppressed: the projected SA token is read from a fixed Kubernetes volume mount set by the trusted control plane; the taint is a false positive from gosec treating struct fields as user-tainted.

### 4.5 kube-linter (Helm chart + manifests)

**Configuration:** `.kube-linter.yaml`.

- Targets scanned: rendered Helm chart (`helm template paddock charts/paddock --namespace paddock-system`) and `config/samples/`.

**Suppressions (with rationale):**

- `no-rolling-update-strategy` — both Deployments lack an explicit `strategy:` field; tracked as finding F-08 and suppressed here so CI remains green while the fix is scheduled.

**Genuine findings:** F-08 (Helm Deployments missing explicit rollout strategy on paddock-broker and paddock-controller-manager).

**Notes:** `config/samples/` produced no findings after suppressions. All kube-linter checks listed under `include:` in `.kube-linter.yaml` (privilege escalation, privileged containers, host network/PID, non-root enforcement, CPU/memory requirements, etc.) passed cleanly on both the Helm chart and sample manifests.
