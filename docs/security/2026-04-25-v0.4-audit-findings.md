### F-01 â€” SSRF via broker endpoint (proxy â†’ broker HTTP client)

| Field          | Value                                                                   |

**Description.**
`BrokerClient.do()` constructs an HTTP request using `c.Endpoint + path` where `Endpoint` is set at client construction time from configuration. Gosec's taint analysis (G704) flags this as a potential SSRF vector: if an attacker can influence the `Endpoint` value (e.g., via a compromised Pod env injection, or a future misconfiguration path), they could direct the proxy to POST authenticated tokens to an arbitrary URL.

The `TokenPath` (G703) is read fresh on every call and sent as the `Authorization` header â€” meaning every request leaks the projected SA token to whoever controls the endpoint.

**Evidence.**
- `internal/proxy/broker_client.go:157` â€” `os.ReadFile(c.TokenPath)` (G703)
- `internal/proxy/broker_client.go:161` â€” `http.NewRequestWithContext(ctx, ..., c.Endpoint+path, ...)` (G704)
- `internal/proxy/broker_client.go:171` â€” `c.hc.Do(req)` (G704)
- gosec rules: G703, G704

**Recommendation.**
Phase 2 should:
1. Validate at `NewBrokerClient` that the endpoint is an HTTPS URL matching the expected in-cluster service pattern (e.g., `https://paddock-broker.<ns>.svc.cluster.local`).
2. Consider pinning the endpoint to a compile-time constant or a validated config field with an allowlist.
3. Ensure the proxy Pod's securityContext prevents env variable injection from workspace containers.

### F-02 â€” Context-less `net.Listen` in transparent proxy mode

| Field          | Value                                                                   |
|----------------|-------------------------------------------------------------------------|
| Component      | proxy                                                                   |
| Threats        | T-? (to be assigned during component review)                            |
| Severity       | Likelihood: Low Â· Impact: Low Â· Risk: Low                               |
| Priority       | P3                                                                      |
| Test coverage  | Pending â€” to be assessed in Stage 5                                     |

**Description.**
`cmd/proxy/main.go:204` calls `net.Listen("tcp", listenAddr)` directly rather than `(*net.ListenConfig).Listen(ctx, ...)`. Without context propagation, the `Listen` call itself cannot be cancelled; if the address is already in use (e.g., during a race between proxy restart and kernel TIME_WAIT), there is no context deadline. This is a resilience concern in a proxy that binds privileged ports.

**Evidence.**
- `cmd/proxy/main.go:204`
- noctx rule

**Recommendation.**
Phase 2 should replace `net.Listen(...)` with `(&net.ListenConfig{}).Listen(ctx, "tcp", listenAddr)` so the listener inherits the process-level context and can be cancelled during shutdown races.

**Test gap correlation.**
Pending â€” assessed in Stage 5.

### F-03 â€” TLS handshake without context in SNI sniffer

| Field          | Value                                                                   |
|----------------|-------------------------------------------------------------------------|
| Component      | proxy                                                                   |
| Threats        | T-? (to be assigned during component review)                            |
| Severity       | Likelihood: Medium Â· Impact: Medium Â· Risk: Medium                      |
| Priority       | P1                                                                      |
| Test coverage  | Pending â€” to be assessed in Stage 5                                     |

**Description.**
`internal/proxy/sniffer.go:64` calls `tls.Server(dummy, cfg).Handshake()` without using `HandshakeContext(ctx)`. The sniffer's purpose is to peek at the TLS ClientHello to extract the SNI hostname before deciding whether to allow the connection. A slow or malicious egress target could hold the connection open by sending a partial ClientHello, causing the goroutine to block indefinitely without any timeout or cancellation.

Since each intercepted outbound TCP connection spawns a goroutine that calls into the sniffer, a burst of slow-TLS connections could exhaust the proxy's goroutine pool.

**Evidence.**
- `internal/proxy/sniffer.go:64`
- noctx rule: `(*crypto/tls.Conn).Handshake must not be called. use (*crypto/tls.Conn).HandshakeContext`

**Recommendation.**
Phase 2 should:
1. Replace `.Handshake()` with `.HandshakeContext(ctx)` where `ctx` carries a short deadline (e.g., 5s, matching the broker timeout budget).
2. Consider whether the `peekConn`/`teeReader` infrastructure correctly propagates deadlines down to the underlying `net.Conn`.

**Test gap correlation.**
Pending â€” assessed in Stage 5.

### F-08 â€” Helm Deployments missing explicit rollout strategy (paddock-broker, paddock-controller-manager)

| Component      | paddock-broker Deployment, paddock-controller-manager Deployment (Helm chart)                                      |

**Description.**
Neither the `paddock-broker` nor the `paddock-controller-manager` Deployment in the Helm chart has an explicit `spec.strategy:` field. Kubernetes defaults to `RollingUpdate` with `maxSurge: 25%` / `maxUnavailable: 25%`, but the absence of an explicit strategy means:

1. The intent is not captured in source control; future chart changes could silently alter rollout behaviour.
2. For a single-replica broker, a 25% maxUnavailable during rollout means a brief period with zero ready replicas, causing auth failures for any agent activity in flight.

kube-linter rule `no-rolling-update-strategy` flags this on both Deployments.

**Evidence.**
- kube-linter scan of `helm template paddock charts/paddock --namespace paddock-system` (run 2026-04-25)
- Rule: `no-rolling-update-strategy`
- Objects: `paddock-system/paddock-broker apps/v1, Kind=Deployment` and `paddock-system/paddock-controller-manager apps/v1, Kind=Deployment`

**Recommendation.**
Add explicit `strategy:` stanzas to both Deployments in the Helm chart (or its kustomize source):

```yaml

  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1
      maxUnavailable: 0  # zero-downtime rollout for single-replica broker
```

Consider `maxUnavailable: 0` for the broker (network-facing) to avoid any gap in auth coverage during updates.

### F-09 â€” Vertical providers do not scope `SubstituteAuth` to allowed hosts

| Field          | Value                                              |
|----------------|----------------------------------------------------|
| Component      | broker                                             |
| Threats        | T-1, T-3, T-5                                      |
| Severity       | Likelihood: Medium Â· Impact: High Â· Risk: High     |
| Priority       | P1                                                 |
| Test coverage  | gap                                                |

**Description.**
The three vertical providers (`AnthropicAPIProvider`, `GitHubAppProvider`, `PATPoolProvider`) implement `SubstituteAuth` in a way that ignores `req.Host`. Once a bearer is recognised by its prefix and not expired, the provider returns the real upstream credential â€” regardless of which host the proxy is currently MITMing. Only `UserSuppliedSecretProvider` checks `req.Host` against `lease.ProxyInjected.Hosts`.

The broker is documented (and treated by the threat model) as the trust root for credential substitution. If a `BrokerPolicy` ever grants egress to two different hosts under the same matching policy (e.g., `api.anthropic.com` and an unrelated host that happens to be MITMed), an Anthropic-bearer-bearing request to the unrelated host would receive the real Anthropic API key in `x-api-key`. The same shape applies to `GitHubApp` (Basic auth with the App installation token) and `PATPool` (Basic auth with a long-lived PAT â€” explicitly the highest-risk provider).

The proxy is supposed to gate which destinations are MITMed and which providers are invoked, but ADR-0014 explicitly states the broker re-checks per request. Per-request host scoping inside the provider is the correct enforcement layer for defense-in-depth: a proxy bug, a misconfigured `BrokerPolicy`, or a future cooperative-mode bypass should not result in real credentials reaching the wrong upstream.

**Evidence.**
- `internal/broker/providers/anthropic.go:169â€“221` â€” `SubstituteAuth` reads `req.Host` only as part of the comment block; never references it in any matching predicate.
- `internal/broker/providers/githubapp.go:244â€“282` â€” same shape; no host check.
- `internal/broker/providers/patpool.go:234â€“282` â€” same shape; no host check; `req.Host` unused.
- Contrast `internal/broker/providers/usersuppliedsecret.go:177â€“180` â€” `if !hostMatchesGlobs(req.Host, lease.ProxyInjected.Hosts)` correctly rejects mismatched hosts.
- `populateDeliveryMetadata` (`internal/broker/server.go:168â€“193`) returns the host list to the response on issuance, so the broker already knows which hosts each provider's bearer is scoped to â€” that knowledge is just not stored on the lease and consulted at substitute time.

**Recommendation.** Phase 2 should record the allowed-hosts list on each provider's lease at `Issue` time (Anthropic: defaults `[api.anthropic.com]` or `grant.Provider.Hosts` override; GitHubApp: `[github.com, api.github.com]` or override; PATPool: `grant.Provider.Hosts`) and reject `SubstituteAuth` calls whose `req.Host` is not in the list. Mirror the `hostMatchesGlobs` shape already used by `UserSuppliedSecretProvider`. Add unit tests that issue a vertical-provider bearer and assert SubstituteAuth on a non-allowed host returns an error.

**Test gap correlation.** Pending â€” assessed in Stage 5.

### F-10 â€” `SubstituteAuth` does not re-validate BrokerPolicy/HarnessRun mid-bearer-lifetime

| Field          | Value                                              |
|----------------|----------------------------------------------------|
| Component      | broker                                             |
| Threats        | T-3, T-8                                           |
| Severity       | Likelihood: Medium Â· Impact: Medium Â· Risk: Medium |
| Priority       | P1                                                 |
| Test coverage  | gap                                                |

**Description.**
`handleSubstituteAuth` (`internal/broker/server.go:412â€“482`) authenticates the proxy's caller token, looks up the lease in the matching provider, checks namespace + TTL, and substitutes. It does **not**:

1. Re-fetch the `HarnessRun` to confirm the run still exists / is not Cancelled / Succeeded.
2. Re-list `BrokerPolicy` to confirm the matching policy still grants the credential.
3. Re-check that the matched policy still applies to the run's template.

ADR-0014 explicitly promises: "Runtime enforcement is independent. Admission is a fast path; the proxy and broker re-check per request with the live BrokerPolicy cache. A BrokerPolicy deleted mid-run causes new upstream connections to be denied within the broker's 10 s cache-refresh interval." The broker's `handleIssue` path does re-check, but `handleSubstituteAuth` â€” the path that actually delivers real upstream credentials onto wire â€” does not.

Concrete attack window: an operator deletes a `BrokerPolicy` to revoke a credential mid-run. The proxy continues calling `SubstituteAuth` with the agent's still-valid bearer (TTL up to 1 h). Each call succeeds and substitutes the real upstream secret. The deletion has no effect until the bearer's lease expires.

This is also the path that lets a Cancelled run continue to make authenticated upstream calls until TTL â€” see also F-11.

**Evidence.**
- `internal/broker/server.go:412â€“482` â€” handler walks providers and substitutes; no `Get` on the run, no `List` of `BrokerPolicy`.
- `internal/broker/providers/anthropic.go:178â€“198` â€” only checks bearer ownership, namespace, expiry.
- `internal/broker/providers/githubapp.go:250â€“270` â€” same.
- `internal/broker/providers/patpool.go:240â€“272` â€” same.
- `internal/broker/providers/usersuppliedsecret.go:160â€“197` â€” same.
- `docs/adr/0014-capability-model-and-admission.md` lines 38 â€” the promised invariant.

**Recommendation.** Phase 2 should re-check at SubstituteAuth time: (a) the `HarnessRun` is in a non-terminal phase; (b) `matchPolicyGrant(ctx, c, ns, tpl, credentialName)` still returns a non-nil grant whose provider kind matches the lease; (c) optionally cache the result for â‰¤10 s per (run, credential) to bound API-server load. ### F-12 â€” Audit-write failures on issuance/denial are logged-and-swallowed; credential is still returned

| Field          | Value                                              |
|----------------|----------------------------------------------------|
| Component      | broker                                             |
| Threats        | T-3, T-5                                           |
| Severity       | Likelihood: Low Â· Impact: High Â· Risk: Medium      |
| Priority       | P2                                                 |
| Test coverage  | gap                                                |

**Description.**
`handleIssue` writes the AuditEvent **after** the provider has already minted the credential and recorded the lease in memory, and treats audit-write failure as informational:

```go

    if wErr := s.Audit.CredentialIssued(ctx, *audit); wErr != nil {

```

The denial path (lines 132â€“137) has the same shape: log and continue. The response is still written.

Spec 0002 Â§4 invariant 5 (per the audit input) says "AuditEvent emitted for every issuance, denial, and admission decision" and ADR-0016's write-once design treats each AuditEvent as the canonical immutable trail. Threat model T-5 (compromised paddock-system component) explicitly lists "modify or suppress AuditEvents" as an attack â€” apiserver write failures during a transient outage suppress them silently today.

This is an audit-completeness gap, not a credential-disclosure issue: the credential is correctly issued, but a watcher cannot rely on AuditEvents being a complete record without correlating against `paddock_broker_*` metrics.

**Evidence.**
- `internal/broker/server.go:128â€“151` â€” record-then-act ordering in `handleIssue`.
- `internal/broker/server.go:412â€“482` â€” `handleSubstituteAuth` emits no AuditEvent at all (separate concern but related to audit completeness).
- `docs/adr/0016-auditevent-retention.md:26` â€” "AuditEvent is write-once. `spec` is set at creation and immutable" â€” the design assumes a successful write.
- Threat model B-3 Repudiation defence: "AuditEvent on every issuance; `kind: credential-issued`" â€” the defence isn't actually fail-closed.

**Recommendation.** Phase 2 should change `handleIssue` to write the AuditEvent **before** returning the credential to the caller, and surface a 503 (or equivalent retryable error) when the audit write fails â€” fail-closed on the audit path. For the denial path, the audit write must succeed before the response is sent (the denial without trail is exactly the suppression attack T-5 cares about). Consider also emitting an AuditEvent for `handleSubstituteAuth` calls (or at minimum: failures), since substitution is the actual upstream-credential-on-wire event.

### F-14 â€” PATPool lease state lost on broker restart; PAT can be re-leased to a second concurrent run

| Field          | Value                                              |
|----------------|----------------------------------------------------|
| Component      | broker                                             |
| Threats        | T-3, T-8                                           |
| Severity       | Likelihood: Medium Â· Impact: Medium Â· Risk: Medium |
| Priority       | P2                                                 |
| Test coverage  | gap                                                |

**Description.**
`PATPoolProvider` stores the `(bearer â†’ pool index)` lease map purely in process memory (`patPool.byBearer`, `internal/broker/providers/patpool.go:108â€“116`). On broker restart the map starts empty. If existing run pods continue to present their old bearers, those bearers fail with `"patpool bearer not recognised"` (correct â€” fail-closed for the agent). However, when the controller reconciles those runs to `BrokerReady=False` and a fresh `Issue` is requested for the *new* lease, the provider picks `firstFreeIndex` over a freshly-zeroed `leased[]` slice and hands out **the same PAT** that an in-flight, not-yet-shut-down run still holds in its env. The other run cannot use that bearer (memory-reset), but the underlying PAT string is now associated with two distinct runs in audit terms.

Spec 0002 Â§4 invariant 7 (per the audit input) requires "PATPool lease/return with leak handling on broker restart." The provider comments (`internal/broker/providers/anthropic.go:74â€“78`) acknowledge "a cold broker also kicks out any in-flight runs via BrokerReady=False" but no broker-side mechanism enforces the leak handling â€” the leases are not persisted to etcd, the run-status doesn't carry LeaseID, and the broker has no recovery-from-crash path.

The audit-trail confusion is mostly cosmetic for short-TTL providers (Anthropic, GitHubApp) but for PATPool the same PAT is now genuinely in two run namespaces' audit logs without a chain-of-custody linking them.

**Evidence.**
- `internal/broker/providers/patpool.go:79â€“93` â€” `PATPoolProvider` struct holds `pools map[patPoolKey]*patPool` in memory only.
- `internal/broker/providers/patpool.go:163â€“227` â€” `Issue` reads-then-leases; no etcd-persisted state.
- `cmd/broker/main.go:143â€“152` â€” provider construction creates fresh in-memory state every process start.
- ADR-0015 Â§"Consequences" lines 49â€“54 â€” the broker is described as stateless because "leases are a function of `(runUID, providerName, requirementName)` reconstructable from HarnessRun watches" â€” the PATPool implementation does not actually reconstruct on startup.

**Recommendation.** Phase 2 should either: (a) persist the `(runUID, leaseID, poolIndex)` triple onto `HarnessRun.status` so the provider can rebuild its state from a HarnessRun list at startup; or (b) treat any restart as a hard fence â€” refuse Issue for the first 60 s and require all run pods to retry, so an operator can validate the BrokerReady=False rebuild before the pool is reused. Option (a) is closer to ADR-0015's stateless-by-reconstruction claim.

**Test gap correlation.** Pending â€” assessed in Stage 5.

### F-15 â€” Broker TLS server has no client-cert verification; ADR-0012 says "mTLS"

| Field          | Value                                              |
|----------------|----------------------------------------------------|
| Component      | broker                                             |
| Threats        | T-4, T-5                                           |
| Severity       | Likelihood: Low Â· Impact: Low Â· Risk: Low          |
| Priority       | P3                                                 |
| Test coverage  | gap                                                |

**Description.**
`cmd/broker/main.go:164â€“176` constructs a `tls.Config` for the broker's `:8443` API with `MinVersion: TLS13` and `GetCertificate`. It does **not** set `ClientAuth` â€” the default is `tls.NoClientCert`. The server therefore accepts any TLS-able client, and authentication is purely SA-token-based (TokenReview).

ADR-0012 Â§"API" says the broker uses "HTTP/JSON over mTLS"; threat model B-3 Tampering defence is "mTLS broker server cert (cert-manager-issued)" (which is server-only). In practice, SA-token authentication is robust on its own (TokenReview rejects forged tokens), and adding mTLS would constrain only "what client may attempt a TokenReview-bearing call" â€” a defense-in-depth gain, not a prerequisite for the trust model to hold. **Evidence.**
- `cmd/broker/main.go:164â€“176` â€” `tls.Config` with no `ClientAuth` field.

- `internal/broker/auth.go:80â€“120` â€” sole auth path; SA-token-only.

**Recommendation.** Phase 2 should either: (a) update ADR-0012 to read "TLS server cert + SA-token bearer" since that is what's implemented, or (b) wire `tls.VerifyClientCertIfGiven` (or `RequireAndVerifyClientCert` against a paddock-system CA bundle) so the documented mTLS posture matches reality. Option (a) is fine â€” the SA-token check is the trust boundary; option (b) is defense-in-depth and would also harden against a stolen SA token from a compromised pod.

### F-17 â€” No request rate limit, body-size limit, or per-run quota on broker handlers

| Field          | Value                                              |
|----------------|----------------------------------------------------|
| Component      | broker                                             |
| Threats        | T-1, T-4                                           |
| Severity       | Likelihood: Medium Â· Impact: Low Â· Risk: Low       |
| Priority       | P2                                                 |
| Test coverage  | gap                                                |

**Description.**
The broker decodes request bodies via `json.NewDecoder(r.Body).Decode(&req)` without first wrapping `r.Body` in `http.MaxBytesReader`. The handler chain has no rate limiter, no per-run quota, and no per-caller concurrency cap. `http.Server` is configured with `ReadHeaderTimeout: 10 * time.Second` but no `ReadTimeout` / `WriteTimeout` / `IdleTimeout`. Threat model B-3 DoS row already flags this: "Per-run rate limiting (TBD audit finding)."

A co-tenant who has obtained their own pod's SA token (legitimate; they have a HarnessRun in their namespace) can:

1. Open many connections, sending slow-write request bodies. Goroutine occupancy grows linearly; no `WriteTimeout` ever cuts them off after the headers parsed.
2. Send oversized request bodies (multi-MB JSON) to push the broker into GC pressure.
3. Spam `/v1/issue` with valid credentials to flood the AuditEvent stream and cost-shift to etcd.

ADR-0016 specifies emitter-side debounce and burst-collapse-into-summary-events for AuditEvents, but `internal/broker/audit.go:43â€“50` is the simplest possible writer: one Create per event. The debounce is unwired.

**Evidence.**
- `internal/broker/server.go:118â€“122, 344â€“353, 432â€“442` â€” `json.NewDecoder` without `MaxBytesReader`.
- `cmd/broker/main.go:178â€“183` â€” `http.Server` with only `ReadHeaderTimeout`.
- `internal/broker/audit.go:38â€“41` â€” comment acknowledges "v0.3 ships the simplest possible writer" and "the debounce + summary story from ADR-0016 is wired at the emitter (per-run counter here + a periodic flush) in later milestones".
- Threat model Â§5 B-3 DoS â€” TBD.

**Recommendation.** Phase 2 should: (a) wrap each handler's body read in `http.MaxBytesReader(w, r.Body, 64<<10)` (issue requests are tiny; no legitimate body exceeds that); (b) add `ReadTimeout`, `WriteTimeout`, `IdleTimeout` on the `http.Server`; (c) add a per-run token bucket on `/v1/issue` and `/v1/substitute-auth` (the substitution path is the hot one â€” proxy-side per-request); (d) wire ADR-0016's debounce in the AuditWriter with the per-`{kind, decision}` summary-event collapse.

## 3. Severity scheme reference

OWASP-style risk = Likelihood Ă— Impact, plus a separate Priority field.

**Likelihood:**

- **High** â€” an attacker who already has tenant-namespace access can trigger this with default Paddock config.
- **Medium** â€” requires a misconfiguration, a co-tenant, or non-default settings.
- **Low** â€” requires a specific chain of conditions or actor capabilities (e.g., compromised paddock-system component).

**Impact:**

- **High** â€” credential disclosure, cross-tenant data leak, audit suppression, or root-equivalent escalation.
- **Medium** â€” single-tenant data leak, denial of service against one run, audit completeness gap.
- **Low** â€” informational, hardening, defence-in-depth.

**Risk = quadrant of Likelihood Ă— Impact** (3Ă—3 matrix; HĂ—H = High Risk, LĂ—L = Low Risk, mixed = Medium Risk).

**Priority (separate from severity):**

- **P0** â€” drop everything; ship a fix or workaround for the next release.
- **P1** â€” must fix before v1.0.
- **P2** â€” should fix; schedule into a milestone.
- **P3** â€” nice-to-have / hardening / track-but-don't-block.
