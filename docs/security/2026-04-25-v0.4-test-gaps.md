## 3. Attack-scenario list

Each TG-NN below is a concrete adversarial scenario with enough detail that Phase 2 can implement directly without re-design. Multiple findings can share a TG-NN where the same test catches them all.

### TG-1 — Agent forges a leaf cert against the per-run MITM CA

**Scenario.** A run is configured with the per-run proxy CA. The hostile agent reads the proxy's CA bundle (which is mounted into the agent container as `/etc/paddock/proxy-ca.crt`) and, having located its private key (deliberately *not* mounted into the agent), proves the negative — the private key is unreachable from the agent's UID, mount points, and SA-token-derived API server reads.

**Expected behaviour.** The harness's `--forge-ca-cert` flag returns `{"result":"key-unreachable"}` in all four read attempts (filesystem walk for `tls.key`-named files, `kubectl get secret` via SA token, env var probe, `/proc/self/mounts` probe). If any attempt yields key bytes, the test fails loudly.

**Realism level.** Real cluster (Kind, any CNI), hostile harness (yes, `--forge-ca-cert`), real upstream not required (assertion is on key-reachability, not on a forged TLS connection's outcome).

**Test layer.** E2E.

**Related findings.** F-18.

**Hostile harness flags used.** `--forge-ca-cert <fqdn>`.

**Phase 2 effort estimate.** Medium — needs hostile harness with `--forge-ca-cert` and an assertion harness that reads the run's proxy-tls Secret outside the run pod to confirm content vs. what agent sees.

---

### TG-2 — Agent unsets HTTPS_PROXY in cooperative mode

**Scenario.** Run admitted with `interception.cooperativeAccepted: true`. The hostile agent unsets `HTTPS_PROXY`, `HTTP_PROXY`, `NO_PROXY`, then attempts a direct TCP connection to `api.openai.com:443` (or any non-allowlisted internet host).

**Expected behaviour.** The connection is blocked by NetworkPolicy egress rules, *or* — if the test reveals NetworkPolicy doesn't actually block the bypass (F-19) — the test must fail loudly with the diagnostic that cooperative mode + current NP shape allows direct egress, vindicating F-19 and providing the regression test for the eventual fix. AuditEvent records the attempt as `kind: egress-block` with the per-connection 5-tuple.

**Realism level.** Real cluster (Kind with Cilium or Calico — kindnet does not enforce NetworkPolicy), hostile harness (yes, `--bypass-proxy-env` + `--connect-raw-tcp`), real upstream not required (assertion is on the egress decision).

**Test layer.** E2E.

**Related findings.** F-19, F-27.

**Hostile harness flags used.** `--bypass-proxy-env`, `--connect-raw-tcp api.openai.com:443`.

**Phase 2 effort estimate.** Medium — depends on Cilium/Calico Kind setup (TG-realism §5 must-have), hostile harness build, and a non-allowlisted domain that resolves but rejects.

---

### TG-3 — Tenant overrides `HTTPS_PROXY`/`SSL_CERT_FILE` via `extraEnv`

**Scenario.** Tenant submits a HarnessRun with `spec.extraEnv: [{name: HTTPS_PROXY, value: ""}, {name: SSL_CERT_FILE, value: "/etc/ssl/certs/ca-certificates.crt"}]`. The controller builds the Pod spec; the K8s Pod env-merge resolves duplicates last-wins, so the tenant value wins.

**Expected behaviour.** Either (a) the webhook rejects the HarnessRun at admission with an error naming the reserved keys (post-fix), or (b) the controller's effective Pod env shows Paddock's HTTPS_PROXY value (because the controller appends extraEnv first and Paddock-authored env last). Until either fix lands, the test asserts the *current* (broken) behaviour and must fail loudly to provide the regression target.

**Realism level.** Unit (controller pod-spec) + integration (webhook envtest). Real cluster not required.

**Test layer.** Unit (controller pod-spec build) + Integration (envtest webhook).

**Related findings.** F-39.

**Hostile harness flags used.** None — this attack is purely admission/spec-level.

**Phase 2 effort estimate.** Small — table-driven webhook test + controller pod-spec test.

---

### TG-4 — Agent bypasses iptables-redirect via RFC1918 IP literal (transparent mode)

**Scenario.** Run uses transparent interception. The hostile agent dials an arbitrary RFC1918 destination by IP — e.g., `10.42.7.1:443` (a co-tenant pod), `10.96.0.1:443` (apiserver), `paddock-broker.paddock-system.svc.cluster.local` resolved to `10.x.y.z:8443`. The iptables init container's RETURN-on-RFC1918 rule means these connections never hit the proxy.

**Expected behaviour.** Either (a) the iptables-init narrows RFC1918 RETURN to a documented per-cluster service CIDR + apiserver IP only and the connection is REDIRECTed to the proxy, where the allowlist denies it; or (b) the test asserts the current behaviour fails loudly so F-20 is regression-pinned. Specific assertions: the broker, kube-apiserver, and a co-tenant pod IP are all blocked from the agent.

**Realism level.** Real cluster (Kind), real CNI (Cilium/Calico for NetworkPolicy backstop), hostile harness (`--connect-ip-literal`), real upstream not required.

**Test layer.** E2E.

**Related findings.** F-20.

**Hostile harness flags used.** `--connect-ip-literal 10.96.0.1:443`, `--connect-ip-literal <broker-clusterip>:8443`, `--connect-ip-literal <co-tenant-pod-ip>:80`.

**Phase 2 effort estimate.** Large — requires Kind cluster with multi-namespace setup (co-tenant), broker reachability test, plus iptables-init narrowing implementation.

---

### TG-5 — Agent attempts DNS rebinding to allowlisted name

**Scenario.** BrokerPolicy allows egress to `api.anthropic.com:443`. The hostile agent (a) configures its own DNS resolver to map `api.anthropic.com` to an attacker-controlled IP (e.g., a probe-server pod), or (b) issues a CONNECT for `api.anthropic.com:443` while the cluster DNS is briefly poisoned/cached to return an attacker IP. The proxy resolves the host string to the attacker IP and dials it; the upstream cert validation is the only remaining gate.

**Expected behaviour.** The proxy resolves the SNI/CONNECT host using its own resolver (independent of the agent's DNS), compares the resolved IP set to the SO_ORIGINAL_DST (transparent) or to the dialed IP (cooperative), and rejects when they don't match. AuditEvent records `kind: egress-block` with `reason: dns-rebinding-detected`.

**Realism level.** Real cluster, hostile harness with `--exfil-via-dns`-like control over a custom resolver, real upstream not required (test uses a controllable fake upstream that stands in for the attacker).

**Test layer.** Unit (resolver double-check function) + E2E (full path).

**Related findings.** F-22.

**Hostile harness flags used.** `--connect-raw-tcp api.anthropic.com:443` with `/etc/resolv.conf` pointing at a sidecar resolver-shim that returns attacker IP for the allowed name.

**Phase 2 effort estimate.** Large — needs a controllable DNS resolver in the run pod's network namespace (or a way to convince the cluster DNS to return a bad answer briefly), plus the proxy-side double-resolve fix.

---

### TG-6 — Per-run MITM CA private key is not actually per-run

**Scenario.** Two HarnessRuns are admitted in different tenant namespaces (or the same namespace, two distinct runs). The test reads `kubectl get secret <run-A>-proxy-tls -o yaml` and `<run-B>-proxy-tls -o yaml` and compares `tls.key`.

**Expected behaviour.** Either (a) the keys differ (post-fix: per-run intermediate CA), or (b) the test asserts the current shared-key reality and must fail with a clear message that this is the F-18 regression pin. Additionally, the hostile harness's `--forge-ca-cert` (TG-1 component) attempts to use `<run-A>` agent's view of `ca.crt` to validate a leaf signed by `<run-B>`'s key — this should fail, but in the current shared-key world it succeeds, which the test asserts is the bug.

**Realism level.** Integration (envtest with two run namespaces) + E2E (real cluster, two runs).

**Test layer.** Integration (envtest) and E2E.

**Related findings.** F-18.

**Hostile harness flags used.** `--forge-ca-cert <victim-run-fqdn>` combined with cross-namespace cert reading.

**Phase 2 effort estimate.** Medium — envtest version is small (compare two Secrets); E2E version requires two-tenant Kind.

---

### TG-7 — Agent forges AuditEvent via collector SA token

**Scenario.** Hostile agent reads `/var/run/secrets/kubernetes.io/serviceaccount/token` (auto-mounted by default) and uses it to `POST /apis/paddock.dev/v1alpha1/namespaces/<run-ns>/auditevents` directly — claiming a forged `kind: egress-allowed host=internal.evil.com` event. Separately, the agent attempts to `PATCH` the run's output ConfigMap.

**Expected behaviour.** Either (a) the agent container has `automountServiceAccountToken: false` (post-fix), making the token absent; or (b) the AuditEvent webhook (F-32 hardening recommendation 4) rejects writes whose writer-SA does not match the run's proxy SA. Either way, the forged event is refused. Assertion: a controller-side audit-stream consumer sees zero AuditEvents authored by the agent's SA.

**Realism level.** Real cluster, hostile harness (`--read-secret-files /var/run/secrets/...`, `--probe-broker https://kubernetes.default.svc/apis/paddock.dev/...`), real upstream not required.

**Test layer.** E2E (the existence-of-token check is unit-level on pod-spec but the actual forgery is E2E).

**Related findings.** F-38, F-48 (seed-Pod equivalent), F-32 (admission audit gap that pairs with this).

**Hostile harness flags used.** `--read-secret-files /var/run/secrets/kubernetes.io/serviceaccount/token`, `--probe-broker https://kubernetes.default.svc:443/apis/paddock.dev/v1alpha1/namespaces/<run-ns>/auditevents` with custom POST body.

**Phase 2 effort estimate.** Medium — small unit test on pod-spec (assert `automountServiceAccountToken: false`) + E2E forgery attempt.

---

### TG-8 — Co-tenant reads another tenant's PVC

**Scenario.** Tenant A and Tenant B each have a HarnessRun with a Workspace-backed PVC. Tenant B's hostile agent attempts to (a) `kubectl get pvc -n <tenant-A-ns>`, (b) submit a HarnessRun in tenant B's namespace that mounts a PVC named `<tenant-A-ns>/<workspace>` (cross-namespace mount — should be impossible by default), (c) read the PVC bytes via a hostPath escape (PSS `restricted` blocks).

**Expected behaviour.** All three paths fail. (a) Forbidden by RBAC (the namespace-scoped agent SA has no list permission elsewhere). (b) Admission rejects: PVCs are namespace-scoped. (c) PSS `restricted` rejects hostPath; even if the agent crafted a `volume: hostPath`, admission denies the Pod creation.

**Realism level.** Real cluster (multi-namespace), hostile harness (`--read-other-tenant-pvc`).

**Test layer.** E2E.

**Related findings.** Threat-model T-4 (no specific F-NN — this is the canonical co-tenant test; the absence of any test for it is itself a finding to be added by the test inventory).

**Hostile harness flags used.** `--read-other-tenant-pvc <victim-namespace>`.

**Phase 2 effort estimate.** Medium — two-namespace Kind setup + harness flag.

### TG-9 — Agent exhausts proxy resources (DoS)

**Scenario.** Hostile agent (a) opens 10,000 concurrent CONNECTs to allowed hosts and never closes them, (b) issues 10,000 distinct synthetic SNIs to allowed wildcard hosts (`*.s3.amazonaws.com`-style) to populate the forged-leaf cache, (c) sends slow-loris HTTP requests to the substitute-auth path. Concurrently, broker is hit with a flood of `/v1/issue` requests.

**Expected behaviour.** Proxy enforces a per-connection cap (e.g., 256), an idle/read/write timeout (e.g., 60s), an LRU on the forged-leaf cache (e.g., 1024 entries). Broker enforces `MaxBytesReader` on body, `ReadTimeout`/`WriteTimeout`, and per-run rate limit. ## 4. Hostile harness ("evil paddock-echo") specification

A test-only harness image that exercises hostile agent behaviours. Used by multiple TG-NN entries above. **Phase 2 deliverable**, specified here so Phase 2 doesn't re-design.

### Image location

`images/evil-echo/` — separate from `images/harness-echo/` so the trustworthy reference harness stays clean. Image tagged `paddock-evil-echo:dev` for local builds; not part of release artefacts; not pushed to public registries. The Makefile target `make evil-echo-image` builds and loads into the Kind test cluster.

### Base

Built on `paddock-echo`'s base layers — same Pod Security Standards conformance — so it can run in the same `restricted` namespaces as the friendly version. The hostile behaviours are CLI-flag-gated; absent any flag, the image behaves identically to `paddock-echo` (i.e., echoes its prompt, writes a result.json, exits 0).

### CLI flags (each is one attack)

Standard set (from the design doc):

- `--connect-raw-tcp <host:port>` — open a raw TCP socket to the host:port; assert the test sees the connection blocked or substituted-via-proxy. Used by TG-2, TG-12.
- `--read-secret-files <glob>` — read files matching the glob (typically `/var/run/secrets/...`); print contents to stdout. Tests assert the harness either fails to read or reads only what's documented as visible to the agent. Used by TG-7.
- `--probe-broker <url>` — send a request directly to the broker's mTLS endpoint, including using stolen / guessed bearers; assert the broker responds with deny. Supports `--with-bearer <token>` and `--with-sa-token <path>`. Used by TG-7, TG-11, TG-18.
- `--bypass-proxy-env` — unset `HTTPS_PROXY`, `HTTP_PROXY`, `NO_PROXY` before subsequent actions. Combined with other flags to test cooperative-mode bypass. Used by TG-2.
- `--exfil-via-dns <host>` — encode payloads in DNS queries to a non-allowlisted DNS resolver; assert resolver requests are blocked. Used by TG-24 (variant).
- `--read-other-tenant-pvc <namespace>` — attempt to mount/read PVCs from another namespace; assert RBAC blocks. Used by TG-8.
- `--forge-ca-cert <fqdn>` — attempt to read the per-run CA private key (filesystem walk, env probe, SA-token-driven kubectl get); if read, generate a cert; attempt MITM. Assert key not readable. Used by TG-1, TG-6.
- `--connect-ip-literal <ip:port>` — connect to an IP literal bypassing DNS; assert the proxy still rejects (allowlist is by name). Used by TG-4, TG-11, TG-12.
- `--probe-imds` — attempt to reach `169.254.169.254:80` (cloud metadata service) via direct TCP; assert blocked by NetworkPolicy or proxy allowlist. Used by TG-12.
- `--connect-loopback <port>` — connect to in-pod loopback (e.g., the proxy itself, the SA-token volume's API server proxy); assert only documented loopback ports respond. Used by TG-12.

Audit-derived additional flags:

- `--probe-env-override` — start a child process whose env table includes a tenant-injected `HTTPS_PROXY=""` (or `SSL_CERT_FILE=...`) overriding controller-emitted values; assert that the in-container effective env reflects the controller's values, not the tenant's. Tests F-39 in-container alongside the admission check. Used by TG-3.
- `--probe-provider-substitution-host <bearer> <wrong-host:port>` — present an Anthropic / GitHubApp / PATPool bearer in a request whose `Host` header (or SNI) is not in the bearer's allowed-hosts list; assert the broker's `SubstituteAuth` returns no substitution. Tests F-09. Used by TG-13.
- `--read-pvc-git-config` — mount the workspace PVC read-only into a sidecar of the harness Pod and dump `.git/config` and `/workspace/.paddock/repos.json` to stdout; assert no embedded credential substring. Tests F-50. Used by TG-25.
- `--flood-connect-raw-tcp <host:port> --count=N` and `--flood-distinct-sni <wildcard-pattern> --count=N` — load-shape variants of `--connect-raw-tcp` for DoS scenarios (TG-9, F-26, F-28). Bounded by N to avoid runaway test resource use.
- `--smuggle-headers <host:port>` — open a substitute-auth path connection with a fixed set of extra headers (`x-api-key: smuggled`, `cookie: stolen`, etc.); assert the upstream stand-in receives only the per-host allowlist. Tests F-21. Used by TG-10.

### Output

Each flag produces structured stdout, one JSON object per attempt:

```json
{"flag": "--connect-raw-tcp", "target": "api.openai.com:443", "result": "denied", "error": "connection refused", "duration_ms": 12}
{"flag": "--forge-ca-cert", "fqdn": "api.anthropic.com", "result": "key-unreachable", "attempts": [{"path": "/etc/paddock/proxy-ca.key", "error": "no such file"}, {"sa_kubectl_get": "forbidden"}]}
{"flag": "--probe-broker", "url": "https://paddock-broker.paddock-system.svc:8443/v1/issue", "result": "denied", "status": 403, "body_excerpt": "cross-namespace ..."}
```

Result values: `denied`, `allowed-as-expected`, `unexpectedly-allowed`, `key-unreachable`, `error`. Tests parse this output to assert the expected outcome for each attack.

### Pod Security Standards conformance

The harness must pass `restricted` PSS — no privileged capabilities, no NET_ADMIN, no hostPath, no `runAsRoot`, `allowPrivilegeEscalation: false`, all caps dropped, seccomp `RuntimeDefault`. Some attacks (e.g., raw TCP outside HTTPS, packet-crafting) are constrained by this; that is intentional — we want to test what a malicious-but-not-privileged agent can do. Attacks needing kernel-level privilege (`NET_ADMIN`, hostPath escape) are explicitly out of scope of the harness — they're operator-posture concerns, not Paddock's containment perimeter.

### Testing the harness itself

A small unit-test suite verifies each flag does what it claims (without external network — tests use loopback or test-doubles for the broker / upstream). The harness's correctness is its own deliverable in Phase 2; tests above use it as a tool, but its correctness is established independently.

### Total flags

15 = 10 standard + 5 audit-derived.

## 5. Test-realism dimensions

For each E2E entry currently in `test/e2e/e2e_test.go` (and each TG-NN above whose layer is E2E), an explicit declaration of which realism axes are exercised. The Phase 2 must-haves and opportunistic items are then derived.

### Axes

- **Real CNI** — does the test cluster use a CNI that *enforces* NetworkPolicy (Cilium / Calico) or a no-op CNI (kindnet default)? Without a real CNI, every TG-NN that asserts NetworkPolicy enforcement is meaningless.
- **Real upstream** — does the test hit real external services (`github.com`, `api.anthropic.com`), or stub them? Real upstream is necessary only when the assertion *requires* observing the actual upstream's response.
- **Hostile harness** — does the test run an actual hostile image (`paddock-evil-echo`), or simulate hostile behaviour via prompt-engineering / mocks? Prompt-driven hostility is fragile and depends on the harness's behaviour; image-driven hostility is deterministic.
- **Concurrency** — does the test exercise parallel runs / mid-flight edits / fault injection (broker restart under load, finalizer races)?

### Phase 2 must-haves

- **Real CNI everywhere.** Switch the Kind setup to install Cilium or Calico. Update `make kind-up` and the e2e `setup-test-e2e` target. Without this, TG-2 / TG-4 / TG-11 / TG-12 / TG-24 are meaningless. This is a one-time setup change with downstream effect on every NetworkPolicy-asserting test.
- **Hostile harness for runtime-isolation tests.** Phase 2 builds the harness from §4. Once it exists, sixteen TG-NN entries become implementable. The build itself is small (Go binary with a flag-table); the test integrations are where the labour lives.

### Phase 2 opportunistic

- **Real upstream where the assertion requires it.** TG-10 (substitute-auth header smuggling) needs an upstream that observes which headers actually arrive — this can be a controllable fake (an in-cluster pod running a header-recording HTTP server) rather than `api.anthropic.com`. TG-24 (workspace seed) benefits from a controllable git host (e.g., a `gitea` pod) rather than `github.com` to avoid CI flakiness.

### Phase 2 flagged but deferred

- **Concurrency / lifecycle.** Multi-run interleaving, broker restart under load, finalizer race tests. TG-22 (bearer revocation) and TG-23 (mid-run mutation) touch this lightly — they exercise the *single*-run case. Genuinely concurrent / racy tests are their own engineering effort and not blocking the core test-gap fill.
